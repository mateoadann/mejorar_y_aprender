Working memory: el estado vivo

La working memory es la capa volatil. Cambia cada pocos minutos, vive el tiempo que dura la tarea, y se vuelve inutil en el momento en que la terminas. Su unico proposito es que manana puedas retomar exactamente donde dejaste.

El error de tratar la memoria como una pila unica

El error mas grande al armar un setup de agentes es tratar la memoria como una sola pila indiferenciada: todo va al mismo lado, todo retiene igual, todo se consulta con la misma query. Ese diseno funciona unas 6 semanas. Despues se rompe.

Se rompe porque distintos tipos de memoria necesitan politicas distintas de retencion (cuanto vive), de retrieval (como se consulta) y de update (cuando se escribe). Mezclarlas es como guardar la lista del supermercado, las escrituras de tu casa y el manual de tu auto en la misma caja de zapatos.

Por eso la memoria bien disenada se separa en capas. La primera — la que arrancamos hoy — es la working memory.

Que es working memory

Working memory es el scratchpad vivo del agente durante una tarea. Pensala como el pizarron del arquitecto mientras esta en obra: esta lleno de flechas, numeros de pieza, mediciones, garabatos. No sirve para otra obra. Solo sirve para esta.

Volatilidad
Minutos

Cambia a cada paso del agente. Cada edicion, cada decision, cada hipotesis nueva la actualiza.

Ciclo de vida
La tarea

Nace cuando arranca la task y muere cuando la task termina. No sobrevive a la siguiente.

Proposito
Resumption

Que manana (o cuando el context window se resetea) podas retomar exactamente donde dejaste.

El archivo canonico: WORKSPACE.md

En un setup serio, la working memory vive en un archivo fijo y localizable: 

memory/
├── working/
│   └── WORKSPACE.md    ← el pizarron de la tarea actual
├── episodic/
├── semantic/
└── personal/

Un solo archivo. Path predecible. El agente lo encuentra siempre sin buscar. Cuando arranca una sesion nueva, lo primero que hace es leer WORKSPACE.md y reconstruir el estado mental.

Estructura tipica de un WORKSPACE.md

El archivo es markdown plano, pero tiene cuatro bloques fijos. Esa rigidez es deliberada: facilita que el agente lo parse rapido y lo actualice sin pensarlo.

memory/working/WORKSPACE.md
# Current task
Agregar rate limiting al endpoint POST /login
Estimado: ~2h restantes

# Open files
- src/auth/login.ts        (lineas 42-78)
- src/middleware/rate.ts   (lineas 1-35, nuevo)
- tests/auth.test.ts       (lineas 104-130)

# Active hypotheses
- H1: el limite por IP no alcanza — hay que agregarlo por user_id
- H2: Redis devuelve null cuando la key no existe, no 0 (verificar)

# Checkpoints
[x] Leer el RFC interno de rate limiting
[x] Decidir backend: Redis sobre in-memory (ver DECISIONS.md)
[ ] Implementar middleware
[ ] Tests de unidad
[ ] Tests de integracion con Redis real
[ ] Actualizar CHANGELOG

Current task te dice que. Open files te dice donde. Active hypotheses te dice que estas probando. Checkpoints te dice cuanto falta. Con esos cuatro bloques, cualquier agente (incluido uno nuevo despues de un reset) retoma la tarea en segundos.

Lifecycle: crear, actualizar, archivar

El ciclo de vida del WORKSPACE tiene tres momentos. Respetalos y no vas a tener problemas; saltealos y el agente se te queda colgado en tareas fantasma. 

1) CREATE               2) UPDATE (cada paso)          3) ARCHIVE
┌────────────────┐       ┌──────────────────────┐     ┌──────────────────────┐
│ task nueva     │  ──►  │  editar WORKSPACE    │ ──► │  task terminada      │
│ crea archivo   │       │  - new hypothesis    │     │  mover a episodic    │
│ con bloques    │       │  - check checkpoint  │     │  memory/snapshots/   │
│ vacios         │       │  - agregar open file │     │  y vaciar WORKSPACE  │
└────────────────┘       └──────────────────────┘     └──────────────────────┘
        │                         │                             │
        ▼                         ▼                             ▼
   minuto 0               cada 1-5 min                   al cerrar

El paso 2 es el que mas se descuida. La regla simple: cada vez que tu agente toma una decision o descubre algo, tiene que tocar el WORKSPACE. No es documentacion, es el hilo que lo mantiene orientado.

El paso 3 es el que salva la cordura. Cuando la tarea termina, el WORKSPACE no se borra: se archiva como episodio (lo vemos en la proxima pagina) y se reemplaza por un archivo vacio, listo para la proxima task.

Por que importa tanto el archivado

Stale workspace: el bug mas costoso

Si no limpias el WORKSPACE al terminar una tarea, el agente despierta al otro dia convencido de que sigue en la tarea vieja. Edita files que ya no corresponden, marca checkpoints que ya estaban hechos, revive hipotesis descartadas. El sintoma es raro porque parece que el agente "alucina" — en realidad esta siguiendo al pie de la letra un pizarron que nadie le borro.

Por eso el contrato ideal tiene dos hooks automaticos:

  • on_task_start: verifica que WORKSPACE este vacio. Si no lo esta, archiva lo que haya y arranca limpio.
  • on_task_complete: archiva el WORKSPACE como episodio, vacia el archivo.

Sin esos hooks vas a depender de que el agente (o vos) se acuerden de limpiar. Y la memoria humana tambien falla.

Working memory vs. context window

Cuidado con confundir estos dos conceptos. El context window es el pedazo de tokens que el modelo ve en una llamada: es un limite fisico del LLM. La working memory es un archivo en disco que sobrevive mas alla de una sola llamada, incluso mas alla de una sola sesion.

Context window
Tokens en RAM

Vive durante una invocacion. Se pierde al compactar o reiniciar. Limite tipico: 200K–1M tokens segun modelo.

Working memory
Archivo en disco

Vive mientras dure la tarea. Sobrevive a resets del context window. Se lee al arrancar cada sesion nueva.

Cuando el context window se resetea (compaction, timeout, cambio de modelo), el WORKSPACE es el puente que permite retomar. Sin el, cada reset es empezar de cero.

Regla de oro

Actualiza siempre, archiva al cerrar

Si dudas entre actualizar o no, actualiza. El costo de escribir 3 lineas de mas en WORKSPACE es trivial. El costo de que un agente despierte desorientado es horas de trabajo perdido. Y cuando la tarea termina, archivala sin piedad — lo que pasaste ayer no pertenece a la tarea de hoy.

Mini-resumen

Que es
Scratchpad de la tarea

Un archivo fijo, memory/working/WORKSPACE.md, con el estado vivo del agente.

Que tiene
Task, files, hipotesis, checkpoints

Cuatro bloques fijos. Markdown plano. Cualquiera puede leerlo — incluido un agente nuevo.

Para que
Resumption

Sobrevive a resets del context window. Es el puente entre sesiones cuando todo lo demas se perdio.
