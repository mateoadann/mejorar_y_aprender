Dream cycle: compresion nocturna

Se llama "dream cycle" porque corre mientras nadie mira (cron 3am) y comprime el dia de logs crudos en lecciones destiladas — igual que el sueno consolida lo que el cerebro vivio durante el dia. Sin este ciclo, la memoria del agente se pudre en vez de componer.

La metafora biologica no es decorativa

En humanos, el sueno REM consolida la memoria del dia: lo importante pasa a memoria de largo plazo, lo irrelevante se descarta, los patterns se abstraen. Si no dormis, tu aprendizaje del dia se evapora.

Los agentes tienen el mismo problema. Si episodic crece indefinidamente sin consolidacion, termina siendo un landfill: ruidoso, pesado y tan caro de consultar que no aporta. El dream cycle hace tres operaciones cada noche para evitar ese destino.

Las tres operaciones del ciclo

                  DREAM CYCLE (3 am)
          ────────────────────────────────

┌─────────────────────────────────────────────────────────┐
│  1. DETECCION de patterns recurrentes                   │
│     clusteriza entries por (skill, action)              │
│     si aparece 2+ veces ─► boost salience               │
│                           set recurrence_count          │
├─────────────────────────────────────────────────────────┤
│  2. PROMOCION a semantic                                │
│     entries con salience >= 7.0                         │
│     ─► append a LESSONS.md                              │
│     (nada de curado manual)                             │
├─────────────────────────────────────────────────────────┤
│  3. DECAY + archive                                     │
│     entries de > 90 dias con salience < 2.0             │
│     ─► movidas a episodic/snapshots/                    │
│     ─► git commit                                       │
│     (no se borran, se archivan)                         │
└─────────────────────────────────────────────────────────┘

Operacion 1 — Deteccion

El script recorre las entradas nuevas de las ultimas 24h y las agrupa por (skill, action). Si un par (skill, action) aparece dos o mas veces, pasan dos cosas:

  • Se incrementa un contador recurrence_count en todas las entradas del cluster.
  • Se da un boost al salience de esas entradas. Lo recurrente importa mas que lo aislado.
Por que recurrencia = salience

Un error que ocurre una vez puede ser mala suerte. El mismo error tres veces es un pattern. Los patterns merecen salir primero durante el retrieval porque tienen mas probabilidad de volver a pasar.

Operacion 2 — Promocion

Este es el paso que convierte episodic en semantic. El script filtra las entradas con salience >= 7.0 (umbral configurable) y las escribe como lecciones en memory/semantic/LESSONS.md.

Pseudocodigo de la promocion
# Ilustrativo. El auto_dream.py real hace esto y mas.
for entry in recent_entries:
    if entry.salience >= PROMOTION_THRESHOLD:
        lesson = distill(entry.reflection)
        append_to("memory/semantic/LESSONS.md", lesson, area=entry.skill)
        entry.promoted = True
Lo clave: no tenes que curar a mano

La promocion es automatica. El umbral y los pesos de salience son configurables, pero una vez los ajustas, tu LESSONS.md crece solo. Eso es lo que convierte al agente en un sistema que compone con el tiempo en vez de resetearse cada semana.

Operacion 3 — Decay + archive (el circuit breaker)

La tercera operacion es la mas importante para la salud del sistema a largo plazo, y la que mas gente olvida implementar. El algoritmo: 

para cada entry en AGENT_LEARNINGS.jsonl:
  si age_days > 90  y  salience < 2.0:
      mover entry a episodic/snapshots/YYYY-MM/batch-N.jsonl
      remover entry del archivo activo
      git commit "archive: N entries from <rango de fechas>"

Dos reglas que parecen obvias pero no lo son:

  • Se archiva, no se borra. Todo lo que sale del archivo activo va a snapshots/ y se commitea a git. Siempre podes ir a buscarlo.
  • Solo se toca lo viejo Y poco relevante. Una entrada de hace 2 anos con salience 9 se queda. Una entrada de hace 1 mes con salience 1 tambien (todavia es joven). El decay cortea solo el ruido antiguo.
Por que el decay es un circuit breaker

Sin decay, una bad lesson (una generalizacion equivocada) se queda para siempre, y peor: alimenta skills que generan mas episodios apoyados en esa leccion mala. El decay rompe ese loop de amplificacion de errores. Es lo que te separa de un agente que se convence de teorias conspirativas propias.

Tambien archiva WORKSPACE stale

Como bonus, el mismo cron chequea la working memory:

  • Si memory/working/WORKSPACE.md no fue tocado en 2+ dias, lo archiva a episodic/snapshots/ y lo deja vacio.

Esto cubre el caso clasico en que te olvidaste de cerrar una tarea y el agente se despierta 3 dias despues creyendo que sigue en ella (el bug del stale workspace que vimos en la pagina 00 del modulo).

El cron real

crontab — una linea
0 3 * * * cd /path/to/project && python .agent/memory/auto_dream.py >> .agent/memory/dream.log 2>&1

Lectura: a las 3:00 de cada dia, corre auto_dream.py y appendea stdout + stderr al log. Nada exotico. Un cron simple y un script que hace las 3 operaciones.

Para chequear que el ciclo esta corriendo sano:

Check rapido del log
tail -5 .agent/memory/dream.log
# dream cycle 2026-04-14 03:00:02: promoted 2, decayed 7, kept 31
# dream cycle 2026-04-15 03:00:01: promoted 1, decayed 4, kept 33
# dream cycle 2026-04-16 03:00:03: promoted 3, decayed 12, kept 30
# dream cycle 2026-04-17 03:00:02: promoted 0, decayed 5, kept 32

Tres numeros para interpretar: promoted = cuantas lecciones subieron a semantic, decayed = cuantas viejas se archivaron, kept = cuantas se quedan en episodic activo. Si promoted es 0 dias seguidos, el umbral esta muy alto; si decayed crece sin parar, estas generando mucho ruido.

git log de memory/ = autobiografia del agente

El detalle que casi nadie usa

Si committeas memory/ al repo (cosa que deberias), el git log --oneline memory/ se convierte en la autobiografia del agente: cuando aprendio que, cuando revisitaste una decision, cuando archivaste ruido. Con un git blame sabes exactamente de que episodio salio cada leccion. 

$ git log --oneline memory/ | head
a3f9b1c archive: 5 stale entries from 2026-01
7e2d4a8 promote: "Guardar code_verifier ANTES del redirect"
c11ab02 decision: session store -> Redis (ver DECISIONS.md)
9fe0d57 archive: WORKSPACE.md (stale 3 dias) -> snapshots/
2b6c3e1 promote: "Nunca force-push a branch compartido"

El ciclo del dia, visto completo

   DIA                                NOCHE (3am)                     MANANA

┌────────────────────┐               ┌─────────────────────┐           ┌───────────────────┐
│ agente trabaja:    │               │ dream cycle corre:  │           │ agente retoma con:│
│  - tool calls      │  ───────────► │  - detecta patterns │  ───────► │  - episodic mas   │
│  - decisiones      │   logs crudos │  - promueve         │  consolid │    salience-scored│
│  - fallos          │   acumulan    │  - decay + archive  │   ado     │  - LESSONS nuevas │
│  - reflections     │               │  - git commit       │           │  - WORKSPACE limpio│
└────────────────────┘               └─────────────────────┘           └───────────────────┘

De nuevo, la idea clave: la memoria no es un almacen, es un proceso. Episodic captura crudo, dream cycle consolida, semantic guarda las abstracciones, working mantiene el estado vivo. Quitale el dream cycle y el sistema entero se pudre en semanas.

Mini-resumen

Cuando corre
Cron 3am

Una vez por dia. Asincrono, no bloquea al agente en horas de trabajo.

Que hace
Detect / promote / decay

Tres operaciones, en ese orden: detecta patterns, promueve a semantic, archiva lo viejo e irrelevante.

Por que importa
Circuit breaker

El decay es lo que evita que lecciones malas se amplifiquen. Sin eso, el agente se convence de sus propios errores.
