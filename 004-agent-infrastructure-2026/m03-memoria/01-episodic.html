Episodic memory: que paso en runs previos

La memoria episodica es el log de decisiones, tool calls, fallos y reflexiones. Suena a historial pero no lo es: los episodios recuperados funcionan como precedentes concretos, y ahi esta su valor real.

Mas que un log

Un log normal te dice que paso. La episodic memory hace algo distinto: te deja recuperar episodios pasados cuando estas por repetir una accion parecida. No es lectura lineal, es lookup selectivo.

Imaginate a un agente que va a correr git push --force. Antes de ejecutar, el retrieval le devuelve dos episodios parecidos: en uno funciono, en otro arruino el branch compartido. Con ese precedente a mano, el agente deja de actuar a ciegas.

Por eso la episodic memory no es un archivo de museo. Es la base de datos que el agente consulta cada vez que esta por hacer algo no trivial.

El archivo canonico: AGENT_LEARNINGS.jsonl

memory/
├── working/
├── episodic/
│   ├── AGENT_LEARNINGS.jsonl    ← append-only, un JSON por linea
│   └── snapshots/              ← archivado por dream cycle (M03-03)
├── semantic/
└── personal/

.jsonl significa JSON Lines: cada linea es un objeto JSON completo. El formato es append-only — nunca se edita una linea vieja, solo se agregan nuevas. Tres razones por las que esto importa:

Concurrencia
Dos agentes a la vez

Escribir al final del archivo es atomico. Sub-agentes paralelos no se pisan entre si.

Auditoria
Historia inmutable

Nunca reescribis el pasado. El git log del archivo es la narrativa completa de lo que aprendio el agente.

Parsing
Linea = registro

Podes leer las ultimas N entradas con un tail, o indexarlas en una DB vectorial sin parser complejo.

Campos clave de cada entrada

Una entrada bien formada tiene siempre los mismos campos. Esto permite filtrar, scorear y buscar sin sorpresas.

Shape de una entrada (ilustrativo)
{
  "timestamp":    "2026-04-13T10:42:11Z",
  "skill":        "oauth_flow",
  "action":       "implement_pkce",
  "result":       "success",
  "pain_score":   3,
  "importance":   7,
  "reflection":   "PKCE needs code_verifier saved BEFORE redirect, not after",
  "context":      { "repo": "api-gateway", "branch": "feat/login" }
}
  • timestamp: ISO 8601, UTC. Base de cualquier ordenamiento temporal.
  • skill: nombre del skill que estaba corriendo (conecta con M04).
  • action: verbo especifico dentro del skill.
  • result: success, failure, blocked.
  • pain_score (1–10): cuanto dolio. Alto = errores caros o bloqueos largos.
  • importance (1–10): cuanto cambia futuras decisiones.
  • reflection: 1 linea. Lo que el agente aprendio. No es el log del stacktrace.
  • context: objeto libre con metadata (repo, file, user, flags).

Dos episodios reales, lado a lado

La forma mas facil de entender el valor de estos campos es mirar dos episodios que se repiten en casi cualquier setup:

Episodio A — PKCE success
{
  "skill": "oauth_flow",
  "action": "implement_pkce",
  "result": "success",
  "pain_score": 3,
  "importance": 7,
  "reflection":
    "Guardar code_verifier
    ANTES del redirect"
}
Episodio B — force-push blocked
{
  "skill": "git_ops",
  "action": "force_push_main",
  "result": "blocked",
  "pain_score": 9,
  "importance": 10,
  "reflection":
    "Nunca force-push a
    branch compartido"
}

El episodio B tiene pain_score: 9 y importance: 10. Eso no es decoracion — determina con que frecuencia ese episodio va a emerger cuando el agente busque precedentes en el futuro.

El rol del salience scoring

Si tenes 40.000 entradas en AGENT_LEARNINGS.jsonl, cargarlas todas en el context window es imposible. Por eso al recuperar episodios se aplica un salience score (ver M02: Context budget). 

salience(e) = similarity(query, e)      ← que tan relevante
            + w1 * pain_score(e)      ← que tan doloroso fue
            + w2 * importance(e)      ← que tanto cambia decisiones
            - w3 * age_days(e)         ← decay temporal suave

Traducido: un episodio con mucho dolor emerge mas seguido que uno trivial, aunque ambos sean semanticamente parecidos a la query. Es el equivalente a como tu cerebro recuerda con nitidez el dia que te quemaste con la plancha y con niebla el dia que doblaste las remeras.

Quien escribe las entradas: los hooks

Esperar que el agente se acuerde de loguear cada accion es utopico. La escritura se delega a hooks que el harness dispara automaticamente en dos momentos clave (hablamos mas de hooks en M04). 

       skill corre                              agente toma decision
            │                                          │
            ▼                                          ▼
    ┌───────────────┐                           ┌───────────────┐
    │ exit code 0   │                           │ exit code ≠ 0 │
    └───────┬───────┘                           └───────┬───────┘
            │                                          │
            ▼                                          ▼
    hook: post_execution                      hook: on_failure
            │                                          │
            ▼                                          ▼
    append entry a AGENT_LEARNINGS.jsonl (con result, pain, importance)
  • post_execution: corre despues de cualquier skill. Loguea result, tiempo tomado, context minimo.
  • on_failure: corre cuando un skill falla. Calcula pain_score (cuanto recurso se gasto, cuanto contexto se tiro) y pide al agente una reflection de una linea.
La reflection es el diamante

Lo que realmente convierte a este archivo en algo util es la reflection. Un stacktrace plano no sirve: no se puede generalizar. Una linea corta que dice "Nunca force-push a branch compartido" si. Esa linea es lo que despues se promueve a semantic memory (proxima pagina).

Episodic vs. semantic: la diferencia que todo el mundo mezcla

Resumen en dos frases

Episodic dice: esto paso este dia. Un hecho especifico atado a un timestamp, un repo, una task.
Semantic dice: esto tiende a pasar. Una generalizacion que ya no depende del cuando ni del donde.

Fijate que no es una diferencia de granularidad. Es una diferencia de funcion. La misma reflection puede vivir primero en episodic (1 ocurrencia puntual) y luego, tras repetirse, migrar a semantic. Eso es exactamente lo que hace el dream cycle que vemos en la pagina 03 de este modulo.

Mini-resumen

Archivo
AGENT_LEARNINGS.jsonl

Append-only. Un objeto JSON por linea. Lo escriben los hooks, no el agente a mano.

Campos
pain + importance

Esos dos numeros son los que hacen que el retrieval priorice lo doloroso y lo decisivo por sobre lo trivial.

Funcion
Precedentes

Los episodios recuperados se usan como ejemplos concretos. No son historial pasivo: son input activo a la proxima decision.
