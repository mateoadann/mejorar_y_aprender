Semantic memory: abstracciones que sobreviven

Patrones y heuristicas que se sostienen a traves de tareas distintas. No atadas a una fecha ni a un repo especifico: son reglas que el agente aprendio y ahora usa siempre.

De "esto paso" a "esto tiende a pasar"

La diferencia entre episodic y semantic no es de granularidad, es de funcion. Episodic te dice: "el 13 de abril 2026, al implementar PKCE en api-gateway, guardamos el code_verifier despues del redirect y fallo". Semantic te dice: "siempre guarda el code_verifier antes del redirect".

La primera es un hecho. La segunda es una regla. Y esa regla ya no necesita contexto temporal: aplica hoy, manana y en el proximo proyecto. 

EPISODIC                                          SEMANTIC
(muchas entradas crudas)                          (pocas lecciones destiladas)

┌──────────────────┐
│ PKCE fallo 13/4  │  ───┐
├──────────────────┤     │
│ PKCE fallo 27/4  │  ───┤     filtrado +           ┌────────────────────────┐
├──────────────────┤     ├──► consolidacion  ────► │ "Guardar code_verifier │
│ PKCE fallo 03/5  │  ───┤                          │  ANTES del redirect"   │
├──────────────────┤     │                          └────────────────────────┘
│ ...              │  ───┘
└──────────────────┘

La flecha del medio — filtrado + consolidacion — es el dream cycle. Lo vemos en la proxima pagina. Por ahora basta saber que semantic memory no se llena a mano: emerge de episodic.

Los tres archivos canonicos

Semantic memory no es un archivo, son tres. Cada uno captura un tipo distinto de abstraccion: 

memory/
├── working/
├── episodic/
├── semantic/
│   ├── LESSONS.md           ← heuristicas tecnicas agrupadas por area
│   ├── DECISIONS.md         ← decisiones arquitectonicas con rationale
│   └── DOMAIN_KNOWLEDGE.md  ← hechos del dominio de negocio
└── personal/
LESSONS.md
Heuristicas

"Al hacer X, tene cuidado con Y." Agrupadas por area: API design, Git workflow, Testing, Performance.

DECISIONS.md
Rationale

"Elegimos Redis sobre in-memory porque [...]." Decision + alternativas + status.

DOMAIN_KNOWLEDGE.md
Hechos de negocio

Vocabulario, reglas del producto, limites conocidos. Lo que vale para tu dominio especifico.

LESSONS.md por adentro

Una leccion es una heuristica condensada. Se escribe en imperativo y cabe en una o dos lineas. Se agrupan por area para poder cargar solo la seccion relevante al context window.

memory/semantic/LESSONS.md
# API design
- Validar schemas en el borde, no adentro del handler
- 4xx si el error es del cliente, 5xx si es tuyo. Nada de 200 con "error" en el body
- Los IDs internos no se exponen: siempre un UUID publico separado

# Git workflow
- Nunca force-push a branch compartido
- Commit messages en ingles, imperativo, <72 chars en la primera linea
- Rebase en features personales, merge en branches que vio otro

# Testing
- Test primero el happy path, despues los bordes. Nunca al reves
- Mock solo lo que cruza el network. Logica propia se testea de verdad
- Si un test es flaky 2 veces, se arregla o se borra. No se ignora

# Performance
- Medir antes de optimizar. Siempre.
- N+1 queries son el bug mas comun y el mas facil de pasar por alto

Notar tres cosas: (1) cada leccion es una linea, no un parrafo, (2) el tono es directo (imperativo), (3) las categorias son anchas — API design, no "API design for POST /login". La especificidad la pone episodic.

DECISIONS.md: el ADR lite del agente

Si trabajaste con ADRs (Architecture Decision Records), esto va a sonarte. Si no, la idea es simple: cada decision arquitectonica no trivial queda documentada con por que se tomo y que alternativas se descartaron.

memory/semantic/DECISIONS.md (ejemplo real)
# 2026-04-13 — Session store para auth tokens

Decision: Redis con TTL de 24h.

Rationale:
  El store in-memory nos hacia perder sesiones al rotar pods, y
  una race condition entre dos replicas nos dejaba tokens
  duplicados por ~200ms. Redis resuelve las dos cosas.

Alternativas consideradas:
  - In-memory (status-quo): descartado por race conditions
  - Postgres: descartado por latencia (p99 > 40ms)
  - DynamoDB: descartado por costo en este volumen

Status: Active
Revisit: cuando pasemos de 100 req/s sostenidos
Por que salva la vida

Sin DECISIONS.md, en 3 meses alguien (un agente o un humano nuevo) va a proponer volver a in-memory "porque es mas simple". Con DECISIONS.md, el rationale esta a un grep de distancia. No se re-debate lo que ya se debatio.

Los campos fijos son: decision (que se eligio), rationale (por que), alternativas consideradas (que se descarto y por que), status (Active o Revisit con condicion), revisit (el trigger que obliga a reconsiderar).

Semantic vs. personal: la ultima capa

Hay una cuarta capa — personal memory — que conviene separar desde el principio. Es donde vive PREFERENCES.md: preferencias estilisticas, convenciones del equipo, cosas que no son verdades universales sino como a nosotros nos gusta trabajar.

Semantic
Reglas universales

"Nunca force-push a branch compartido" — vale aca, vale afuera, vale en cualquier equipo.

Personal
Preferencias

"Preferimos functional patterns sobre OOP" — contexto, no regla. Vale en este proyecto/equipo, no necesariamente en otro.

El error: mezclar los dos

Si metes "preferimos TypeScript estricto" en LESSONS.md, tu agente va a empezar a pelear con codigos JS perfectamente razonables afuera de este repo. Es contexto, no heuristica. Va a PREFERENCES.md.

Como llega algo a semantic: la promocion

La pregunta obvia: quien escribe LESSONS.md?

Respuesta corta: no vos. La mayoria de las entradas llegan por promocion automatica desde episodic, ejecutada por el dream cycle (M03-03). La idea: 

episodic                    dream cycle                 semantic
(raw log)                   (cron 3am)                  (destilado)

┌──────────────┐           ┌────────────────┐          ┌──────────────┐
│ entrada #412 │           │ salience >= 7  │          │ LESSONS.md   │
│ salience 8.2 │  ───────► │  → promover    │  ──────► │  + nueva     │
└──────────────┘           └────────────────┘          │    leccion   │
                                                        └──────────────┘

Podes tambien editar los archivos a mano cuando queres sembrar conocimiento de entrada (por ejemplo convenciones del equipo que no se van a aprender por dolor). Pero el motor principal de crecimiento de semantic es la promocion automatica desde episodic.

Mini-resumen

Que es
Reglas destiladas

Heuristicas, decisiones y conocimiento de dominio que no caducan con la tarea.

Archivos
LESSONS / DECISIONS / DOMAIN

Tres archivos, tres funciones: heuristicas, ADRs lite, hechos de negocio.

Como se llena
Promocion desde episodic

El dream cycle filtra las entradas con salience alto y las promueve sin que tengas que curar a mano.
