Los 6 feedback loops que hacen compounding

En M00 sembramos la idea: el agente mejora solo porque los 3 modulos se alimentan entre si. Aca desarmamos los 6 loops concretos que hacen que la cosa componga. Cada uno tiene una direccion, un trigger, un efecto — y todos juntos forman el circuito self-reinforcing del agente.

La idea en una imagen

Memoria, skills, y protocols son tres modulos. Si fueran independientes, el agente seria una suma lineal: memoria archiva, skills ejecutan, protocols gobiernan. No compone. El compounding aparece cuando los tres se escriben entre si. 

                    el circuito self-reinforcing

                      ┌─────────────────┐
                      │     MEMORY      │
                      │  (4 capas)      │
                      └───┬──────▲──────┘
                          │      │
                    loop 1│      │loop 2    loop 5│  │loop 6
                          │      │                │  │
                          ▼      │                │  │
                      ┌─────────────────┐         │  │
                      │     SKILLS      │         │  │
                      │  (self-rewrite) │         │  │
                      └───┬──────▲──────┘         │  │
                          │      │                │  │
                    loop 3│      │loop 4          ▼  │
                          │      │                   │
                          ▼      │              ┌────┴────────┐
                      ┌─────────────────────────┤  PROTOCOLS  │
                      │                         │  (hooks,    │
                      │                         │   schemas,  │
                      │                         │   permisos) │
                      └─────────────────────────┴─────────────┘

                  circuit breakers atravesando todo:
                  dream cycle decay + self-rewrite policy conservative

Seis flechas. Tres modulos. Y una caja alrededor que evita que los loops amplifiquen errores. Vamos uno por uno.

Loop 1 · Memory → Skills creation

El primer loop es el que hace que el sistema crezca. Cuando memory-manager detecta un pattern recurrente — la misma task manejada de la misma forma 3 veces o mas — triggerea skillforge para que cree un skill nuevo a partir de ese pattern. 

  LOOP 1:  MEMORY  ───►  SKILLS

  trigger:    memory-manager detecta recurrence ≥ 3
  efecto:     skillforge crea un skill nuevo con el patron
  evidencia:  AGENT_LEARNINGS.jsonl muestra 3 entries similares
              → nuevo archivo en /skills/NUEVO.md

Loop 2 · Skills → Memory

Espejo del loop 1. Cada vez que un skill se ejecuta, el post_execution hook lo logea a episodic memory con un pain_score (o success_score). Exito o fracaso, el resultado queda grabado. 

  LOOP 2:  SKILLS  ───►  MEMORY

  trigger:    cada vez que un skill termina de ejecutar
  efecto:     append a AGENT_LEARNINGS.jsonl con pain/success
  evidencia:  {"skill": "api-endpoint-builder", "outcome": "ok",
               "pain": 2, "tags": ["fastapi","routes"]}

Loop 3 · Skills → Protocols

Cuando un skill necesita llamar a una tool externa (git, deploy, API), pasa por el pre_tool_call hook. El skill dice "que hacer", el protocol gobierna "como, y si esta permitido". Esta separacion es lo que permite que los skills sean reusables entre proyectos con reglas de seguridad distintas. 

  LOOP 3:  SKILLS  ───►  PROTOCOLS

  trigger:    skill invoca una tool (write_file, run_cmd, api_call)
  efecto:     pre_tool_call valida permisos/constraints del protocol
  evidencia:  intento de `rm -rf` → bloqueado por permissions.md
              intento de `git push --force` → requiere aprobacion

Loop 4 · Protocols → Skills

La vuelta: cuando tenes typed tool schemas bien escritos, es mucho mas facil escribir skills que los usen correctamente. El schema dice exactly que args toma, que preconditions, que constraints, que puede y no puede hacer. Writing schemas primero, skills despues, > writing skills primero y descubriendo las reglas despues. 

  LOOP 4:  PROTOCOLS  ───►  SKILLS

  trigger:    existe un typed schema para una tool
  efecto:     skillforge usa ese schema para escribir skills
              que llaman la tool correctamente
  evidencia:  skill nuevo menciona args exactos + preconditions
              del schema, sin adivinar

Loop 5 · Memory → Protocol routing

Este es mas sutil. La episodic memory registra donde fallan las cosas. Si un API endpoint especifico falla frecuentemente, el agente puede aprender a preferir una ruta alternativa — sin que vos tengas que cambiar el skill, sin que cambies el schema. El routing del protocol se adapta a los patrones de fallo que la memoria tiene grabados. 

  LOOP 5:  MEMORY  ───►  PROTOCOLS

  trigger:    episodic muestra failure rate alto en API X
  efecto:     protocol prefiere ruta alternativa Y en siguientes calls
  evidencia:  policy.yaml o routing hint actualizado por self-rewrite
              basado en entries de LESSONS.md

Loop 6 · Protocols → Memory

Y el cierre del circuito. Tool outputs, approval events, error payloads — todos vienen por el protocol layer y todos quedan loggeados a episodic via post_execution. Sin los hooks del protocol, la memoria no tendria evidencia para aprender. El protocol layer es lo que produce la evidencia que la memoria almacena. 

  LOOP 6:  PROTOCOLS  ───►  MEMORY

  trigger:    cada tool call / error / approval event
  efecto:     post_execution hace append a episodic con metadata
  evidencia:  event log = fuente de todo learning posterior
              sin esto, la memoria no sabria de que alimentarse

Por que compone: reinforcement

Los 6 loops no son independientes — forman un ciclo. Cada uno alimenta al siguiente, y el resultado es un sistema self-reinforcing: 

  better memory
       │
       ▼
  better skill creation (loop 1)
       │
       ▼
  richer execution traces (loop 2, loop 6)
       │
       ▼
  better memory (de vuelta al principio)
       │
       ▼  cada vuelta, el sistema es un poquito mejor

No requiere setup manual despues de la configuracion inicial. Una vez que los hooks estan enchufados y los skills self-rewriteables ya existen, el agente mejora solo con cada sesion. Eso es lo que da ese salto de "tool que me ayuda" a "sistema que me conoce y mejora".

Por que puede salir mal: amplificacion

El mismo circuito que compone bien puede amplificar errores. Si entra una bad lesson en semantic memory (algo mal aprendido, una correccion equivocada), el sistema la puede reforzar en vez de corregirla: 

  CIRCUITO DE AMPLIFICACION

  bad lesson en LESSONS.md
       │
       ▼
  influye la creacion del proximo skill
       │
       ▼
  skill ejecuta bajo esa leccion equivocada
       │
       ▼
  genera mas entries episodicos confirmando la (mala) leccion
       │
       ▼
  memory-manager refuerza la leccion (la ve mas)
       │
       ▼  cada vuelta, el error se atrinchera
Esto pasa en la realidad

No es un escenario teorico. Si no hay circuit breakers, sucede: una correccion mal dada un dia se convierte en lesson, que sesga un skill, que genera entries consistentes con esa leccion, y a la semana el agente esta haciendo algo mal de forma sistematica — con documentacion interna que "confirma" que eso es lo correcto.

Los circuit breakers que previenen amplificacion

Dos componentes del stack estan especificamente para cortar la amplificacion:

Circuit breaker 1
Dream cycle decay

El dream cycle (M03) no solo consolida — tambien decae entries viejas con salience baja. Lecciones que no se refuerzan concretamente se van apagando. Si una bad lesson es artificial, sin evidencia real, el decay la mata.

Circuit breaker 2
Self-rewrite conservative

Los hooks de self-rewrite (M04) son conservadores: no modifican un skill con una sola lesson nueva, requieren confirmacion en multiples ocurrencias. Eso impide que un incidente aislado reescriba el skill.

Parte manual que no automatizes

Ademas de los dos breakers automaticos, hay una tercera capa que tiene que ser humana: revisar LESSONS.md cada cierto tiempo (2-3 semanas), tirar las que no aplican mas, y usar git revert si hace falta. Esa revision no se automatiza.

Resumen: que es lo que hace el compounding

Loop Direccion Trigger Efecto
1 Memory → Skills Pattern recurrente ≥ 3 Crea skill nuevo
2 Skills → Memory Skill ejecuta Append a episodic
3 Skills → Protocols Skill invoca tool Pre-hook valida
4 Protocols → Skills Existe typed schema Skills mejor escritos
5 Memory → Protocols Failure rate alto en X Routing adaptado
6 Protocols → Memory Cada event / error Evidencia logged

Los 6 loops son el motor. Los circuit breakers son el freno. Los dos juntos hacen que el agente aprenda sin volverse inestable. Si falta cualquiera de los dos lados, tenes un sistema que o no aprende nada, o aprende mal.
