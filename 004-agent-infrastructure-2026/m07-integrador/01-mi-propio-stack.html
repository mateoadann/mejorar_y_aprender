Armá tu propio stack: checklist

Ya tenes el modelo mental. Ahora, ¿que haces el lunes a la manana si queres empezar? Este es el orden de implementacion que te ahorra semanas de frustracion — y los anti-patterns que te van a tentar y deberias evitar.

Primero: arranca THIN

Tentacion enorme: armar todo el stack de una. Memoria en 4 capas + skills + protocols + knowledge graph + dream cycles. No lo hagas.

El consejo que funciona: empezar con la version minima (thin). Heartbeat simple + permissions.md. Unas semanas corriendo asi te van a mostrar exactamente donde poner los siguientes guardrails. Si arrancas con todo, no sabes cual componente es el que te ayuda y cual te estorba.

Thin stack = suficiente para empezar

AGENTS.md (30 lineas) + permissions.md (lista de comandos permitidos/prohibidos) + 2 skills basicos. Eso alcanza para sentir la primera semana. El resto se construye sobre evidencia de uso real.

Timeline realista: 10 semanas

Ojo con la expectativa. El primer mes se siente frustrante. El segundo mes empieza a pagar. Solo al tercer mes ves el compounding que el curso te prometio. Este es el timeline que el autor del articulo documento, y coincide con lo que ves en la practica: 

  semana 1-2 · frustrante
                      ─────────────
  "me olvida todo lo que discutimos ayer"
  archivos de memoria estan pero no se leen consistentemente
  hooks no enganchan bien, sesgos en AGENTS.md
  sentis que perdiste tiempo


  semana 2-4 · empieza a clickear
                      ─────────────────────
  agregas memory-manager como skill
  antes de decisiones, chequea LESSONS.md
  la continuidad entre sesiones empieza a aparecer
  las correcciones que diste ayer, hoy se aplican solas


  semana 4-5 · empieza a editarse solo
                      ─────────────────────────
  abris KNOWLEDGE.md y encontras lineas que VOS NO ESCRIBISTE
  el dream cycle consolido un pattern que notaste de pasada
  → el sistema arranca a evolucionar


  semana 6-8 · compone
                      ────────────
  tenes skills escritos por skillforge que ni te acordas
  el agente usa lessons que surgieron de fallos que olvidaste
  la suma empieza a ser mayor que las partes


  semana 10 · hit the wall
                      ────────────────
  30 skills + LESSONS.md fat + tool schemas
  = 90K tokens cargados antes de pensar
  algo tiene que ceder
  → progressive disclosure (M02-03) rescata
  _index.md + lazy-loading de skills por trigger
El wall de la semana 10

Este wall es esperable, no un error de diseno. Es la senal de que el stack crecio bien: tenes tantos assets que ahora cargar todo es caro. Progressive disclosure (cargar solo el header y lazy-loadear el detalle) te saca del pozo. Si arrancaste con progressive disclosure desde day 1 sin necesitarlo, estas preoptimizando.

Checklist de implementacion (orden recomendado)

Estos son los 14 items que moven la aguja. Orden sugerido — no tenes que hacerlos todos juntos, pero esta secuencia minimiza frustracion.

1 · Decidir harness
Claude Code / Cursor si queres something off-the-shelf, o custom harness menos de 200 lineas. Grande simple le gana a framework gigante complicado.
2 · Folder structure .agent/
Subdirs: memory/, skills/, protocols/, hooks/. No flat folder, siempre separado por concern.
3 · AGENTS.md raíz
Root config. Bajo 30 lineas. Lista donde esta cada cosa. Zero logica, solo indice.
4 · memory/personal/PREFERENCES.md
Tu stack (Angular, Python, Rioplatense Spanish, etc), tu style, tus convenciones. Esto nunca cambia con el tiempo.
5 · protocols/permissions.md day 1
No lo pospongas a la semana 6. Ese fue el error del autor y le costo varios incidentes. Buckets: always allowed / requires approval / never allowed / approved domains.
6 · memory/working/WORKSPACE.md template
Template de working memory. Se reescribe cada sesion. Es el "que estabamos haciendo" que hace que la resumption funcione.
7 · memory/episodic/AGENT_LEARNINGS.jsonl vacio
Archivo vacio. Lo empieza a llenar el post_execution hook. Append-only, jamas editas a mano.
8 · 2-3 skills iniciales con self-rewrite hooks
No 20. Dos o tres, de tareas que haces seguido. Cada uno con self-rewrite hook al final. Mejor tres excelentes que veinte mediocres.
9 · memory-manager skill day 1
Esto lo hubiese ahorrado 3 semanas al autor. Arrancar con memory-manager desde day 1 hace que el agente empiece a leer la memoria desde la primera sesion, no desde la cuarta.
10 · skillforge skill
El skill que crea skills cuando memory-manager detecta pattern recurrente. Es meta, pero es lo que hace que el sistema crezca solo.
11 · Hooks: pre_tool_call, post_execution, on_failure
Los 3 hooks minimos. pre_tool_call valida permisos. post_execution logea a episodic. on_failure dispara con pain=8 y threshold 3 en 14d (si mismo fallo pasa 3 veces en 2 semanas, promote a LESSONS.md).
12 · Tool schemas para tools criticas
Empezá por las destructivas: git, deploy, DB writes. Si no hay schema, el skill tiene que adivinar args — y adivina mal con cosas que rompen.
13 · auto_dream.py + cron 3am
Script que corre el dream cycle todas las noches. Consolida episodic → semantic, decae entries viejas, detecta patterns para skillforge.
14 · Git init en .agent/
Version control de TODO el stack. Memoria, skills, protocols, hooks — todo en git. Asi podes hacer git revert de una lesson mal aprendida sin perder el resto.

5 anti-patterns a evitar

Estos los vas a encontrar. Todos parecen buenas ideas hasta que explotan:

Anti-pattern 1

Context budget bloat

90k tokens cargados antes de empezar a pensar. Todos los skills + LESSONS.md + schemas se leen al inicio.

Solucion: progressive disclosure (M02-03). Truncar LESSONS.md a top-N por salience. Top-k de episodic por similitud, no historico completo.

Anti-pattern 2

Stale skills

La API cambio hace 3 meses, el skill sigue con las instrucciones viejas. Se ejecuta y rompe cosas.

Solucion: manifest con version+date por skill. Flaggear automaticamente los que no se tocaron en 60+ dias. Review manual periodico.

Anti-pattern 3

Unsafe composition

2 skills cada uno seguro, combinados hacen algo destructivo. Skill A: "lista files para limpiar". Skill B: "borra files de la lista". Juntos: borro prod.

Solucion: constraints en pre_tool_call hook del protocol, no adentro de los skills. Que sea imposible combinarlos sin validacion central.

Anti-pattern 4

Literal execution

El skill tiene 12 pasos procedurales, el agente los sigue aunque no apliquen a la situacion actual. Ciega y ruidosa.

Solucion: destinations and fences, no driving directions (M04-00). Describi el destino, no la ruta paso-a-paso.

Anti-pattern 5

Error amplification loops

Una bad lesson entra a LESSONS.md. Sesga el proximo skill. El skill genera mas entries consistentes. La leccion se refuerza. El sistema se enquista en el error. Esto es lo que vimos como amplificacion en la pagina anterior.

Solucion (tres capas):
— Dream cycle decay (automatico: mata entries sin refuerzo).
— Self-rewrite conservative (skill no se reescribe con una sola entry).
— Review manual de LESSONS.md cada few weeks (no automatizar esta parte — si el sistema se auto-corrige puede tambien auto-equivocarse).
— Git revert si hace falta volver atras.

Cierre: owneas tu memoria, owneas tus skills

Hay una idea que quiero que te lleves, porque es la que distingue un setup que sobrevive a cambios de modelo de uno que no:

El modelo es swap. La memoria y los skills no.

Cuando salga un modelo mejor (y va a salir), lo swapeas. Cambias una linea de config y lo nuevo entra. Los skills y la memoria no los podes reemplazar asi. Ellos encodan tus mistakes, tus decisions, tu specific way of working.

Own your memory. Own your skills. Keep them in plain text files and in git. Que sean tuyos, no de la plataforma.
