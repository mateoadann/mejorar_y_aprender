Context rot y la ventana de 1M tokens

Tener 1 millon de tokens de ventana no significa que podes llenarla. A medida que crece el context, el modelo empieza a rotarse: la atencion se spread mas fina, el contenido viejo distrae de la task actual, y la inteligencia se degrada justo antes del momento en que tenes que compactar. Aca lo vemos con curvas.

Que es el context window, de verdad

Mucha gente piensa el context window como un cajon: le tiras cosas adentro y mientras quepan, todo bien. Esa imagen es enganiosa. El context window es todo lo que el modelo "ve" cuando esta generando la proxima respuesta. No es almacenamiento pasivo, es input activo para cada token que produce.

En cada turno de Claude Code, el contexto incluye: 

  ┌─────────────────────────────────────────────────────────┐
  │  context window (1M tokens en Claude Code)            │
  │                                                         │
  │    system prompt          (reglas del harness)          │
  │    +                                                    │
  │    conversacion entera    (vos + el modelo, todo)       │
  │    +                                                    │
  │    tool calls             (cada vez que leyo un file)   │
  │    +                                                    │
  │    tool outputs           (contenido de esos files)     │
  │    +                                                    │
  │    archivos leidos        (quedan en el stream)         │
  │    +                                                    │
  │    tu mensaje actual                                    │
  │                                                         │
  │  → todo esto entra en cada generacion, no solo el       │
  │    mensaje mas reciente                                 │
  └─────────────────────────────────────────────────────────┘
Conexion con M02

En M02 vimos el context budget como problema de arquitectura en un harness custom. Aca vemos el mismo problema, pero desde el lado del usuario de Claude Code. Mismo fenomeno, distinta palanca para controlarlo: alla era build_context(), aca es session management.

La curva del context rot

Context rot = la performance del modelo degrada a medida que crece el context. No es un corte binario ("hasta aca bien, despues mal"). Es una curva. La atencion se reparte entre mas y mas tokens, y contenido viejo o irrelevante empieza a competir con la task actual por ese "presupuesto de atencion".

Para un modelo de 1M como Claude Code, el rot empieza a morder alrededor de 300-400k tokens. No es fast rule — depende mucho de la task. Una busqueda simple sigue funcionando bien con 500k de context. Un debugging que necesita razonar cruzando 8 archivos lejanos, ese se empieza a confundir mucho antes.

Por que degrada (el mental model)

1
Atencion spread

El modelo tiene que atender a mas tokens simultaneamente. La atencion por token baja. Tokens importantes de hace 200k posiciones compiten con ruido reciente.

2
Contenido stale

Archivos que leyo hace 50 turnos y que ya no aplican siguen en el contexto. El modelo no sabe que descartar, entonces los considera todos igual de relevantes.

3
Intentos fallidos

Si probo approach X y no funciono, el modelo ve todo el intento fallido + la correccion. Esa memoria de "fallo" puede sesgar la generacion siguiente hacia repetir patrones similares.

El hard cutoff y el momento mas peligroso

Cuando el context se acerca al limite del window, Claude tiene que compactar: resumir lo que vio y seguir el resto de la conversacion trabajando sobre ese resumen. Suena razonable, pero hay una trampa. 

  tokens usados   performance   decision point
  ──────────────  ───────────   ──────────────────────────────────
  50k             maxima        mandale con todo
  300k            buena         ok, seguis normal
  500k            baja          pensar en /compact proactivo
  800k            mala          el modelo ya esta confundido
  950k            pesima        auto-compact dispara → summariza
                                (pero en su peor momento)
La paradoja del compact

Cuando el auto-compact se dispara cerca del limite, el modelo esta en su punto menos inteligente de toda la sesion. Y justo en ese punto le pedis que decida que fue importante y que se puede tirar. Esa decision la toma un modelo cognitivamente degradado. Por eso el summary sale mal: dropea cosas que ibas a necesitar, y preserva ruido que ya no importa.

La leccion practica es simple: no esperes a que el auto-compact se dispare por vos. Compactar proactivamente cuando todavia estas en la zona verde te da un summary mucho mejor — porque el modelo todavia piensa claro.

Dos mitos que conviene tirar

Mito

"Tengo 1M de ventana, puedo meter todo el codebase y trabajar en una sola sesion gigante."

Tecnicamente entra. Practicamente, a los 400k tokens el modelo empieza a confundirse. No vas a darte cuenta hasta que te devuelva una respuesta que ignora un file que leyo 30 turnos atras.

Realidad

"Mas context != mas inteligencia. Mas context = mas material para distraerse."

El context window es un recurso escaso, no un regalo. Lo mejor que podes hacer es mantenerlo limpio: solo lo relevante a la task actual.

Cierre: la ventana no es un buffer, es una lupa

Pensa la ventana de 1M tokens como una lupa: cuanto mas pequeno el area que enfocas, mas nitido el detalle. Cuanto mas expandis, mas cosas ves, pero cada una con menos resolucion. El trabajo del usuario de Claude Code es decidir, turno a turno, cuanto meter bajo la lupa.

En las proximas tres paginas vemos las palancas concretas: como pensar cada turno como un branching point, cuando /rewind le gana a corregir en el mismo hilo, y como usar subagents para delegar chunks pesados sin contaminar tu context principal.

Idea para guardar

Context rot empieza mucho antes del hard cutoff. No mires el % de ventana usada — mira la calidad de las respuestas. Si el modelo esta "olvidando" cosas que dijiste hace rato, ya estas en la zona amarilla. Momento de compactar o resetear.
