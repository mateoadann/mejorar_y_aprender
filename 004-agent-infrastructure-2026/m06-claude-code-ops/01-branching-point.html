Cada turno es un branching point

Cuando Claude termina una respuesta, la mayoria de los usuarios hace una sola cosa: escriben el mensaje siguiente. Pero ahi hay 5 opciones, y elegir la correcta es la diferencia entre un context limpio y un context que se pudre. Aca desarmamos las 5 y cuando usar cada una.

Las 5 opciones al final de cada vuelta

Cada vez que el modelo termina de responder, vos tenes la decision. La mayoria elige "continue" por default. Esta bien cuando es la accion correcta; el problema es que muchas veces no lo es.

1 · Continue
El default

Mandas otro mensaje en la misma sesion. El context crece. Es lo correcto si la task actual sigue exactamente en la misma linea y lo anterior va a seguir siendo util.

2 · /rewind
esc esc

Jump back a un mensaje previo. Los mensajes posteriores (incluida la vuelta del modelo) se dropean. Retry desde ese punto con otro prompt. El context queda limpio.

3 · /clear
Reset total

Nueva sesion, con un brief destilado de lo aprendido. Vos escribis que importa llevarse. El context resultante es lo que VOS decidiste relevante.

4 · /compact
Summary

Claude resume la session y reemplaza el historial con ese summary. Seguis trabajando "sobre" la sintesis. Lossy (confias en Claude para decidir que importo) pero no te hace escribir nada.

5 · Subagent
Delegar

Spawneas un agente con context limpio para hacer un chunk (verificar algo, leer otro codebase, escribir docs). Vuelve solo con el resultado final, sin todos los tool outputs intermedios.

La regla general: nueva task, nueva sesion

Tener 1M tokens de ventana no significa que debas usarla toda. La regla default que conviene adoptar es "nueva task, nueva sesion". Cada task arranca con context limpio, y eso te ahorra mas problemas de los que crea.

Sesion monstruo (mal)

Una sola sesion de 6 horas:

Implementar auth → debuggear tests → leer docs de Stripe → refactorear carrito → escribir docs → arreglar bug en emails.

Para la hora 3 el contexto esta saturado con archivos que ya no importan, intentos fallidos, tool outputs de grep que pedis en la hora 1. El modelo empieza a "mezclar" tasks.

Una sesion por task (bien)

6 sesiones cortas:

Cada task en su sesion propia. Entre una y otra, /clear con un brief corto de lo importante que sobrevive ("estamos refactorizando carrito, el constraint es X, el approach descartado es Y").

Cada sesion arranca limpia. El modelo solo ve lo que aplica a lo que estas haciendo ahora.

La zona gris: cuando continue es lo correcto

No todo pide sesion nueva. Hay casos donde ya tenes en el context cosas que re-leer seria caro, y donde la task que sigue no es super-intelligence-sensitive.

Ejemplo clasico

Acabas de implementar un feature con Claude. Leyo 8 archivos, hizo los cambios, los tests pasan. Ahora queres docs para ese feature.

Podrias hacer /clear y arrancar fresh. Pero entonces Claude tiene que re-leer los 8 archivos → mas tokens, mas plata, mas tiempo. Y escribir docs no es tan intelligence-sensitive como el debugging.

Conclusion: continue. Seguis en la misma sesion, le pedis docs. El context usado no molesta para esa task, y te ahorras la relectura.

La heuristica es: ¿lo que sigue es cognitivamente dificil?

  • Dificil (debugging raro, razonar cross-file, arquitectura) → considera reset aunque cueste re-leer.
  • Facil (docs, rename, formateo, commit message) → continue, aprovecha el contexto que ya esta.

Mental model: cada turno es un savepoint

Si venis de jugar RPGs, pensa cada turno como un savepoint. El modelo termino, vos decidis: ¿guardo en este slot (continue), cargo un savepoint anterior (rewind), empiezo una partida nueva con solo mis items (clear), sintetizo la partida y sigo (compact), o mando un NPC que va al dungeon y me trae el loot (subagent)? 

  turno 1 → turno 2 → turno 3 → turno 4 → AHORA (decision point)
                                              │
                                              ├─► continue  (turno 5 en mismo hilo)
                                              │
                                              ├─► rewind a turno 2  (dropea 3, 4)
                                              │   → retry con mejor prompt
                                              │
                                              ├─► /clear  (nuevo hilo, brief tuyo)
                                              │
                                              ├─► /compact  (summary de 1-4, seguir)
                                              │
                                              └─► subagent  (chunk con context propio,
                                                             vuelve con resultado)
Habito que conviene entrenar

Antes de escribir tu proximo mensaje, tomate 2 segundos. Mira las 5 opciones. La mayoria de las veces continue va a ser la correcta — y esta bien. Pero entrenar el habito de pensarlo hace que el 20% de las veces que NO es continue, te des cuenta y elijas lo correcto.
