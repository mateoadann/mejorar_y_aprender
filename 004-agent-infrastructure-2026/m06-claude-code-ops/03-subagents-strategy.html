Subagents: delegation strategy

Los subagents no son un feature de paralelismo — son una forma de context management. Cuando sabes de antemano que un chunk de laburo va a generar mucho output intermedio que no vas a necesitar despues, delegas. Vuelve solo la conclusion, sin la basura.

Que es un subagent, de verdad

Cuando Claude Code lanza un subagent (via la tool Agent), ese subagent recibe su propio context window, en blanco. Hace el laburo que le delegaste, ve todos los tool outputs, toma sus decisiones, y cuando termina sintetiza el resultado y devuelve solo ese final report al parent. 

  main context (parent)
  ┌──────────────────────────────────────────────────┐
  │  sistema prompt                                  │
  │  conversacion tuya + claude                      │
  │  files leidos hasta aca                          │
  │                                                  │
  │  tu mensaje: "spawn subagent para verificar      │
  │               que la implementacion cumple       │
  │               el spec en docs/auth-spec.md"      │
  │                                                  │
  │           │                                      │
  │           ▼                                      │
  │  ┌─────────────────────────────────────────┐     │
  │  │ subagent context (FRESH)                │     │
  │  │  - sistema prompt minimo                │     │
  │  │  - instruccion concreta: verificar spec │     │
  │  │  - lee docs/auth-spec.md                │     │
  │  │  - lee src/auth/*.ts                    │     │
  │  │  - corre tests, ve output               │     │
  │  │  - mas lecturas, mas outputs            │     │
  │  │  - sintetiza: "cumple 7/8, falla en X"  │     │
  │  └────────────────┬────────────────────────┘     │
  │                   │                              │
  │                   ▼ (solo el report)             │
  │  ← result: "cumple 7/8, falla en X"              │
  │                                                  │
  │  (todos los tool outputs intermedios quedaron        │
  │   adentro del subagent, nunca llegaron al main)       │
  └──────────────────────────────────────────────────┘

Fijate lo importante: los tool outputs (contenido de archivos, output de tests, greps, etc.) jamas cruzan la frontera. Solo viene la sintesis final. Eso es lo que hace a un subagent una herramienta de context management, no de paralelismo.

El mental test: ¿voy a necesitar esto de nuevo?

La pregunta que usas para decidir si delegar o no es una sola, y es fundamental:

La pregunta

¿Voy a necesitar el tool output DE NUEVO en turnos siguientes, o solo me interesa la conclusion?

"De nuevo" → main context

Si voy a seguir trabajando con esos archivos, a referenciar ese output, a razonar cruzando esa info — mejor que Claude lo lea en main context, asi lo tiene disponible.

Ejemplo: "lee los 3 archivos del modulo de carrito que vamos a refactorear". Los vas a tocar. Que queden.

"Solo la conclusion" → subagent

Si solo me interesa el resultado (pasa/no pasa, cumple/no cumple, X archivos encontrados), todo el proceso intermedio es ruido para el main context. Delegalo.

Ejemplo: "confirma que ningun archivo del codebase importa la lib deprecated foo". Solo queres el si/no final.

Casos de uso clasicos

Claude Code puede decidir spawnear subagents automaticamente cuando detecta tareas que encajan. Pero podes (y conviene) pedirlos explicitamente cuando vos los ves venir.

Caso 1
Verify vs spec

"Spin up a subagent to verify the implementation matches the spec in docs/feature-x-spec.md". El subagent lee ambos, los compara, devuelve el diff. Vos solo recibis el report.

Caso 2
Study other codebase

"Subagent: lee ese otro codebase en ../old-project y summariza como implementaron auth. Despues implementamos algo parecido aca." El viejo codebase no queda en tu context.

Caso 3
Docs desde diffs

"Subagent para escribir docs de los cambios que acabamos de hacer, basado en el git diff". El subagent lee el diff, genera docs, vuelve con el markdown final.

Caso 4
Bulk search

"Subagent: buscame todos los lugares del codebase donde se usa la lib foo, y devuelve una lista de archivos + la lineas exactas." Tu main context no carga con 40 tool outputs de grep.

Caso 5
Research

"Subagent: investiga como hace X la libreria Y. Mira sus docs, su source, y devolveme un resumen del patron que usa." Investigacion con muchos reads, en context aparte.

Caso 6
Analysis

"Subagent: correr este script y analizar su output (que suele ser largo). Devolveme un summary de que fallo y donde." Output largo se queda en el subagent.

Cuando delegar vs cuando mantener

Situacion Mantener en main Delegar a subagent
Leer archivos que voy a editar Si — los vas a necesitar No
Validar que algo cumple un spec No Si — solo queres el pasa/no pasa
Debug iterativo de un bug Si — vas a querer ver outputs No (perdes visibilidad)
Buscar "donde se usa X" No Si — lista final basta
Investigar otra codebase No Si — no contamines main
Leer docs de la lib que vamos a usar Depende — si es corta, mantener; si es larga, delegar Delegar si son docs gordas
Correr test suite para ver que falla No (los logs llenan context) Si — devuelve "falla X por Y"
Refactor activo, varios archivos abiertos Si — necesitas los archivos vivos No

Conexion con progressive disclosure (M04)

Esto que vimos en subagents es exactamente el mismo patron que vimos en M04 con skills y progressive disclosure. Distinto envase, misma idea.

  SKILL.md completo se carga solo cuando aplica

  tool outputs intermedios se quedan en el subagent

Patron compartido

"cargalo solo si lo necesitas, y nunca cargues el laburo intermedio". Esa regla esta en progressive disclosure, en subagents, y la vas a ver tambien en como se carga episodic memory (top-k relevantes, no el historico completo). Es el principio central del context management.

Anti-patrones con subagents

Igual que cualquier herramienta, se puede usar mal. Los mas comunes:

  • Delegar algo que necesitas iterar: si el resultado del subagent lo vas a usar para un debug en el que vas a ir y venir, mejor en main context. Si no, despues tenes que mandar otro subagent y otro, perdiendo visibilidad.
  • Subagents anidados sin fin: si un subagent spawnea otro que spawnea otro, el result que vuelve puede ser tan abstracto que pierde lo util. Mantenelo un nivel.
  • Delegar para "ahorrar context" cuando el main esta vacio: si recien arrancas la sesion y tenes 20k tokens usados, no hace falta delegar nada. Mantener simple primero, optimizar context despues.
  • Subagent sin instruccion clara de que devolver: si no le decis que forma queres en el final report, te puede traer cualquier cosa. "Devolveme una lista de files + lineas" es mejor que "fijate si esta usado".

Cierre: subagent = delegacion con frontera de context

Pensalo asi: cuando contratas a alguien para una task en tu empresa, no le pedis que te cuente cada decision intermedia que tomo — solo te interesa el resultado. Si sos micromanager, contaminas tu propio tiempo con detalles que no aportan. Si sabes delegar bien, recibis conclusiones accionables.

Los subagents son eso. Delegas bien cuando:

  • Especificas claro el resultado que queres.
  • El proceso intermedio no te interesa.
  • Confias en que la sintesis final va a ser util.
