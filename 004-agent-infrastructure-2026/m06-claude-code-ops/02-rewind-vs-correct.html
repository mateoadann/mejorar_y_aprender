Rewind vs corregir: el habito senal

Si te digo que elijas un solo habito que te senale buen context management en Claude Code, lo tengo claro: rewind. Porque significa que pensas el contexto como algo que hay que mantener limpio, no como un registro historico que se acumula.

El escenario tipico

Imaginate esto, porque te pasa todas las semanas si usas Claude Code:

La situacion

Le pedis a Claude implementar algo. El modelo lee 5 archivos para entender el codebase. Prueba un approach (llamemosle A). Ejecuta tests. No funciona. Ahora vos tenes que responder.

La mayoria de los usuarios escribe algo asi: 

  "eso no funciono, el problema es que el modulo foo no expone
  ese metodo. Probá approach B, usando bar directamente."

Natural. Humano. Y sin embargo, no es lo que conviene.

Dos timelines, dos contexts

  INSTINCT MALO: continue con correccion
  ─────────────────────────────────────────
  msg 1: "implementa X"
  msg 2: (lee 5 files, prueba approach A, falla)
  msg 3: "eso no funciono, proba B usando bar"
  msg 4: (lee 2 files mas, prueba B, funciona)

  context final =
    5 files + intento fallido A + explicacion del fallo
    + tu correccion + intento exitoso B

  → el modelo ahora carga memoria de A-que-no-va, y eso
    puede sesgar generaciones siguientes ("¿sera parecido
    a ese caso del bar?")


  MOVE MEJOR: rewind + re-prompt
  ─────────────────────────────────────────
  msg 1: "implementa X"
  msg 2: (lee 5 files, prueba approach A, falla)
                                     │
                              rewind a msg 2
                              justo despues de los reads
                                     │
                                     ▼
  msg 3 (nuevo): "implementa X. Importante: el modulo foo no
                 expone ese metodo, andá directo por bar."
  msg 4: (prueba B, funciona)

  context final =
    5 files + intento exitoso B

  → sin basura del fallo, sin ruido de correcciones,
    sin sesgo hacia errores anteriores

El truco esta en donde hacer el rewind: no al principio (perderias los file reads, tendrias que re-leerlos). Te parar justo despues de las lecturas y antes del intento fallido. Re-prompteas con lo aprendido del fallo inyectado como instruccion nueva.

La tesis: rewind es el habito senal

Si tengo que elegir el habito que indique que alguien maneja bien el context, elijo rewind. Porque significa que tratas el contexto como algo que hay que mantener limpio, no como un log.

La gente que solo hace continue/correccion trata el contexto como un archivo historico — "todo lo que paso queda, y Claude se va a acordar". La gente que hace rewind trata el contexto como una herramienta de trabajo viva: si hay basura adentro, la saco. Si una correccion la puedo incorporar al prompt original, la incorporo, porque el resultado es un contexto mas limpio.

Summarize from here: el feature que facilita todo

Claude Code tiene un feature llamado "summarize from here". Como se usa: antes de tirar el fin de una sesion (o antes de un rewind grande), le decis a Claude que sintetice los learnings en un mensaje handoff. El modelo te devuelve un parrafo denso con lo aprendido. Vos lo pegas en el prompt nuevo.

Es como una nota del futuro al pasado de si mismo. Justo despues de entender bien algo, sintetiza ese entendimiento para que se lo puedas pasar a la version "antes de saberlo" sin que tenga que reconstruirlo.

Patron combinado

1. Claude termina una exploracion, aprendio X e Y.
2. Le decis "summarize from here, focus en lo que tiene que saber el proximo turno".
3. Guardas ese summary.
4. Rewind al punto anterior a la exploracion.
5. Re-promptear con "aca tenes lo que sabemos: [summary]. Ahora implementa Z."

Resultado: context limpio + conocimiento destilado.

/compact vs /clear: parecidos, distintos

Ambos resetean el context, pero de formas muy diferentes. Confundirlos te lleva a usar el equivocado en el momento equivocado.

/compact

Claude decide que se preserva.

El modelo sintetiza toda la sesion y reemplaza el historial con ese summary. Sigues trabajando "sobre" la sintesis.

Ventaja: no te hace escribir nada. Rapido.
Desventaja: lossy. Confias en Claude para decidir que importo. Si adivina mal, perdes info.

Podes steerearlo: /compact focus on the auth refactor, drop the test debugging.

/clear

Vos decides que se preserva.

Sesion nueva, en blanco. Vos escribis el brief: "estamos refactorizando auth, constraint es X, archivos relevantes son A y B, approach Y ya lo descartamos por Z".

Ventaja: el contexto resultante es exactamente lo que vos marcaste como relevante. Zero ruido.
Desventaja: mas trabajo manual. Requiere que hayas entendido vos que importa.

El bad compact: cuando /compact sale mal

El auto-compact dispara cuando el context se acerca al limite. El problema es que el modelo que decide que preservar esta en su peor momento cognitivo (viste eso en la pagina anterior). Ademas, no puede predecir hacia donde va tu laburo. 

  caso real: autocompact que salio mal
  ─────────────────────────────────────────

  Sesion larga de 900k tokens:
    - 500k: debuggeando un problema en foo.ts
    - 400k: investigando warnings varios (incluido uno
             en bar.ts que parecio menor)

  ALCANZA EL LIMITE → auto-compact dispara
  summary resultante:
    "Resolvimos el bug en foo.ts. Warnings menores revisados."
    ↑ dropeo los detalles del warning de bar.ts — parecio menor

  TU PROXIMO MENSAJE:
    "ahora arregla ese otro warning de bar.ts que habiamos visto"

  CLAUDE:
    "¿que warning de bar.ts?" (ya no lo tiene en context)

  → perdiste info que ibas a necesitar, porque el modelo no
    sabia que la ibas a necesitar
Se agrava con context rot

Lo que hace peor el bad compact: el modelo que decide que preservar esta operando con context rot maximo porque justo estas cerca del limite. Su capacidad de razonar "¿que es importante para lo que sigue?" esta degradada.

Como se evita: no dejes que el auto-compact decida solo. Con 1M de ventana tenes margen para hacer /compact proactivo mucho antes del limite, y ademas con instrucciones explicitas: 

  en la zona amarilla (~400-500k tokens):

  /compact focus on:
    - the auth refactor we're doing
    - keep mention of the warning in bar.ts (to revisit)
    - drop the stripe integration research (not using it)

Cuando rewind no es la correcta

Rewind es excelente para errores en el approach. No es la herramienta para todo. Casos donde continue o clear son mejores:

  • El fallo te ensena algo que afecta futuros turnos: si el error te revelo una caracteristica del codebase que vas a necesitar recordar varias veces, mejor que quede en el contexto (o que lo guardes en la memoria del proyecto).
  • Pasaron muchos turnos despues del fallo: si ya estas 5 turnos adelante y ahora queres corregir el primero, el rewind te hace perder laburo util.
  • La sesion entera cambio de direccion: si la task ya no es la misma, /clear le gana a rewind.

Cierre: tratá el contexto como el codigo

Un buen developer no deja codigo muerto en el repo — lo borra cuando deja de aplicar. Un buen usuario de Claude Code no deja intentos fallidos en el contexto — los remueve con rewind. El contexto es un recurso escaso, y cuanto mas limpio, mejor piensa el modelo.

Regla de oro

Cuando algo no funciona: primer instinto, rewind. Segundo instinto, corregir. Revertilo: la mayoria hace corregir primero y rewind nunca. Dale vuelta esa proporcion y el context se te va a mantener limpio solo.
