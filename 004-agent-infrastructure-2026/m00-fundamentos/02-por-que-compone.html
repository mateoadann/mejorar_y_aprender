Por que compone con el tiempo

Un agente con infraestructura no es una herramienta mejor. Es una herramienta que mejora sola. La razon son los feedback loops entre las capas. Pero ojo: si los loops alimentan errores, tambien se amplifican.

Compounding en sistemas de agentes

El interes compuesto es la idea de que el valor generado en un periodo se reinvierte, y el proximo periodo genera mas valor sobre una base mas grande. En un agente con infraestructura pasa lo mismo, pero con conocimiento, procedimientos y contexto en vez de plata.

Cada interaccion con el agente genera material que alimenta las capas inferiores. Una conversacion util se destila en memoria, la memoria acumulada permite crear una skill, la skill usada descubre un protocolo que falta, el protocolo nuevo devuelve datos que alimentan la memoria. Y vuelta a empezar.

La formula informal

Valor(t+1) = Valor(t) + (uso_semanal × tasa_de_destilacion × calidad_de_loops)

Si la tasa de destilacion es alta (el agente aprende de lo que hace) y los loops estan bien diseñados, la curva se despega.

Los 6 feedback loops

Entre las tres capas superiores (memoria, skills, protocolos) hay 6 loops posibles — 2 direcciones entre cada par. Todos suman si estan sanos. El M07 los desmenuza uno por uno; aca va el mapa. 

                  ┌─────────────────────────┐
                  │      MEMORIA            │
                  │  working · episodic     │
                  │  semantic · personal    │
                  └──┬────────────────────▲─┘
                     │                    │
              loop 1                loop 2
            destila a skill         skill guarda trace
                     │                    │
                     ▼                    │
                  ┌─────────────────────────┐
                  │      SKILLS             │
                  │  procedimientos         │
                  │  self-rewrite           │
                  └──┬────────────────────▲─┘
                     │                    │
              loop 3                loop 4
           pide nuevo protocolo    protocolo refina skill
                     │                    │
                     ▼                    │
                  ┌─────────────────────────┐
                  │    PROTOCOLOS           │
                  │  MCP, hooks, schemas    │
                  └──┬────────────────────▲─┘
                     │                    │
              loop 5                loop 6
           memoria rutea protocolo   protocolo devuelve data
                     │                    │
                     └──────► MEMORIA ◄────┘
Loop 1
Memoria → Skills

Patrones repetidos en memoria se destilan en una skill nueva. "Esto ya lo hice 5 veces, convirtamoslo en procedimiento".

Loop 2
Skills → Memoria

Cada ejecucion de skill guarda traces en memoria episodic. Alimenta el loop 1 con data fresca.

Loop 3
Skills → Protocolos

La skill descubre que necesita un tool que no existe. Propone el schema para un nuevo MCP endpoint.

Loop 4
Protocolos → Skills

Un protocolo nuevo desbloquea formas mejores de resolver tareas, y la skill se re-escribe para aprovecharlo (self-rewrite).

Loop 5
Memoria → Protocolos (routing)

La memoria personal dice "para este cliente uso el protocolo staging, no prod". Contexto ruteando permisos.

Loop 6
Protocolos → Memoria

Cada llamada a un protocolo devuelve datos del mundo: facts para semantic memory, eventos para episodic.

La curva de valor

Lo que se siente al construir infraestructura es un valle antes del despegue. Las primeras 1-3 semanas parece que invertis mucho sin retorno. Despues la curva se despega.

      s1      s4          s8              mes 3      mes 6
  • Semana 1 — frustrante. Armas la infra, escribis las primeras paginas del KBL, configurar MCP servers, te preguntas si vale la pena. Zero retorno visible.
  • Semana 4 — empieza a funcionar. El agente ya tiene suficiente contexto acumulado para responder bien la mayoria de los pedidos. Se nota diferencia vs. sesiones fresh.
  • Semana 8 — empieza a componer. Las skills generadas desde memoria empiezan a pagar. Aparecen mejoras que no programaste.
  • Mes 3+ — compounding. La infra es un activo. Cada tarea es mas barata que la anterior. Migrar de modelo es 5 minutos.

El riesgo: amplificacion de errores

Los loops amplifican lo que les metes adentro. Si el primer loop destila un mal lesson ("para cerrar deals, ofrece descuentos agresivos"), esa lesson se convierte en skill, la skill se ejecuta en mas casos, cada ejecucion deja trace en memoria que confirma la lesson...

La paradoja del sistema que aprende

Lo mismo que hace que tu agente componga positivamente, tambien puede hacer que componga negativamente. No es un bug, es la naturaleza de un sistema con feedback. Necesitas circuit breakers.

Los dos circuit breakers

La buena noticia: la arquitectura moderna incluye dos mecanismos para romper ciclos viciosos antes de que se afiancen.

Breaker 1

Dream cycle

Periodicamente (por ejemplo overnight o semanal), el agente re-procesa su memoria reciente contra sus lessons existentes. Detecta contradicciones, propone revisiones al humano. Inspirado en cognitive architectures como Soar y en el proceso de consolidacion del sueño humano.

Breaker 2

Self-rewrite con validation gate

Cuando una skill propone actualizarse a si misma, el cambio no se aplica directo. Pasa por un gate: test cases, review humana, rollback facil. La infra reconoce que el agente puede equivocarse y construye redundancia.

La regla de oro

Mientras mas autonomo es tu agente, mas importantes son los breakers. Un agente que solo sugiere puede darse el lujo de equivocarse. Un agente que ejecuta necesita mecanismos explicitos de deteccion y reversion.

Mini-resumen

  • Compone porque las 3 capas superiores tienen 6 loops de feedback que se refuerzan entre si.
  • Valle de valor: semanas 1-3 dolorosas, despegue entre semana 4 y mes 3.
  • Amplifica errores tanto como beneficios. Los breakers (dream cycle + self-rewrite con gate) son obligatorios.
