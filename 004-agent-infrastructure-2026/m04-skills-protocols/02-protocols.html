Protocols: tool schemas + permissions

La capa que casi nadie construye — y la que determina si tu agente queda como toy o como algo que podes dejar trabajando con real work. Dos piezas: tool schemas (contratos) y permissions.md (limites).

Por que protocols decide todo

Los protocols son los contratos que gobiernan como tu agente habla con sistemas externos. Sin ellos, el modelo improvisa formatos de mensaje, estructuras de argumentos, y chequeos de permisos. Ahi nacen la mayoria de los failures de produccion.

Lo que pasa sin protocols

El agente llama un tool con un argumento inventado. El tool responde con un error poco claro. El agente reintenta con otro argumento inventado. Cinco retries despues, hizo algo que no queria. O peor: el retry fue exitoso y ahora hay un commit en main que nadie autorizo.

Protocols = rigor. Tool schemas definen la forma, permissions definen los limites, hooks (proxima pagina) hacen el enforcement.

Tool schemas: de "guess how" a "fill in the blanks"

Un tool schema convierte la tarea del modelo de "adivina como se llama este tool" a "completa estos campos". El modelo deja de improvisar y empieza a ejecutar contra un contrato tipado. 

  // protocols/tools/github.json
  {
    "tool": "github",
    "operations": {
      "create_pr": {
        "required_args": {
          "title":  { "type": "string", "max_length": 100 },
          "body":   { "type": "string" },
          "base":   { "type": "string", "enum": ["main", "dev"] }
        },
        "preconditions":     ["all_tests_pass", "branch_not_empty"],
        "side_effects":      ["triggers_ci", "notifies_reviewers"],
        "requires_approval": false
      },
      "merge_pr": {
        "preconditions":     ["pr_approved", "ci_green"],
        "side_effects":      ["modifies_main_branch"],
        "requires_approval": true
      },
      "force_push": {
        "blocked_targets":   ["main", "prod", "staging"],
        "requires_approval": true
      }
    }
  }

Mira las piezas. No es solo "que argumentos acepto". Es:

required_args
La forma

Nombre, tipo, constraints (max_length, enum). El modelo no puede pasar basura; el schema lo rechaza antes de llegar al tool.

preconditions
Gates antes

"Tests tienen que pasar", "branch no puede estar vacia". Se chequean en el hook antes de ejecutar.

side_effects
Qué va a pasar

"Triggerea CI", "notifica reviewers". El agente (y el humano) saben que operaciones colaterales dispara esta accion.

requires_approval
Boolean clave

true = pausa y pedi confirmacion humana. El hook lo lee y corta la ejecucion. Sin esto, no hay supervision.

blocked_targets
Hard constraints

Array de cosas que nunca se pueden tocar. Enforced por el harness. No hay flag de "yes, I'm sure" que lo salte.

Permissions.md: la regla del new hire

Si no dejarias a un new hire hacerlo unsupervised en la primera semana, el agente tampoco deberia hacerlo solo.

Esa es la unica regla que importa para decidir que va en cada seccion. Pensalo como onboarding: los new hires leen cosas sin pedir permiso, piden feedback antes de hacer cambios importantes, nunca tocan producción sin supervisión.

permissions.md tiene 4 secciones. Nada mas.

Always allowed
read files, run tests, create branches, write to memory/skills, lint, format, query dbs read-only.
Libre
Requires approval
merge PRs, deploy to staging, delete files, install dependencies, modify CI config, send emails externos.
Gate
Never allowed
force push to main/prod/staging, access secrets directamente, modify permissions.md, disable hooks, drop tables en prod.
Block
Approved external domains
Whitelist de dominios para network calls: github.com, api.openai.com, internal company subdomains.
Whitelist
El archivo mas importante del protocol layer

permissions.md es la diferencia entre un agente que podes dejar corriendo solo y uno que tenes que babysittear. Invertir 30 minutos en escribirlo bien al principio te ahorra semanas de problemas despues. Y aca viene el detalle que la gente saltea: el propio permissions.md tiene que estar en "Never allowed modify". Si no, el agente puede reescribirlo y ya no tiene sentido.

Flujo de una tool call (protocols en accion)

Cuando el modelo decide llamar un tool, la llamada no va directo. Pasa por dos gates antes de ejecutar. Si falla cualquiera, la llamada se bloquea y el modelo recibe el error como feedback. 

    MODELO DECIDE LLAMAR: github.merge_pr(pr_id=42)
                     │
                     ▼
    ┌──────────────────────────────────────┐
    │  GATE 1 · SCHEMA VALIDATION        │
    │  ────────────────────────────────  │
    │  - ¿args match el schema?          │
    │  - ¿preconditions cumplen?         │
    │  - ¿target en blocked_targets?     │
    └──────────────┬───────────────────────┘
                   │ pass
                   ▼
    ┌──────────────────────────────────────┐
    │  GATE 2 · PERMISSIONS CHECK         │
    │  ────────────────────────────────  │
    │  - ¿operacion en always_allowed?   │
    │  - si no -> ¿requires_approval?    │
    │  - si no -> ¿never_allowed? BLOCK  │
    └──────────────┬───────────────────────┘
                   │ pass o approval OK
                   ▼
    ┌──────────────────────────────────────┐
    │  EXECUTION                          │
    │  github.merge_pr ejecuta            │
    └──────────────┬───────────────────────┘
                   │
                   ▼
    ┌──────────────────────────────────────┐
    │  POST_EXECUTION HOOK                │
    │  log a episodic memory              │
    │  (lo vemos en la siguiente pagina)  │
    └──────────────────────────────────────┘

    - - - - - -  ruta alternativa  - - - - - -

    ┌──────────────────────────────────────┐
    │  BLOCKED                           │
    │  modelo recibe error estructurado   │
    │  tool retorna {blocked: true,       │
    │                reason: "..."}       │
    └──────────────────────────────────────┘

Ejemplos: que va en cada seccion (regla new hire)

Accion ¿Lo dejarias a un new hire solo? Donde va
Leer un README Si, obvio Always allowed
Correr pytest Si, es leer-sólo el comportamiento Always allowed
Crear una branch Si, no afecta a nadie mas Always allowed
Mergear un PR a main No — quiero ver el PR antes Requires approval
Instalar un paquete nuevo No — supply chain risk Requires approval
Deploy a staging No — afecta ambientes compartidos Requires approval
git push --force main Nunca jamas Never allowed
Leer archivos .env directamente Nunca — secrets exposure Never allowed
Modificar permissions.md NUNCA — subvertir al guardia Never allowed

Cierre: protocols = el contrato con el mundo

Tool schemas convierten intencion en ejecucion tipada. Permissions definen el jardin en el que el agente puede moverse. Los dos juntos forman el protocol layer — la frontera entre el modelo y todo lo demas.

Lo que te vas a llevar

Si tu agente no tiene un permissions.md con las 4 secciones y tool schemas por operacion, no tenes un agente, tenes un LLM con acceso a tu laptop. La diferencia se ve el dia que hace algo destructivo.
