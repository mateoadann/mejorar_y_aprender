M04 · Skills & Protocols

Skillforge: el skill que crea skills

Un meta-skill. Su unico trabajo es ensenarle al agente como crear skills nuevas con el formato consistente. Las primeras las va a hacer mal. El self-rewrite hook hace que cada intento mejore al siguiente. Asi la mejora compone sola.

Por que un meta-skill

Sin un skillforge, cada vez que necesitas una capability nueva, la escribis a mano — o peor, dejas al agente improvisar un skill con formato distinto cada vez. Resultado: un zoo de skills sin anatomia comun, imposibles de mantener.

Con un skillforge:

  • Trigger estandar: "create skill", "new capability", "build skill for X".
  • Formato consistente: frontmatter + instrucciones + self-rewrite hook.
  • Integracion con el index y el manifest desde el primer intento.
  • Log en DECISIONS.md — queda rastro de por que se creo.

Flujo cuando el agente detecta que falta una capability

El humano dice algo como "nunca hiciste esto, ¿podes armarte un skill?". El skillforge se activa y corre este flujo:

Paso 1 · Index check
¿Ya existe algo similar?

Abre skills/_index.md. Si ya hay un skill parecido, propone extender el existente en lugar de duplicar.

Paso 2 · Lessons check
¿Hay patterns relacionados?

Lee memory/semantic/LESSONS.md. Si hay lessons previos de skills que fallaron en dominio parecido, los incorpora como constraints.

Paso 3 · Scaffold
Crear folder + SKILL.md

Genera skills/<nombre>/SKILL.md con el template: frontmatter (name, version, triggers, tools, constraints), instrucciones (destinations + fences), self-rewrite hook al final.

Paso 4 · Register
Update index + manifest

Agrega entry en _index.md (tabla humana) y en _manifest.jsonl (maquina-lectable para el harness).

Paso 5 · Scripts
Agregar scripts/ o references/

Si el skill necesita helpers, los crea bajo scripts/. Si necesita ejemplos de "good looks like", los guarda en references/.

Paso 6 · Log & commit
DECISIONS.md + git

Escribe una entry en memory/semantic/DECISIONS.md explicando por que se creo el skill. Commit atomico.

Honestidad: los primeros skills son ROUGH DRAFTS

Expectativa realista

Los primeros skills que genera skillforge NO van a ser perfectos. Van a tener instrucciones demasiado procedurales, triggers demasiado anchos, constraints faltantes. Esta bien. Lo importante es que salgan con formato consistente y con self-rewrite hook desde el dia 1. La mejora viene despues.

Esta es una idea contraintuitiva: estamos acostumbrados a pensar que un skill hay que disenarlo bien antes de usarlo. Pero cuando tenes self-rewrite incorporado, el skill mejora solo con cada uso. La primera version no necesita ser buena — solo necesita existir con el formato correcto.

El ciclo de mejora compuesta

La magia del skillforge es que el mismo skillforge tiene self-rewrite hook. Cada vez que genera un skill que termina fallando, aprende. Cada vez que genera uno que funciona bien, tambien aprende. Su template interno mejora con cada intento. 

        CICLO DE SKILLFORGE

        observacion (humano pide capability nueva)
                │
                ▼
        ┌───────────────┐
        │  DRAFT v1     │   ← skillforge genera skill rough
        └───────┬───────┘
                │ el agente usa el skill en tareas reales
                ▼
        ┌───────────────┐
        │  USAGE        │   ← logs van a episodic memory
        │               │   ← success/failure taggeados
        └───────┬───────┘
                │ after 5 uses OR on failure
                ▼
        ┌───────────────┐
        │  SELF-REWRITE │   ← skill lee sus propios logs
        │  del skill    │   ← update triggers, constraints
        └───────┬───────┘
                │ patterns recurrentes
                ▼
        ┌───────────────┐
        │  SELF-REWRITE │   ← skillforge mismo aprende:
        │  de skillforge│     que templates funcionan mejor
        └───────┬───────┘
                │
                ▼
        siguiente DRAFT es mejor que el anterior

Este es el argumento central: no estas optimizando un skill, estas optimizando la fabrica de skills. Es un loop que gana composicion con el tiempo.

Memory-manager: el segundo meta-skill critico

Skillforge tiene un hermano. Sin el, toda la memoria queda como "filing cabinet que nadie abre". Memory-manager es el skill que fuerza al agente a leer, reflexionar y promover memoria.

Sin memory-manager

Los archivos de memoria existen. El agente podria leerlos. Pero no lo hace consistentemente. Cada sesion empieza desde cero-ish. Los errores se repiten.

"Filing cabinet que nadie abrio nunca."

Con memory-manager

Triggers: "remember", "reflect", "distill", "what did I learn". Cada vez que se dispara, el agente pulls top salience, detecta recurrencia, promueve a semantic, archiva working resuelto.

"El filing cabinet se convirtio en feedback loop."

Sin estos dos meta-skills (skillforge + memory-manager), el sistema no compone. Los otros modulos (knowledge, memoria, protocolos) existen como archivos pero nunca se usan de forma sistematica.

El self-rewrite hook (template generico)

Cada skill que sale de skillforge incluye este hook al final. Es generico, pero aplica en cada uno porque cada skill tiene logs propios en episodic memory. 

  # --- SELF-REWRITE HOOK (al final de cada SKILL.md) ---

  TRIGGER:
    after every 5 uses  OR  on any failure

  PROCESO:
    1. read episodic entries tagged con este skill
    2. read current KNOWLEDGE.md (lo que sabe el skill hoy)

    3. si hay patterns nuevos:
         append a KNOWLEDGE.md
         update triggers en _manifest.jsonl
         update constraints si hubo safety-relevant pattern

    4. si se violo un constraint:
         ESCALATE a global semantic/LESSONS.md
         (asi el aprendizaje de este skill
          se vuelve system-wide — otros skills
          tambien lo van a leer)

  CONSERVATIVE:
    no rewrite on every run
    solo cuando la evidencia es clara (>=3 occurrences)
Por que "conservative"

El hook no reescribe el skill cada vez que corre. Espera evidencia clara (3+ ocurrencias de un pattern). Si reescribe agresivo, el skill se vuelve inestable y aprende ruido. Conservative = trust the signal, not the noise.

Cierre: dos skills que hacen que todo lo demas funcione

Si solo podes construir dos skills antes de tener tu sistema corriendo, que sean estos dos:

skillforge
Meta-skill para crear skills consistentes. Sin el, cada skill tiene formato propio y nada compone.
Meta
memory-manager
Fuerza al agente a leer, reflexionar, y promover memoria. Sin el, los archivos existen pero nadie los abre.
Meta
La idea clave

Los skills no se disenan perfectos. Se evolucionan. Lo que necesitas al dia 1 no es un skill bueno — es un skill con formato consistente y self-rewrite activado. El resto lo hace el tiempo.
