M04 · Skills & Protocols

Anatomia de un skill

Un skill bien escrito no es una receta paso-a-paso. Es un jardin con cerca: destinos claros, limites duros, y espacio para que el agente decida el camino. Aca desarmamos la estructura de un SKILL.md y la linea invisible entre estructura util y micromanagement.

Las 3 zonas de un SKILL.md

Todo skill serio tiene la misma anatomia: frontmatter arriba, instrucciones en el medio, self-rewrite hook al final. Cada zona tiene un proposito distinto y un publico distinto (el harness, el modelo, el propio skill). 

  ┌─────────────────────────────────────────────────────────┐
  │  1. YAML FRONTMATTER   (metadata para el harness)    │
  │  ─────────────────────────────────────────────────────  │
  │    name: api-endpoint-builder                           │
  │    version: 0.3.1                                       │
  │    triggers: [new endpoint, add route, build api]       │
  │    tools: [editor, run_tests, git]                      │
  │    preconditions: [repo_clean, tests_green]             │
  │    constraints: [no_secrets_in_code, rate_limit_mw]     │
  └────────────────────────▲────────────────────────────────┘
                           │ lo lee el harness al load
                           ▼
  ┌─────────────────────────────────────────────────────────┐
  │  2. CORE INSTRUCTIONS  (lo que el modelo lee)         │
  │  ─────────────────────────────────────────────────────  │
  │    - Contexto: WHY, no solo WHAT                        │
  │    - Destinations: "a good endpoint looks like..."      │
  │    - Fences: "nunca toques auth/middleware.py"          │
  │    - Ejemplos buenos + ejemplo fallido y porque fallo   │
  └────────────────────────▲────────────────────────────────┘
                           │ inspira, no manda
                           ▼
  ┌─────────────────────────────────────────────────────────┐
  │  3. SELF-REWRITE HOOK   (al final, siempre)            │
  │  ─────────────────────────────────────────────────────  │
  │    After every 5 uses OR on any failure:                │
  │      read episodic entries tagged con este skill        │
  │      si hay patterns nuevos -> append a KNOWLEDGE.md    │
  │      si se violo constraint -> promote a LESSONS.md     │
  └─────────────────────────────────────────────────────────┘

El principio: destinations, no driving directions

No escribas driving directions. Escribi destinations and fences. Mostrale al agente como se ve lo bueno, dejale claro donde no puede pisar, y confia en que va a encontrar el camino.
— Daniel, sobre como escribir skills que no se desactualicen.

Los skills mal escritos son procedurales: 12 pasos numerados, cada uno con un comando exacto. Al dia siguiente cambia una flag, un path, un nombre de tool, y todo el skill queda roto. El agente sigue los pasos aunque no tengan sentido.

Los skills bien escritos son declarativos sobre el destino: describen como se ve un buen resultado, muestran ejemplos reales, y marcan limites duros que no se cruzan. El agente elige el camino segun el contexto.

Driving directions vs destinations (ejemplo real)

Mira la diferencia. Los dos skills intentan lograr lo mismo — ayudar al agente a construir un endpoint de API nuevo — pero uno envejece mal y el otro escala.

Driving directions (mal)

skill-build-endpoint (v1)

1. Abri src/api/routes.py.
2. Buscar la funcion register_routes.
3. Agregar @router.post("/tu-endpoint").
4. Definir request_model en models/requests.py.
5. Agregar rate limit decorator.
6. Correr pytest tests/api/.
7. Grep "FAIL" en la salida.
8. Si falla, abrir logs en /tmp/api.log.
9. ... (12 pasos mas)

Se rompe el dia que cambia un path. Asume contexto que puede no aplicar. El agente no entiende por que.

Destinations + fences (bien)

skill-build-endpoint (v2)

Construimos APIs en FastAPI, convenciones REST, typed request bodies, rate limiting en middleware.

Aca tenes 3 endpoints bien hechos: /users/create, /orders/list, /webhooks/stripe.

Aca uno que fallo y por que: /admin/reset — olvido rate limit, expuso operacion destructiva sin auth.

Construi algo que se parezca a los buenos. No inventes nuevos patrones sin razon.

Envejece bien. El agente entiende el destino, no memoriza la ruta. Si cambia un path, sigue funcionando.

Los 3 componentes que balancean un skill

Un skill util combina tres tipos de instruccion. Demasiado de uno y el skill se rompe: todo procedures se desactualiza, todo heuristics se pierde, todo constraints se paraliza.

Procedures
Skeleton

Evita que el agente saltee una fase. No es un script paso-a-paso, es el esqueleto: "primero explorar, despues disenar, despues implementar, despues verificar".

Heuristics
Defaults

En cada bifurcacion, dale una preferencia. "Si dudas entre copy y rewrite, rewrite". Evita que el agente se freeze en decisiones menores.

Constraints
Fence

La cerca alrededor del jardin. "Nunca modifiques el schema sin migration". Hard limits. El agente puede moverse libre adentro, no puede salir.

La linea fina: structure vs micromanagement

Escribir un skill es un ejercicio de cuanto dejar sin decir. Demasiada instruccion y el agente se vuelve frágil. Muy poca y el agente improvisa en lugares criticos.

Structure (util)

"Verify tests pass before committing."

Define el destino (tests verdes antes del commit). El agente elige el comando, el parser, el timeout.

Micromanagement (fragil)

"Run npm test -- --verbose, grep for 'passed:', extract the number, then git add -A if > 0."

Se rompe con un npm upgrade. Asume formato de output. El agente no entiende por que verificar.

Audit por linea

Mira cada linea de tu skill y preguntate: ¿esta linea dice COMO hacer algo, o dice QUE GOOD LOOKS LIKE? Si dice COMO, probablemente no hace falta — borrala y confia en que el agente lo resuelve solo.

Donde van las preferencias del humano

Tentacion comun: meter preferencias personales adentro del skill. "Siempre usa comillas dobles", "no uses semicolons", "el cliente prefiere tablas con bordes finos". Todo eso NO va aca. 

  MAL (preferencia dentro del skill):
  skill: write-customer-email
    - "Siempre firmar con 'Saludos cordiales'"
    - "Nunca usar emojis"
    - "Mateo prefiere tono informal con clientes locales"

  BIEN (preferencia en personal memory):
  personal/memory.md
    - tone: informal con locales, formal con enterprise
    - sign-off: "Saludos cordiales"
    - no emojis

  skill: write-customer-email
    - A good email respects the customer's tone.
    - Pull sign-off and tone from personal memory.

Las preferencias son contexto, no instrucciones procedurales. Viven en personal memory (M03). Si las metes en el skill, se duplican, se desactualizan, y se hace imposible cambiar el tono sin tocar 12 skills distintos.

Cierre: skill bien escrito = skill que envejece bien

La metrica real de un skill no es "funciona hoy". Es "sigue funcionando en 6 meses con codebase que cambio". Los que envejecen bien tienen esto en comun:

  • Frontmatter corto pero con triggers y constraints explicitos.
  • Instrucciones que describen destinos, no rutas.
  • Ejemplos concretos (2-3 buenos, 1 fallido y por que fallo).
  • Self-rewrite hook al final, para que el skill mejore con el uso.
  • Cero preferencias personales — eso va en personal memory.
Regla mnemotecnica

Destinations and fences. Cada vez que escribas una linea de skill, pregúntate si estas describiendo un destino (bien) o dictando una ruta (mal). Si es ruta, tirala.
