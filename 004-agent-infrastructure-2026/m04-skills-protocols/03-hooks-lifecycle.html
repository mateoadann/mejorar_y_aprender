M04 · Skills & Protocols

Hooks: el enforcement layer

Los hooks corren antes y despues de cada accion del agente. Implementan lo que permissions y tool schemas definen. Son la pieza que hace que el sistema compose automaticamente — sin ellos, todo el andamiaje anterior queda en papel.

Tres hooks, tres momentos

El lifecycle de una accion pasa por tres puntos donde el sistema puede intervenir: antes de ejecutar, despues de ejecutar, y cuando falla. Cada uno con un hook dedicado.

pre_tool_call
Gate de entrada

Corre antes de invocar un tool. Carga el schema, chequea permissions, valida preconditions. Puede bloquear.

post_execution
Logger automatico

Corre despues de cada accion. Appendea a episodic memory con score de dolor, importance, y reflexion.

on_failure
Specialized

Se dispara en errores. Pain score mas alto, tracking por skill, triggering de rewrite cuando el threshold se supera.

pre_tool_call: el gate de permissions

Este hook es el que hace que permissions.md y los tool schemas sean enforcement real y no decoracion. Cuando el modelo intenta llamar un tool, la llamada pasa por aca primero. 

  # hooks/pre_tool_call.py  (pseudocodigo)

  def pre_tool_call(tool_name, args, target=None):
      schema = load_schema(tool_name)
      perms  = load_permissions()

      # 1. blocked_targets -> hard block, sin approval posible
      if target in schema.blocked_targets:
          return (False, "blocked_target: " + target)

      # 2. requires_approval -> pausar y pedir confirmacion
      if schema.requires_approval:
          return ("approval_needed", schema.side_effects)

      # 3. fallback: matchear contra permissions.md
      #    seccion "Never allowed", por keywords
      match_count = keyword_match(tool_name + args, perms.never_allowed)
      if match_count >= 2:
          return (False, "matches never_allowed patterns")

      # 4. pass -> let it through
      return (True, "ok")
Por que keywords y no exact match

El modelo puede llamar un tool con formas sutilmente distintas ("force push", "push --force", "hard push"). Un match exacto se saltea. Keyword matching con threshold (>=2 hits) captura las variantes. No es perfecto — por eso los blocked_targets y requires_approval son las lineas duras.

post_execution: el secreto del compound learning

Cada accion del agente — exitosa o fallida — queda registrada en memory/episodic/AGENT_LEARNINGS.jsonl automaticamente. Esto es lo que hace que el sistema aprenda sin que vos le ensenes nada. 

  # Cada entry en episodic/AGENT_LEARNINGS.jsonl

  {
    "timestamp":   "2026-04-17T14:22:01Z",
    "skill":       "api-endpoint-builder",
    "action":      "create_route",
    "result":      "success",
    "pain_score":  2,              // 2 si ok, 7 si fallo
    "importance":  "medium",
    "reflection":  "rate-limit middleware se aplico automatico
                   gracias al constraint del skill; antes habia
                   que recordarlo manual"
  }

El hook es fully automatic. No tenes que decirle al agente "por favor logeá lo que hiciste". Los scores (pain, importance) surge en el retrieval: entries con pain alto se levantan primero, failures recurrentes disparan rewrite.

El loop completo, sin intervencion

1. El hook logea cada accion a episodic.
2. Salience scoring levanta entries con pain > threshold.
3. El dream cycle nocturno promueve patterns recurrentes a semantic.
4. La proxima sesion arranca con el learning ya consolidado.

Vos no hacés nada manual. Solo escribir el hook una vez y confiar en la composicion.

on_failure: pain scoring especializado

Los failures merecen un hook aparte porque:

  • Su pain score default es alto (8, no 2) — tienen mas peso en retrieval.
  • Agregan error_type y context al log, para que la reflection sea accionable.
  • Cuentan recent_failures por skill, mirando el history jsonl.
  • Disparan rewrites cuando el threshold se supera.
  # hooks/on_failure.py  (pseudocodigo)

  FAILURE_THRESHOLD = 3   # en 14 dias

  def on_failure(skill, action, error, context):
      entry = {
          "skill":       skill,
          "action":      action,
          "error_type":  classify(error),
          "context":     context,
          "pain_score":  8,           # default alto
          "timestamp":   now()
      }
      log_to_episodic(entry)

      recent = count_failures(skill, days=14)
      if recent >= FAILURE_THRESHOLD:
          flag_skill_for_rewrite(skill)
          entry["pain_score"] = 10    # escala
          promote_pattern(skill, context)

Cuando un skill falla 3+ veces en 14 dias, el flag dispara el self-rewrite hook del propio skill (que vimos en la pagina de Skillforge). El loop se cierra: failures → flag → rewrite → skill mejor.

Lifecycle completo: de input a learning consolidado

Ahora podemos ver como se conecta todo. Una interaccion tipica pasa por estos puntos: 

  HUMANO ENVIA INPUT
         │
         ▼
  ┌──────────────────────┐
  │  build_context       │  ← pull personal + semantic + episodic
  └────────┬─────────────┘    relevantes (M03)
           ▼
  ┌──────────────────────┐
  │  MODEL REASONING     │  ← LLM decide: ¿que skill?
  │                      │    ¿que tool?
  └────────┬─────────────┘
           │ tool call
           ▼
  ┌──────────────────────┐
  │  pre_tool_call      │  ← chequea schema + permissions
  │  hook                │    puede BLOCKEAR o pedir approval
  └────────┬─────────────┘
           │ allowed
           ▼
  ┌──────────────────────┐
  │  EXECUTION           │  ← el tool corre
  │  (tool runs)         │    retorna resultado o error
  └────────┬─────────────┘
           ▼
  ┌──────────────────────┐
  │  post_execution     │  ← appendea entry a episodic
  │  hook                │    con pain_score, importance
  └────────┬─────────────┘
           │ (si hubo error)
           ▼
  ┌──────────────────────┐
  │  on_failure hook   │  ← pain_score alto
  │                      │    chequea threshold
  └────────┬─────────────┘    flag skill si supera
           │
           ▼
       RESPONSE
           │
           │ [3am — dream cycle, async]
           ▼
  ┌──────────────────────┐
  │  DREAM CYCLE         │  ← recurrentes a semantic
  │  (cronjob)           │    archiva resueltos
  └──────────────────────┘

Cada flecha es un punto donde el sistema aprende o enforza. El diseno es deliberado: el modelo no tiene que acordarse de logear, no tiene que acordarse de chequear permissions, no tiene que saber que existe el dream cycle. Todo eso es infraestructura.

Anti-patterns: lo que rompe el enforcement

Dos errores que ves repetido en agentes que "funcionan solos hasta que no funcionan":

Anti-pattern 1

Unsafe composition

Dos skills safe por separado se combinan en algo destructivo. Ejemplo clasico: merge-pr es safe, auto-push es safe. Los dos juntos = deploy accidental a produccion sin review.

Solucion: los constraints van en pre_tool_call, no dentro de los skills. Los skills no saben que existen otros skills. El hook si — y es el que ve la composicion.

Anti-pattern 2

Literal execution

Skill escrito procedural ("1. commit, 2. push"). El agente sigue los pasos aunque los tests hayan fallado. Resultado: push de codigo roto.

Solucion: vuelve a la pagina 00 — escribi destinations and fences, no driving directions. El agente tiene que entender el destino, no memorizar la secuencia.

Principio clave

Los skills son locales — no ven el sistema. Los hooks son globales — ven todo. Por eso cualquier constraint cross-skill (no hacer X despues de Y, no combinar A con B) tiene que vivir en los hooks, nunca dentro de un skill.

Cierre: la infraestructura que se esconde

Los hooks son el componente menos visible del stack y el que mas determina si tu sistema compone. Sin ellos:

  • permissions.md queda como comentario que nadie lee.
  • Los tool schemas son aspiracionales.
  • Episodic memory nunca se puebla.
  • El dream cycle no tiene data que consolidar.

Con ellos:

  • Permissions se enforza antes de cada tool call.
  • Cada accion queda en episodic sin esfuerzo manual.
  • Los failures se detectan, escalan, y disparan rewrites.
  • El aprendizaje es automatico y compuesto.
M04 en una linea

Un skill bien escrito (destinations + fences) + un permissions.md con la regla del new hire + tres hooks (pre, post, failure) = agente que aprende solo, respeta limites, y no hace nada destructivo sin tu aprobación. Eso es el M04 entero.
