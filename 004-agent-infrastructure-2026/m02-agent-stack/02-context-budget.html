Context budget: asignacion de tokens

El context window no es un chat-log infinito —es una caja de computacion con presupuesto. Cada pieza adentro compite por lugar. Tu trabajo es decidir, antes de cada llamada, que fragmentos entran y con que prioridad.

El context window es una caja, no un stream

La intuicion comun es pensar el contexto como una conversacion continua: el modelo "recuerda" lo que dijiste. No es asi. En cada llamada, el harness arma un paquete de tokens y se lo pasa al modelo. Todo lo que esta afuera de ese paquete no existe para el modelo en ese turno.

Esto cambia la pregunta. Ya no es "como hago para que el modelo recuerde". Es: que piezas necesita el modelo AHORA para responder bien.

CAJA DE COMPUTACION
                   +---------------------------------+
                   |   CONTEXT WINDOW (128k tokens)  |
  build_context -> |                                 | -> modelo
                   |   todo lo que entra ES visible  |
                   |   todo lo que queda afuera NO   |
                   +---------------------------------+
                                   ^
                                   |
                      resto de la memoria, skills,
                        historial, archivos, etc.
                       (existen, pero no cuentan)
Cada token dentro es presupuesto gastado. Fuera de la caja = no existe para este turno.

Targeted fragments = senal. Lo demas = ruido

Cada pieza que metes tiene que justificarse: "el modelo necesita esto para esta decision". A esos los llamamos targeted fragments —fragmentos dirigidos.

Lo que no se justifica es ruido. Y el ruido no es neutral: compite por atencion con la senal, ocupa budget, y (peor) hay evidencia fuerte de que la informacion en el medio de contextos largos se pierde. Mas no es mejor.

senal (targeted)

Lo que entra

  • Preferencias personales del usuario (siempre)
  • El workspace activo
  • Lecciones semanticas que matchean el task
  • Top 5 entradas episodicas por salience
  • Skills disparados por triggers
  • Permisos aplicables
ruido (lo que NO entra)

Lo que se queda afuera

  • Memoria vieja sin relevancia a este task
  • Skills que no disparo ningun trigger
  • Historial completo de la conversacion anterior
  • Archivos del proyecto que el modelo no va a tocar
  • Documentacion generica ya internalizada en el training
Corolario incomodo

Si cargas todo "por las dudas", el modelo performa peor que si cargas lo justo. La tentacion de tirarle contexto al modelo es exactamente lo que degrada la respuesta.

Orden de carga: el orden importa

Cuando build_context arma el paquete, sigue una prioridad fija. Si se termina el budget, lo ultimo de la lista simplemente no se carga —pero nunca se corta lo primero.

BARRA DE TOKENS: 5 BLOQUES POR PRIORIDAD
  [========================================================] 128000 tokens
  |                                                        |
  |  1. PERSONAL + WORKSPACE     ~2k   always              |
  |  +----+                                                |
  |  |####|                                                |
  |  +----+                                                |
  |                                                        |
  |  2. SEMANTIC LESSONS         ~8k   truncate si hay que |
  |  +--------+                                            |
  |  |########|                                            |
  |  +--------+                                            |
  |                                                        |
  |  3. EPISODIC TOP-5 x SALIENCE ~6k  ordenado por score  |
  |  +------+                                              |
  |  |######|                                              |
  |  +------+                                              |
  |                                                        |
  |  4. SKILLS MATCHED (progressive disclosure)            |
  |     +----------------+                                 |
  |     |################| ~20k  solo los triggereados     |
  |     +----------------+                                 |
  |                                                        |
  |  5. PERMISSIONS              ~1k   always (safety)     |
  |  +--+                                                  |
  |  |##|                                                  |
  |  +--+                                                  |
  |                                                        |
  |  reasoning reserve  ~40k                               |
  |  +--------------------------------+                    |
  |  |                                |                    |
  |  +--------------------------------+                    |
  |                                                        |
  |  user input + response                                 |
  |  +------------------+                                  |
  |  |                  |                                  |
  |  +------------------+                                  |
Los bloques 1 y 5 son inamovibles. 2, 3 y 4 se truncan por orden de prioridad.

Los dos extremos de la lista son no-negociables: preferencias del usuario (primero) y permissions (ultimo, por seguridad). En el medio entran las capas de memoria y los skills. Si un skill no disparo, simplemente no entra —no hay "por las dudas cargalo".

01 · always
Personal preferences + active workspace
Quien es el usuario, que esta editando, que convenciones tiene. Corto, estable, siempre presente.
02 · truncate si hace falta
Semantic lessons
Aprendizajes destilados de episodios pasados. Si el budget se complica, se trunca de atras para adelante.
03 · top-k fijo
Top 5 episodic por salience
Las 5 entradas mas recientes + dolorosas + importantes. Ni una mas, ni una menos. El scoring se explica en M02-04.
04 · progressive
Matched skills
Solo los skills cuyo trigger matcheo la request. Los demas quedan sin cargar, aunque existan en _index.md.
05 · always
Permissions (safety-critical)
Lista de que puede y que no puede hacer. Nunca se omite por budget.

Constantes tipicas

No hay valores magicos universales, pero hay numeros que se repiten en implementaciones serias. El ajuste fino depende del modelo y del dominio.

context_budget.py constantes tipicas 
# El tamano de la caja y la reserva para razonamiento
MAX_CONTEXT_TOKENS = 128000
RESERVED_FOR_REASONING = 40000

# Limites duros por categoria
PERSONAL_BUDGET = 2000
SEMANTIC_BUDGET = 8000
EPISODIC_BUDGET = 6000
SKILLS_BUDGET = 20000
PERMISSIONS_BUDGET = 1000

# Top-k fijos
EPISODIC_TOP_K = 5
SEMANTIC_MAX_ITEMS = 10

def available_for_input():
    used = (PERSONAL_BUDGET + SEMANTIC_BUDGET + EPISODIC_BUDGET
            + SKILLS_BUDGET + PERMISSIONS_BUDGET + RESERVED_FOR_REASONING)
    return MAX_CONTEXT_TOKENS - used

Estos limites son techos, no pisos. Si el usuario tiene pocas preferencias, no se infla para llegar a 2000. Si los skills disparados son chicos, el budget sobrante no se "rellena" con otras cosas —queda libre para el reasoning.

Para llevarte

Reglas mentales
  • El context window es caja, no stream. Lo que no entra, no existe.
  • Cada pieza que cargas compite por atencion. Mas contexto no es mejor.
  • Hay un orden fijo de carga con prioridades. Si falta budget, se sacrifica lo ultimo de la lista.
  • Si un skill no trigereo, no se carga. Punto.
