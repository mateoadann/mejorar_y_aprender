M02 · Agent Stack

Anatomia del stack: donde vive el cerebro

El stack de agentes de 2026 se parte en dos mitades que no se mezclan: un harness operativo que no piensa, y un estado acumulativo que es todo lo que piensa. Entender donde vive cada cosa es la decision arquitectonica mas importante del modulo.

La separacion clave: harness vs estado

Durante anos pensamos el agente como un bloque unico: "el agente hace X". En 2026 eso cambio. Lo que llamamos "agente" es en realidad dos piezas con responsabilidades muy distintas, que se conectan por un contrato simple: archivos en disco.

Del lado operativo esta el harness —el conductor. Lee archivos, llama al modelo, invoca tools, corre hooks, escribe logs. No toma decisiones sobre que hacer. Es ejecutor puro.

Del lado del conocimiento esta el estado: skills en markdown, memoria en markdown/jsonl, protocols en JSON schemas + markdown. Aca vive todo lo que el agente sabe y todo lo que aprendio. Lo que acumula valor con el uso.

Regla de oro

Si algo decide, pertenece al estado (skills/memory/protocols). Si algo ejecuta, pertenece al harness. Cuando empezas a mezclar, el stack se rompe.

Diagrama: las dos mitades del agente

AGENTE = HARNESS + ESTADO
 +---------------------------------------------------------+
 |                      AGENTE                              |
 |                                                          |
 |  +------------------+         +-----------------------+  |
 |  |   HARNESS        |  lee    |   ESTADO              |  |
 |  |   (operativo)    | ------> |   (acumulativo)       |  |
 |  |                  |         |                       |  |
 |  |  - load files    |         |  .agent/              |  |
 |  |  - call model    |         |   ├─ skills/*.md      |  |
 |  |  - run tools     |         |   ├─ memory/*.jsonl   |  |
 |  |  - run hooks     |         |   ├─ protocols/*.json |  |
 |  |  - write logs    |         |   └─ AGENTS.md        |  |
 |  |                  |         |                       |  |
 |  |  THIN, INMUTABLE |         |  CRECE CON EL USO     |  |
 |  +------------------+         +-----------------------+  |
 |                                                          |
 +---------------------------------------------------------+
El harness es comodity. El estado es tuyo. El valor vive del lado derecho.

La asimetria es intencional. Un lado es intercambiable (harness), el otro es propio y acumulativo (estado). Esta asimetria es lo que habilita las tres implicaciones de abajo.

Tres implicaciones que cambian como construis

01 · Portabilidad
Swap de harness sin perder nada

Si tu estado vive en .agent/ (archivos planos), podes cambiar el harness —de un script propio a Claude Code, o al reves— sin reentrenar ni migrar. El estado sigue igual.

02 · Independencia del modelo
Swap de modelo sin perder nada

Los skills son markdown. Los prompts los construye el harness. Si cambias de Claude a otro modelo, lo unico que ajustas es el cliente API del harness. Skills y memoria siguen siendo validos.

03 · Acumulacion
Solo el estado genera valor

Cada sesion deja marca en memory/. Cada patron nuevo aparece como skill. En git, con diffs revisables. El agente es mas util a los 6 meses que en el dia cero —porque el estado crece.

Consecuencia practica

Lo unico que vale la pena versionar con cuidado en git es .agent/. El harness podes reescribirlo o reemplazarlo. La inversion de tiempo esta en skills + memoria + protocols.

El folder tree canonico

En la practica, el estado se materializa en un directorio .agent/ en la raiz del proyecto. Subcarpetas separadas por tipo de conocimiento:

ESTRUCTURA DE .agent/
 proyecto/
 ├─ AGENTS.md                  ← tabla de contenidos del cerebro
 ├─ .agent/
 │   ├─ harness/               ← codigo del conductor (opcional)
 │   │   ├─ main.py            ← < 200 LOC
 │   │   ├─ build_context.py
 │   │   └─ log_execution.py
 │   │
 │   ├─ skills/                ← conocimiento procedural
 │   │   ├─ _index.md          ← siempre cargado (nombres + triggers)
 │   │   ├─ _manifest.jsonl    ← machine-readable
 │   │   ├─ git-proxy/
 │   │   │   └─ SKILL.md
 │   │   ├─ memory-manager/
 │   │   │   └─ SKILL.md
 │   │   └─ skillforge/
 │   │       └─ SKILL.md
 │   │
 │   ├─ memory/                ← memoria persistente (4 capas)
 │   │   ├─ working.md         ← contexto activo del task
 │   │   ├─ episodic.jsonl     ← eventos con pain/salience
 │   │   ├─ semantic.md        ← lecciones destiladas
 │   │   └─ personal.md        ← preferencias del usuario
 │   │
 │   ├─ protocols/             ← contratos con herramientas
 │   │   ├─ mcp-servers.json
 │   │   ├─ tool-schemas.json
 │   │   └─ permissions.md
 │   │
 │   └─ tools/                 ← scripts que el harness puede llamar
 │       └─ custom-tool.py
 │
 └─ src/                       ← tu codigo real
Cada carpeta responde a una pregunta distinta: que se, como actuo, que aprendi, con quien hablo.

El archivo AGENTS.md en la raiz funciona como indice maestro: apunta a las secciones relevantes, lista los skills activos y define el orden de carga. Es lo primero que lee cualquier harness al arrancar.

Ejemplo vivo: una ejecucion

Veamos que pasa cuando el usuario pide "fix el bug del login". El harness NO decide nada sobre el fix —solo orquesta. El estado informa cada decision.

01 · HARNESS
Recibe input
"fix el bug del login"
↓
02 · HARNESS lee ESTADO
Carga AGENTS.md + _index.md
ve que hay skill "debug-flow" y memoria previa sobre auth
↓
03 · HARNESS
Arma contexto + llama al modelo
injecta skill debug-flow + top-5 episodic
↓
04 · MODELO piensa
Decide tools, escribe codigo
toda la inteligencia ocurre aca
↓
05 · HARNESS escribe ESTADO
Append a episodic.jsonl
registra resultado, pain score, tools usados

Fijate: el harness toca el estado al principio (para armar contexto) y al final (para registrar lo que paso). En el medio, el modelo manda. El conductor nunca opina.

Para llevarte

Tres ideas fuerza
  • El harness no piensa —lee, ejecuta, escribe logs.
  • Toda la inteligencia vive en skills + memoria + protocols, markdown plano en git.
  • Esta separacion te da portabilidad (harness), independencia (modelo) y acumulacion de valor (estado) con el tiempo.
