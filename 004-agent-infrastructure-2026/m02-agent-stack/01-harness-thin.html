Harness thin: conductor < 200 LOC

El harness es el conductor del agente. Su trabajo no es pensar —es orquestar. Si tu conductor se pasa de 200 lineas, probablemente metiste inteligencia en el lugar equivocado.

La regla: 200 lineas, ni una mas

El harness hace pocas cosas, pero las hace bien: arma contexto, llama al modelo, ejecuta tools, corre hooks, escribe logs. Si te estas pasando de 200 lineas de codigo, fijate: casi siempre hay logica de decision ahi adentro que pertenece a un skill file.

"Si la request menciona bases de datos, cargo el skill de SQL..." — no. Esa regla vive en _manifest.jsonl como un trigger. El harness solo lee el manifest, mira los triggers, y carga lo que matchee. No interpreta nada.

Senales de que metiste inteligencia en el harness
  • Condicionales que inspeccionan el contenido del prompt
  • Heuristicas para decidir que skill cargar
  • Reglas de negocio o dominio en el codigo del conductor
  • Transformaciones "inteligentes" del contexto

Las tres funciones del conductor

Un harness minimo se resume en tres funciones puras. Todo lo demas es plomeria alrededor.

LIFECYCLE DEL HARNESS
  user input
      |
      v
  +-------------------+
  | build_context()   |   lee AGENTS.md, _index.md, memoria
  |                   |   respeta budget, arma prompt final
  +-------------------+
      |
      v
  +-------------------+
  | call_model()      |   manda prompt + tools al LLM
  |                   |   recibe respuesta con tool_calls
  +-------------------+
      |
      v
  +-------------------+
  | execute_tools()   |   corre cada tool_call con permisos
  |                   |   dispara pre_tool_call hook antes
  +-------------------+
      |
      v
  +-------------------+
  | log_execution()   |   append a episodic.jsonl
  |                   |   dispara post_execution hook
  +-------------------+
      |
      v
  user output
Input entra, output sale. En el medio, cuatro pasos mecanicos.

En pseudo-codigo, el loop entero cabe en 30 lineas:

harness/main.py pseudo-codigo conceptual 
# El conductor NO razona. Solo orquesta.
def run(user_input):
    ctx = build_context(user_input)        # lee estado
    while not done:
        response = call_model(ctx)         # el modelo piensa
        if response.tool_calls:
            for call in response.tool_calls:
                run_hook("pre_tool_call", call)
                result = execute_tool(call)
                ctx.append(result)
        else:
            done = True
    log_execution(ctx, response)           # escribe estado
    return response.content

Todo lo que decide QUE hacer esta afuera: en skills, en triggers, en la memoria que el modelo recibe como contexto. El harness solo pregunta "que leo, a quien llamo, donde escribo".

Alternativa: no escribas el harness, usa uno

Hay una opcion todavia mas barata: no escribir harness propio para nada. Claude Code, Cursor y harnesses similares ya traen el loop, el tool calling, el file watching y el logging. Apuntan al .agent/ folder y listo.

Usar harness existente

Claude Code / Cursor como harness

  • Loop ya resuelto, bien optimizado
  • Tool calling, hooks, permissions incluidos
  • File watching y diffs automaticos
  • Te enfocas en skills + memory, que es lo que agrega valor
  • Upgrades del harness vienen gratis
Escribir harness propio

Solo si lo necesitas

  • Mas flexibilidad, si tenes caso extremo
  • Te toca mantenerlo (bugs, upgrades, modelos nuevos)
  • Mas superficie para meter inteligencia en el lugar equivocado
  • Ganan dias/semanas de ingenieria que podias invertir en estado
Regla practica

Empeza con Claude Code o Cursor como harness. Solo escribe uno propio cuando choques contra un limite real (ej: un agente autonomo corriendo en servidor sin IDE). El 90% de los casos no lo necesita.

Los tres hooks que todo harness necesita

Los hooks son puntos de extension del harness. No son "codigo que decide" —son codigo que se dispara en momentos predecibles. Tres son no-negociables.

pre_tool_call
Corre ANTES de ejecutar cualquier tool. Sirve para enforcement: chequear permisos, validar paths, bloquear operaciones destructivas, confirmar con el usuario.
enforcement
post_execution
Corre al final de cada turno del agente. Registra todo a episodic.jsonl sin que el modelo se entere. Logging automatico, sin trabajo del modelo.
logging auto
on_failure
Se dispara cuando un tool tira error o la ejecucion excede algun limite. Escala al usuario, guarda el stack y (en modelos modernos) inyecta una entrada de reflection en la memoria semantica.
escalation

La gracia de los hooks es que siguen siendo codigo operativo —no toman decisiones de negocio, siguen reglas. pre_tool_call no decide "si este archivo es importante"; dice "si el path matchea la lista de bloqueados, paro". Las reglas vienen del estado (protocols/permissions.md), no del harness.

Para llevarte

Resumen operativo
  • Un harness saludable cabe en 200 LOC. Si te pasas, tenes logica de decision en el lugar equivocado.
  • Tres funciones puras: build_context, call_model, log_execution.
  • Usa Claude Code o Cursor como harness antes de escribir uno. Casi siempre alcanza.
  • Tres hooks no-negociables: pre_tool_call (enforcement), post_execution (logging), on_failure (escalation + reflection).
