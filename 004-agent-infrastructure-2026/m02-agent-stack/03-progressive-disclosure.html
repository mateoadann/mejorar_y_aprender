Progressive disclosure de skills

Si cargas todos los skills en cada run, desperdicias tokens y el modelo performa peor. Progressive disclosure resuelve esto con tres niveles: un indice corto siempre cargado, un manifest machine-readable y el SKILL.md completo solo cuando un trigger matchea.

El problema: no podes cargar todo

Supongamos que tenes 50 skills armados: git-proxy, memory-manager, skillforge, debug-flow, review-pr, deploy-check, etc. Cada uno es un archivo markdown de 2000-5000 tokens. Cargarlos a todos en cada request son 100k+ tokens quemados antes de empezar.

Peor: hay evidencia de que la informacion en el medio de contextos muy largos se pierde (lost in the middle). Cargar skills que no vas a usar no solo desperdicia budget —degrada las respuestas de los skills que si usas.

Anti-patron clasico

"Meto todos los skills en el system prompt, asi el modelo decide cual usar". Suena razonable, es desastroso. El modelo performa mejor con 3 skills relevantes que con 50 skills de los cuales 47 son ruido.

Tres niveles de disclosure

La idea es revelar informacion progresivamente. El modelo ve solo los nombres y descripciones cortas hasta que un trigger justifica cargar el skill completo.

ARBOL DE DISCLOSURE
  +---------------------------------------------------+
  |  NIVEL 1 · _index.md        SIEMPRE CARGADO       |
  |  tokens: ~500                                     |
  |  nombre + descripcion corta + trigger phrases     |
  |  "git-proxy: ops seguras de git con dry-run"      |
  |  "memory-manager: guardar/buscar en episodic"     |
  |  "skillforge: crear nuevos skills"                |
  +---------------------------------------------------+
                          |
                          | harness lee triggers
                          v
  +---------------------------------------------------+
  |  NIVEL 2 · _manifest.jsonl  MACHINE-READABLE      |
  |  tokens: 0  (no va al modelo, es para el harness) |
  |  name, triggers, tools, preconditions,            |
  |  constraints, category                            |
  +---------------------------------------------------+
                          |
                          | si un trigger matchea...
                          v
  +---------------------------------------------------+
  |  NIVEL 3 · SKILL.md         CARGA COMPLETA        |
  |  tokens: 2000-5000                                |
  |  instrucciones detalladas, ejemplos, edge cases,  |
  |  plantillas, checklists                           |
  +---------------------------------------------------+
El modelo ve Nivel 1 siempre. El harness lee Nivel 2 para decidir. Nivel 3 entra solo si hubo match.
Nivel 1
_index.md

Siempre cargado en el context. Corto. Nombres, descripciones de una linea, trigger phrases. Es el "menu" que el modelo ve.

Nivel 2
_manifest.jsonl

Para el harness, no para el modelo. Machine-readable: triggers, tools, preconditions, constraints, category. Un skill por linea.

Nivel 3
SKILL.md

El archivo completo. Se carga SOLO si un trigger del manifest matcheo la request. 2000-5000 tokens entran a la caja solo cuando hace falta.

Ejemplo: un _index.md real

El indice es el unico skill-asset que siempre entra al context. Por eso debe ser ultra-corto: el modelo solo necesita saber que skills existen, no como funcionan.

.agent/skills/_index.md siempre cargado · ~500 tokens 
# Skills disponibles

## git-proxy
Operaciones seguras de git con dry-run y validacion de paths.
Triggers: "commit", "push", "rebase", "cambios pendientes".

## memory-manager
Guardar y buscar entradas en episodic/semantic memory.
Triggers: "recorda que", "acordate de", "que aprendimos de".

## skillforge
Crear un skill nuevo siguiendo la convencion del repo.
Triggers: "crear skill", "nuevo skill", "documentar patron".

## debug-flow
Investigacion metodica de bugs: repro, hipotesis, fix, regression test.
Triggers: "bug", "error", "no funciona", "falla".

## review-pr
Review de pull requests: scope, coverage, seguridad, perfomance.
Triggers: "review PR", "revisar cambios", "aprobar MR".

Es un menu, no una enciclopedia. El modelo lee "ah, hay un debug-flow con triggers bug/error", y el harness se encarga de cargar SKILL.md completo cuando matchee.

Los campos del _manifest.jsonl

El manifest es machine-readable, no va al modelo. Una linea JSON por skill. Los 6 campos clave:

Campo Tipo Que hace
name string Identifier unico del skill, matchea la carpeta
triggers string[] Palabras o frases que disparan el load. Case-insensitive
tools string[] Que tools puede usar el skill (bash, write_file, etc)
preconditions string[] Que debe existir en el workspace para correr el skill
constraints string[] Limites duros (ej: "no tocar archivos en prod/")
category string Para agrupar y filtrar (git, memory, testing, etc)
.agent/skills/_manifest.jsonl una linea por skill 
{"name":"git-proxy","triggers":["commit","push","rebase"],"tools":["bash"],"preconditions":["git repo"],"constraints":["dry-run first"],"category":"git"}
{"name":"memory-manager","triggers":["recorda","acordate","que aprendimos"],"tools":["read_file","write_file"],"preconditions":[],"constraints":[],"category":"memory"}
{"name":"debug-flow","triggers":["bug","error","no funciona","falla"],"tools":["bash","read_file","write_file"],"preconditions":[],"constraints":["repro first"],"category":"testing"}

Escalamiento: de 50 a 500 skills

Progressive disclosure con triggers por keyword funciona bien hasta cierto tamano. Cuando el set de skills crece, conviene cambiar la estrategia de matching.

~50 skills

Keyword trigger matching

  • Simple y predecible
  • Cero dependencias adicionales
  • Debuggeable con la vista: leer _index.md
  • Falsos positivos bajos si elegis bien las keywords
  • Funciona sin GPU/embeddings
~500 skills

Semantic matching con embeddings

  • Embeddings de descripcion + trigger phrases
  • Vector search sobre la request
  • Tolera sinonimos y parafrasis
  • Requiere infra: index, refresh, eval
  • Top-K de matches + threshold de similarity
Cuando saltar

La transicion natural es cuando empezas a ver falsos negativos (skills relevantes que no disparan) o cuando el _index.md ya es demasiado largo para leer de un vistazo. Mientras tanto, keywords alcanzan.

Para llevarte

Cuatro ideas clave
  • Cargar todos los skills en cada run es desperdicio + ruido. Siempre.
  • Tres niveles: _index.md (always, ~500 tok), _manifest.jsonl (para el harness), SKILL.md (solo si matchea).
  • El manifest tiene 6 campos: name, triggers, tools, preconditions, constraints, category.
  • Keywords alcanzan hasta ~50 skills. Arriba de eso, pasar a embeddings.
