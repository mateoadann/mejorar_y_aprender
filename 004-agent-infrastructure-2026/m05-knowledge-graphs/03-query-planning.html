Query planning y join ordering

Un query SPARQL o Cypher es un set de restricciones simultaneas: "matchea pattern A Y pattern B Y pattern C". El motor las resuelve en un orden. Ese orden es la diferencia entre 50 milisegundos y 50 minutos. Vemos por que, como se estiman cardinalidades, y el algoritmo worst-case-optimal que cambio el estado del arte: Leapfrog Triejoin.

Un query es muchos patterns a la vez

Recordá el query del M05-00: "persona que conoce a otra persona que trabaja en una institucion partner de una empresa que produce comida". En SPARQL eso es algo asi (pseudo): 

# Pseudocodigo estilo SPARQL, conceptual
{
  ?a  KNOWS        ?b     # pattern 1
  ?b  WORKS_AT     ?inst  # pattern 2
  ?inst PARTNER_OF ?emp   # pattern 3
  ?emp PRODUCES    ?prod  # pattern 4
  ?prod category   "Food" # pattern 5  (el unico filled)
}

Son 5 patterns que deben ser verdaderos simultaneamente. El motor tiene que decidir en que orden los evalua y junta los resultados. A esa decision se la llama query plan.

El espacio de planes explota

Si tenés N patterns, hay N! (factorial) ordenes de evaluacion posibles:

4 patterns
4! = 24 planes

Aun buscable exhaustivamente.

7 patterns
7! = 5.040 planes

El planner empieza a usar heuristicas.

10 patterns
10! = 3.628.800 planes

Explorar todo es absurdo. Heuristicas obligatorias.

Regla de oro: patterns mas selectivos primero

Hay una heuristica que gana casi siempre: evaluar primero lo que menos matchea. Si pattern A devuelve 12 filas y pattern B devuelve 2 millones, arrancar por A hace que despues B se filtre contra un set chico. 

  Mal plan                        Buen plan
  ─────────                        ─────────
  1. Pattern B (2M matches)        1. Pattern A (12 matches)
  2. Join con A  ─┐                2. Join con B  ─┐
       ↓          │ 2M × 12 =           ↓          │ 12 × 2M =
       intermedio │ 24M rows            intermedio │ 24M rows
       gigante    │ PERO               chico      │ PERO
                  │ despues             filter B  │ filter con
                  │ filtrar 2M          con 12    │ 12 subjects
                  │ es caro             subjects  │ baja a ms

Los dos planes tienen el mismo producto cartesiano teorico (24M). La diferencia es que el buen plan nunca materializa ese intermedio gigante. Con los 12 subjects fijos, el lookup sobre B se hace 12 veces con prefix match en el index, no una vez sobre 2M.

Cardinality estimation: como se sabe cuanto matchea

Para aplicar la regla anterior, el planner necesita predecir antes de correrlo cuantos matches devuelve cada pattern. A esto se le llama cardinality estimation. Hay tres tecnicas canonicas:

Tecnica 1
Predicate statistics precomputadas

Antes de aceptar queries, el motor cuenta cuantos triples hay por cada (P, O) pair. "BORN_IN Ulm" = 47 triples. "KNOWS *" = 840.000 triples. Lookup O(1), muy preciso para patterns con P y/u O fijos.

Tecnica 2
Characteristic sets

Agrupa entities por el conjunto de predicates que comparten. Los cientificos tienen {BORN_IN, WORKED_AT, AUTHORED, DEGREE_FROM}. Las ciudades tienen {LOCATED_IN, POPULATION, COUNTRY_OF}. Permite estimar correlaciones — "si tiene AUTHORED, probablemente tenga WORKED_AT".

Tecnica 3
Sampling

Correr el query sobre el 1% del grafo, multiplicar por 100. Rapido, bastante preciso en distribuciones uniformes. Se rompe feo en grafos skewed (donde un par de super-nodos tienen millones de vecinos y el resto tiene 3).

El drama del skew

Los KG reales son muy skewed. En Wikidata, un puñado de entities tiene grado > 100.000 mientras la media es ~10. Las estimaciones que asumen uniformidad se equivocan por ordenes de magnitud. Characteristic sets ayudan pero no eliminan el problema.

Leapfrog Triejoin — el join worst-case-optimal

Hasta ~2012, la sabiduria convencional decia que los joins clasicos (hash, merge) eran lo mejor posible. Entonces llegaron los worst-case-optimal joins. El mas famoso: Leapfrog Triejoin, propuesto por Todd Veldhuizen en LogicBlox (2014).

Que significa "worst-case-optimal"

Existe una cota teorica inferior para cuantas operaciones mínimas necesita cualquier algoritmo para resolver un join. Los joins clasicos pueden quedar exponencialmente peores que esa cota en ciertos queries (caso tipico: triangle counting). Worst-case-optimal = nunca peor que esa cota. Veldhuizen probo que Leapfrog lo logra.

La intuicion del algoritmo: cada "iterator" (que representa un pattern) esta posicionado sobre un valor ordenado y puede hacer seek al siguiente valor ≥ target. El algoritmo baila entre los iterators haciendo que el mas atrasado salte al valor del mas adelantado. Si todos coinciden → match. Si no, el que se quedo atras salta mas. 

# Leapfrog Triejoin conceptual (estructural, no ejecutable)
def leapfrog_join(iterators, variable_order):
    # iterators = lista de iterators sobre los patterns
    # cada uno esta posicionado en el primer valor de su variable

    def solve(var_idx):
        if var_idx == len(variable_order):
            emit_tuple()             # todas las vars coinciden
            return

        iters = iterators_for_var(variable_order[var_idx])

        # posicionar todos al menor valor
        for it in iters:
            it.seek_start()

        while True:
            min_val = min(it.key() for it in iters)
            max_val = max(it.key() for it in iters)

            if min_val == max_val:
                # todos coinciden en este valor → recurse
                solve(var_idx + 1)
                # avanzar el primero y volver a intentar
                iters[0].next()
            else:
                # el mas atrasado salta al mas adelantado
                lagging = iter_with_key(iters, min_val)
                lagging.seek(max_val)

            if any_iter_exhausted(iters):
                break

El truco de fondo: cada seek sobre un trie ordenado (equivalente a un index SPO) cuesta O(log n), y el algoritmo skipea directamente sobre valores que no pueden participar en ningun join result. Nunca genera intermedios grandes que despues haya que filtrar.

Leapfrog visualizado

Supongamos tres iterators posicionados en tres columnas ordenadas. El algoritmo los mueve en sincronia hasta que coinciden: 

  Iter A:  [ 1, 3, 7, 9, 12, 15, ... ]   pos = 5
  Iter B:  [ 2, 5, 12, 14, 18, ... ]    pos = 3
  Iter C:  [ 4, 8, 12, 13, 21, ... ]    pos = 3

  Estado:  min=12, max=12   ◀─── match!
           → emit tuple, avanzar
           → proxima iteracion:

  Iter A:  [ ..., 12, 15, ... ]          pos = 6
  Iter B:  [ ..., 12, 14, 18, ... ]      pos = 4
  Iter C:  [ ..., 12, 13, 21, ... ]      pos = 4

  Estado:  min=13, max=15
           lagging = Iter C (13) → seek(15)
           → Iter C salta a 21 (primer valor ≥ 15)

  Iter A:  [ ..., 15, ... ]              pos = 6
  Iter B:  [ ..., 14, 18, ... ]          pos = 4
  Iter C:  [ ..., 21, ... ]              pos = 4

  Estado:  min=14, max=21
           lagging = Iter B (14) → seek(21)
           → Iter B salta a 21 (o al primero ≥ 21, digamos 25)
  ...

Fijate como saltan sobre valores que no tienen match posible. En un join clasico, materializas todos los pares y despues filtras. Leapfrog nunca los mira. Por eso es worst-case-optimal.

Resumen mental del query planner

1 · Parsear
Convertir el SPARQL/Cypher en un set de patterns (triples parcialmente instanciados).
2 · Estimar cardinalidades
Para cada pattern, cuantos matches se esperan. Predicate stats + characteristic sets + sampling.
3 · Elegir orden
Patterns mas selectivos primero. Muchos motores usan dynamic programming para grafos hasta ~12 patterns; heuristicas greedy mas alla.
4 · Elegir algoritmo de join
Hash join, merge join, o — para queries complejos con muchos patterns compartiendo variables — Leapfrog Triejoin.
5 · Ejecutar
Pipeline los resultados. Streaming si es posible, materializar si la query lo requiere.
Take-away

El planner es el cerebro del motor. Indexes y algoritmos de traversal son las manos. Un motor con indexes perfectos y mal planner pierde contra un motor con indexes decentes y buen planner. Por eso empresas como Neo4j, Virtuoso, GraphDB gastan años pulir el planner.
