Traversal: BFS, DFS, Dijkstra, A*, Bidirectional

El index te deposita en la puerta del grafo. Lo que sigue es caminarlo. El algoritmo que elegis es la diferencia entre 50ms y 50 minutos. Vemos los cinco clasicos, cuando usar cada uno, y el truco que reduce un query de 6 hops de 15.6 billion a 250K operaciones.

Que significa "traversal"

Los indexes sirven para empezar. Si pregunto "¿donde nacio Einstein?" el index resuelve eso de una. Pero si pregunto "dame todas las personas a 3 saltos de Einstein via KNOWS que trabajan en una empresa de tech", el index me da el primer salto y despues tengo que caminar vecino por vecino, decidiendo a cada paso para donde ir.

Esa decision — a que vecino visitar proximo, cual guardar para mas tarde, cuando cortar — es lo que define el algoritmo de traversal. Cambiar de algoritmo sin tocar los indexes puede bajar una query de minutos a milisegundos.

BFS — Breadth-First Search

La forma mas simple: explorar en capas. Primero todos los vecinos directos. Despues todos los vecinos-de-vecinos. Despues los de esos. Se mantiene una queue (FIFO). 

# Pseudocodigo conceptual (no ejecutable, estructural)
def bfs(graph, start, max_depth):
    visited = {start}
    parent = {start: None}
    queue = [(start, 0)]            # (node, depth)

    while queue:
        node, depth = queue.pop(0)  # FIFO
        if depth == max_depth:
            continue
        for neighbor in graph.neighbors(node):
            if neighbor not in visited:
                visited.add(neighbor)
                parent[neighbor] = node
                queue.append((neighbor, depth + 1))
    return parent
Usalo cuando

Querés el shortest path en un grafo sin pesos, o explorar todo lo que esta a N hops exactos. El orden por capas garantiza que el primer match encontrado es el mas cercano.

DFS — Depth-First Search

El opuesto de BFS. Clavás profundo primero: elegis un vecino, bajás, bajás, bajás, hasta que no podés mas o llegás al target. Despues backtrack: volvés un paso y probás otra rama. Se usa un stack (LIFO) en vez de queue. 

# DFS conceptual con stack explicito
def dfs(graph, start, target, max_depth):
    stack = [(start, [start])]   # (node, path-so-far)

    while stack:
        node, path = stack.pop()  # LIFO
        if node == target:
            return path
        if len(path) >= max_depth:
            continue
        for neighbor in graph.neighbors(node):
            if neighbor not in path:
                stack.append((neighbor, path + [neighbor]))
    return None
Cuidado con KG

En un grafo denso, sin max_depth, DFS puede desaparecer en cadenas larguisimas antes de probar otra rama. Siempre poné un limite. La ventaja de DFS es la memoria: solo guarda el camino actual (no el frontier entero como BFS).

Dijkstra — shortest path con pesos

Cuando las aristas tienen pesos (mas fuerte / mas debil, distancia, confianza), BFS no sirve: la ruta mas corta en numero de hops puede no ser la mas barata en peso total. Dijkstra resuelve eso con una min-priority queue ordenada por distancia acumulada. 

# Dijkstra con heap (priority queue)
def dijkstra(graph, start, target):
    dist = {start: 0}
    heap = [(0, start)]   # (cost, node)

    while heap:
        cost, node = pop_min(heap)   # extract cheapest
        if node == target:
            return cost
        if cost > dist[node]:
            continue                  # stale entry, skip
        for neighbor, weight in graph.edges(node):
            new_cost = cost + weight
            if new_cost < dist.get(neighbor, INF):
                dist[neighbor] = new_cost
                push(heap, (new_cost, neighbor))
    return INF
El invariante clave

En Dijkstra, el nodo mas barato conocido siempre se expande primero. Esto garantiza que cuando extraes el target, ese costo es optimo — ningun camino mas barato puede llegar despues, porque todos los nodos abiertos cuestan mas. Con binary heap cada pop/push es O(log V).

A* — Dijkstra guiado por heuristica

Dijkstra expande en todas las direcciones por igual. Si sabés aproximadamente donde esta el target, podés sesgar la busqueda hacia alla. A* hace eso sumando una heuristica al costo real: 

    Dijkstra:           priority = g(n)
    A*:                 priority = g(n) + h(n)

       g(n) = costo real desde start hasta n
       h(n) = estimacion del costo de n hasta target

Heuristicas comunes en KG:

  • Ontological distance: cuantos hops de clase-a-clase hay entre la clase de n y la del target. Si ambas son subclass de "Person", h = 0. Si una es "City" y la otra "Drug", h es alto.
  • Embedding distance: cada nodo tiene un vector (ver M05-04), la distancia euclidea entre vectores estima disimilitud semantica.
  • Domain-specific: en grafos geograficos, distancia euclidea de coordenadas. En bio, similitud de secuencias.
Admisibilidad

A* devuelve el optimal path solo si la heuristica es admissible: h(n) nunca sobrestima el costo real. Si tu heuristica es optimista (siempre menor o igual al real), A* es correcto. Si sobrestima, podés terminar con un path sub-optimo. No es trivial diseñarla.

Bidirectional Search — el truco magico

Todo lo anterior expande desde el start hacia afuera. Si sabés cual es el target, ¿por que no expandir desde los dos extremos al mismo tiempo? Cuando los dos frontiers se cruzan, conectás las rutas. 

  Unidireccional (depth d, branching k):
       ● ──▶ ○ ──▶ ○ ──▶ ○ ──▶ ○ ──▶ ●
       |←──────── explora  k^d ────────────→|

  Bidireccional (cada lado explora d/2):
       ● ──▶ ○ ──▶ ○  ◀──  ○ ◀── ○ ◀── ●
       |←── k^(d/2) ──→|  |←── k^(d/2) ──→|

  Costo total: 2 × k^(d/2)   en vez de k^d

La magia esta en el exponente. Veamos los numeros para k=50 y d=6 (el query de 6 hops del M05-00):

Unidireccional
50⁶ = 15.625M

15.625 billion de operaciones. Imposible.

Bidireccional
2 × 50³ = 250.000

250K operaciones. Manejable en milisegundos.

Reduccion
62.500x

Cuatro+ ordenes de magnitud. Solo por cambiar el algoritmo.

Regla practica

En cada iteracion, expandí el frontier mas chico. Si la expansion desde el start tiene 200 nodos pendientes y la del target tiene 50, expandí la del target. Mantener los dos frontiers balanceados es lo que garantiza el speedup.

Los 5 algoritmos, en comparacion

Algoritmo Estructura Usás cuando NO usás cuando
BFS Queue (FIFO) Shortest path sin pesos, explore N hops Hay pesos en las aristas
DFS Stack (LIFO) Baja memoria, hay que encontrar cualquier path Grafo denso sin max_depth, queres optimo
Dijkstra Min-heap Aristas con pesos, queres path mas barato Tenés buena heuristica (usá A*)
A* Min-heap + h(n) Tenés heuristica admisible al target No sabés el target o h() no es admisible
Bidirectional Dos frontiers Sabés start y target, grafo no super-irregular Target es un set amplio, no un nodo

Resumen mental

  • Elegir algoritmo es gratis, no requiere cambiar storage ni indexes.
  • Siempre ponele max_depth a tus traversal. KG denso + sin limite = fuga de memoria segura.
  • Si tenés target, usá bidirectional antes que cualquier otra cosa. El ahorro es obsceno.
  • Heuristica + A* = la cereza cuando ademas tenés embeddings o senal ontologica.
  • Todo esto asume que el index ya te deposito cerca. Sin indexes, cada paso del traversal es otro full scan.
