Intermediate · Proyecto 02

Self-improving Coding Agent

Un chatbot espera un prompt. Un agente espera un goal. Build an autonomous agent que escribe código, corre tests, y aprende de sus fallas.

Qué prueba este proyecto

Agentic loops + production debugging. El primer salto real: dejás de pensar en request/response y empezás a pensar en loops, reflection, y recovery de fallas.

El challenge

La gente le dice "agente" a un chatbot con tools y no lo es. Un agente de verdad recibe un goal ("arreglá este test que falla"), se arma un plan, ejecuta, verifica el outcome contra el goal, y si falla reflexiona sobre por qué. Y la próxima vez que ve un problema parecido, no repite el error.

Eso suena hermoso en un blog post. En la práctica el loop se rompe por cinco lados: el código generado rompe el sandbox, el agente se queda en un infinite loop corrigiendo el mismo error, o peor, borra archivos que no tenía que tocar. Acá es donde aprendés a poner límites.

El loop central

Todo el proyecto vive en este loop. Si el loop no termina, no tenés un agente; tenés un while(true) con un LLM adentro.

Fase 01
Plan

LLM recibe el goal, inspecciona contexto, propone pasos concretos. Output estructurado, no free-form.

Fase 02
Execute

Corre los pasos en sandbox. Cada paso tiene un timeout. Outputs capturados.

Fase 03
Test

Corre los tests contra el resultado. Pasa o no pasa. Binario. Nada de "parece que anda".

Fase 04
Reflect

Si falló, analiza por qué, extrae el pattern, y vuelve a plan con ese contexto. Si pasó, guarda el éxito.

Circuit breaker

El loop tiene max iterations (5 por default) y un circuit breaker: si las últimas 3 reflexiones son idénticas, cortás. Eso significa que el agente está loopeando en el mismo bug y no va a salir solo. Escalás al humano.

Decisiones arquitectónicas

Execution loop

Plan → execute → test → reflect. Max iterations configurables. Circuit breaker si el agente repite el mismo fix dos veces seguidas. Timeout global para cortar runs descontrolados.

Sandboxing

Ambiente aislado (Docker o Firecracker), con resource limits: CPU quota, memory cap, walltime, filesystem restringido a un working dir. Sin network a menos que el goal lo requiera.

Memory hierarchy

Short-term: últimas 5 iteraciones del loop. Long-term: patterns exitosos indexados. Failure memory: errores con solución que funcionó. Las tres se buscan antes de planear.

Reflection mechanism

Al fallar, extraer error pattern + root cause, no el error literal. Comparar con failure memory via vector similarity. Si hay match, usar esa solución como starting point.

Learning from mistakes

Cada falla se guarda con contexto completo: goal, plan, error, fix que funcionó. La próxima vez que el agente ve un problema parecido, retrieve antes de generar plan nuevo desde cero.

Code safety

Static analysis antes de ejecutar (AST parsing, linting). Operaciones de filesystem o network requieren approval explícito del usuario. El agente nunca corre rm -rf sin confirmar.

Por qué este es intermediate

Porque introduce agentic loops reales. Dejás de pensar "el LLM responde una vez" y empezás a pensar en términos de iteraciones, estado entre turnos y recovery. También te fuerza a enfrentar production debugging: cómo sabés qué pasó en la iteración 4 si tu agente corrió 7 y falló en la última.

Observability desde el día 1

Cada iteración del loop loggeada con trace completo. Goal, plan, output del execute, tests, reflection. Sin esto, no podés debuggear. Con esto, cada falla es una lección.
