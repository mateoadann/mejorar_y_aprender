Master · Proyecto 05 · Final boss

Enterprise Workflow Agent

El final boss. Un agente que runs a business. Monitorea Slack y Jira, planea execution, delega tasks, reporta outcomes con full audit logs. Production-grade en serio.

Qué prueba este proyecto

Production-grade orchestration. Combina orchestration, security, observability y multi-agent delegation en un sistema escalable. Esto es lo que demuestra que estás listo para el tier de $150K+.

Conexión con Módulo 1

Este proyecto es donde Managed Agents brilla. Podés armar todo esto con managed agents + MCP en lugar de construir cada primitiva desde cero. La diferencia son meses de infra vs un sistema operativo la semana que viene.

El challenge

Un agente personal falla silenciosamente y como mucho te arruina un día. Un agente enterprise falla y te manda un email a todo el board con data equivocada, borra un ticket en Jira que no tenía que tocar, o aprueba un PR roto en producción. Esto no es juguete.

Por eso este proyecto no es "hacé que funcione", es "hacé que falle bien". RBAC, audit trails, observability, human-in-the-loop para ops críticas, circuit breakers. La mayoría de estas decisiones no son sobre el LLM, son sobre el sistema que lo rodea.

Arquitectura completa

Event bus que ingesta eventos de múltiples sistemas, un orchestrator que planea, specialists que ejecutan, audit log immutable y observability stack sobre todo.

Layer 01
Event Bus

Slack events, Jira webhooks, emails, monitoring alerts. Pattern recognition → trigger workflow templates.

Layer 02
Orchestrator

Recibe el trigger, planea workflow multi-step, decide paralelismo, delega a specialists, maneja durable state.

Layer 03a
Communication Specialist

Slack, email, notificaciones. Con tone control por audiencia.

Layer 03b
Data Specialist

Queries a DBs, ETL, reportes. Read-first, write con approval.

Layer 03c
Analysis Specialist

Crunching de métricas, detección de anomalías, forecasting.

Layer 03d
Documentation Specialist

Actualiza Notion, Confluence, changelogs. Templates versionados.

Layer 04
Audit Log + Observability

Immutable log de cada decisión, trace de cada LLM call, métricas de success rate, latencia, costo. Todo queryable.

Decisiones arquitectónicas

Event-driven architecture

Listeners en Slack, Jira, email y monitoring systems. Pattern recognition mapea eventos a workflow templates conocidos. Si el patrón es nuevo, se escala al humano antes de improvisar.

Workflow orchestration

Steps con dependencies explícitas, paralelismo donde los steps son independientes, durable state para recovery después de crash o restart. Nada se pierde a mitad de ejecución.

Multi-agent delegation

Orchestrator es coordinator puro. Delega a specialists (communication, data, analysis, documentation). Cada specialist tiene su system prompt, sus tools y sus límites.

Self-healing

Monitor de success/failure por step. Decide retry vs escalate según contexto. Exponential backoff para errores transient, circuit breakers para errores que repiten.

Audit trail

Immutable log de todo: qué se decidió, por qué, quién autorizó, qué outcome tuvo. Append-only, con hash chains. Esto no es nice-to-have; es el requirement legal.

RBAC

Los permisos del agente están limitados a los del usuario que invocó el workflow. Ops sensitivas (pagos, borrar recursos, cambios en prod) requieren approval humano explícito. Sin excepciones.

Observability

Trace de cada LLM call: inputs, outputs, latency, tokens. Métricas agregadas: success rate por workflow, time-to-complete, cost per outcome. Dashboards vivos.

Human-in-the-loop

Antes de ejecutar workflows críticos, el agente propone el plan completo y highlight-ea los pasos high-risk. Esperar aprobación por Slack. Ejecutar solo si llega el OK.

Workflow learning

Post-workflow, evaluación automática del outcome. Patterns exitosos se guardan como templates reutilizables. Los templates evolucionan con uso.

Cost management

Track de tokens por workflow, budget limits por team, alertas cuando un workflow se sale de lo esperado. Prompt optimization continua en los templates caros.

Por qué este es master

Porque no es un proyecto de LLM. Es un proyecto de sistemas distribuidos con un LLM adentro. El orchestrator, el event bus, el audit log, el RBAC, la observability stack — todo esto lo tenés que entender bien antes de siquiera pensar en el prompt engineering. Es ingeniería de verdad.
