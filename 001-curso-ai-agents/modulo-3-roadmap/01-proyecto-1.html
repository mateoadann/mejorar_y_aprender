Beginner · Proyecto 01

SLM Mobile App

Build an offline-first mobile app usando small language models. Zero API costs. Completa privacidad. El modelo vive en el bolsillo del usuario.

Qué prueba este proyecto

Edge AI + resource optimization. No estás llamando una API, estás peleando con quantization, memory pressure y battery. Esto demuestra que entendés los constraints reales del hardware.

El challenge

La mayoría de apps con LLMs hoy dependen de una API. Eso significa costos por token, latencia de red y data del usuario saliendo del device. Este proyecto va al extremo opuesto: correr un SLM local, offline, encrypted, con sync opcional.

Suena simple hasta que lo intentás. Un modelo cuantizado de 4GB no entra en memory de un mid-range Android. Si lo cargás a demanda tardás 8 segundos en responder. Si lo dejás cargado te come la batería. Si no manejás el context window te explota en long conversations. Esa pelea es la lección.

Decisiones arquitectónicas

Model management

Lazy loading del modelo al primer uso, unload on memory pressure (onLowMemory), preload en idle cuando el user vuelve a la app. El modelo no está siempre en RAM, pero tampoco se carga desde cero cada vez.

Context window

Sliding window con semantic chunking. Cuando la conversación supera el límite, resumís los bloques antiguos con el mismo modelo y los reemplazás por el summary. El usuario no nota el cutoff.

Quantization strategy

Dinámica según las capacidades del device. Pre-2020 o low-end → 4-bit (Q4_K_M). Device más nuevo con más RAM → 8-bit (Q8_0) para mejor calidad. Detectás al instalar y bajás el modelo correcto.

Battery optimization

Batch inference cuando podés, throttle agresivo en low battery (< 20%), suspender tasks background si la temperatura sube. Un agente que deja el phone a 38C no lo usa nadie.

Offline-first sync

Todo encrypted local (SQLCipher). Sync opcional cuando hay conexión y el user da permiso explícito. El device es source of truth; el cloud es backup, no dependencia.

Arquitectura de model loading

El modelo transita por 3 estados y cada transición tiene costos. Entender esto es el 80% del problema.

Estado 01
Unloaded

Modelo en disco (encrypted). 0 MB de RAM. El user abre la app → transition a idle.

Estado 02
Idle

Modelo cargado en RAM, sin inference activa. ~2-4 GB según quantization. Listo para responder en <500ms.

Estado 03
Active

Inference corriendo. GPU/NPU ocupada, batería drenando. Vuelve a idle al terminar el token stream.

Triggers de transición

unloaded → idle: app foreground + battery > 30% + memory disponible. idle → unloaded: 5 min sin uso, o onTrimMemory del OS. idle → active: user envía prompt. Cada transición tiene que estar medida.

Stack sugerido

Inference engine

llama.cpp con bindings para mobile. Maduro, rápido, soporta Metal en iOS y OpenCL/Vulkan en Android.

Formato de modelo

GGUF. Quantizaciones Q4_K_M a Q8_0. Modelos base recomendados: Phi-3-mini, Gemma 2B, Llama 3.2 1B/3B.

Framework UI

React Native o Flutter. Ambos tienen bindings nativos a llama.cpp. Si querés full control del lifecycle, nativo es mejor.

Storage

SQLite con SQLCipher para conversaciones. Encrypted at rest con key derivada del keychain/keystore del device.

Por qué este es el nivel beginner

Porque no hay agentic loop, no hay multi-agent, no hay orchestration. Hay un modelo, un device y constraints físicos. Si resolvés esto, tenés las bases para todo lo que viene. Si no, vas a estar todo el roadmap tirando OOMs.
