Lección 02

Las 4 capas de memoria

Esta es la lección más importante del módulo. Vamos a recorrer las 4 capas de memoria en orden histórico. Cada una resuelve el problema anterior y, al hacerlo, destapa uno nuevo. Si entendés esta evolución, entendés por qué hoy hablamos de grafos.

Cómo leer esta lección

Cada capa tiene 3 partes: (1) el código o estructura mínima, (2) qué problema resuelve y (3) qué problema nuevo crea. No te saltees ninguna. La Capa 3 sin haber sufrido la Capa 2 no tiene sentido.

Capa 0

El agente mínimo (stateless)

Antes que nada, el punto de partida: un agente que no tiene memoria. Literalmente. Cada llamada es nueva, cada llamada olvida.

python · agente mínimo 
class Agent:
    def __init__(self, client, model="claude-sonnet-4"):
        self.client = client
        self.model = model

    def chat(self, user_message: str) -> str:
        response = self.client.messages.create(
            model=self.model,
            max_tokens=1024,
            messages=[{"role": "user", "content": user_message}],
        )
        return response.content[0].text

Problema

No recuerda ni el turno anterior. Le decís "me llamo Mateo" y al siguiente mensaje ya no sabe quién sos. Es la línea cero desde la que arrancamos.

Capa 1

Python list (in-memory)

La primera capa de memoria real: una lista de Python. El agente guarda cada mensaje en una estructura y la reenvía con cada llamada. Es la memoria más simple posible.

python · python list memory 
class Agent:
    def __init__(self, client, model="claude-sonnet-4"):
        self.client = client
        self.model = model
        self.messages = []  # memoria efímera

    def chat(self, user_message: str) -> str:
        self.messages.append(
            {"role": "user", "content": user_message}
        )
        response = self.client.messages.create(
            model=self.model,
            max_tokens=1024,
            messages=self.messages,
        )
        reply = response.content[0].text
        self.messages.append(
            {"role": "assistant", "content": reply}
        )
        return reply

Resuelve

Multi-turn: el agente ahora recuerda los turnos anteriores DENTRO de la misma ejecución.

Problemas nuevos

(1) la lista crece sin límite hasta reventar el context window, (2) al cerrar el proceso se pierde todo, (3) sin prioridad: es puro orden cronológico.

Capa 2

Markdown files (persistencia)

La solución obvia al problema anterior: guardá todo en archivos de texto. Un archivo por conversación, o un archivo grande con todo lo aprendido. Al arrancar el agente, leés los archivos y los metés en el prompt.

python · markdown memory agent 
class MarkdownMemoryAgent:
    def __init__(self, client, memory_path="memory.md"):
        self.client = client
        self.memory_path = memory_path
        self.memory = self._load_memory()

    def _load_memory(self) -> str:
        try:
            with open(self.memory_path, "r") as f:
                return f.read()
        except FileNotFoundError:
            return ""

    def _save_memory(self, new_fact: str) -> None:
        with open(self.memory_path, "a") as f:
            f.write(f"\n- {new_fact}")

    def chat(self, user_message: str) -> str:
        prompt = f"""Memoria previa:
{self.memory}

Usuario: {user_message}"""
        response = self.client.messages.create(
            model="claude-sonnet-4",
            max_tokens=1024,
            messages=[{"role": "user", "content": prompt}],
        )
        return response.content[0].text

Resuelve

Persistencia real. El agente sobrevive al cierre del proceso y puede arrancar mañana sabiendo lo de hoy.

Problemas nuevos

Con 500K+ tokens de markdown no podés cargar todo en el prompt. Necesitás búsqueda. Y keyword search es brittle: los sinónimos fallan de manera silenciosa.

Ejemplo concreto

Guardaste un fact que dice "cloud migration plan approved for Q2". El usuario pregunta "¿qué sabés del proyecto de migrating to AWS region?". Un grep de keywords NO encuentra nada. Son conceptos relacionados pero ni una sola palabra en común. La capa 2 te deja en banda acá.

Capa 3

Vector search (embeddings)

La solución al problema de los sinónimos: embeddings. Transformás cada pieza de memoria en un vector de alta dimensión que captura SIGNIFICADO, no palabras. Cuando buscás, comparás tu query (también como vector) con todos los vectores guardados y traés los más cercanos por cosine similarity.

Resuelve

Sinónimos y similitud semántica. "cloud migration" y "moving to AWS" caen cerca en el espacio vectorial. Esto fue un salto ENORME en 2022/2023.

Problema nuevo

Vector search NO hace conexiones relacionales entre facts. Busca un fact parecido al query, no una cadena de facts que juntos responden.

Ejemplo crítico (Alice + Postgres + Tuesday)

Tu base de memoria tiene estos 3 facts guardados en momentos distintos:

  • Fact A: "Alice is the lead engineer on Project Atlas."
  • Fact B: "Project Atlas uses PostgreSQL as its primary DB."
  • Fact C: "On Tuesday, the production PostgreSQL cluster went down for 4 hours."

El usuario pregunta: "Was Alice's project affected by Tuesday's outage?"

Vector search busca facts cercanos a la pregunta. El Fact A habla de Alice, el Fact C habla de Tuesday y outage. El Fact B, el que conecta a Atlas con Postgres, no comparte casi ningún token con la pregunta. Queda invisible. Y sin el Fact B puente, el razonamiento se rompe. Vector por sí solo NO puede responder.

Capa 4

Graph + Vector hybrid

Y acá llegamos al estado del arte. En lugar de guardar cada fact como un vector suelto, los guardás también como nodos y aristas en un knowledge graph. El grafo captura RELACIONES explícitas. El vector captura similitud semántica. Combinados, podés hacer queries multi-hop: saltar de un nodo a otro siguiendo las aristas.

Alice Person
Atlas Project
Postgres Service
Tuesday outage Event

Multi-hop: Alice → leads → Atlas → uses → Postgres → affected_by → Outage. Respuesta: sí.

Resuelve

Queries multi-hop con contexto relacional. El agente puede razonar sobre cómo se conectan las cosas, no solo sobre qué se parece a qué.

Cómo elegir

Si tu agente es un chatbot simple con 1 sesión y sin persistencia: capa 1 alcanza. Si tenés un dominio con muchos conceptos relacionados entre sí (soporte, legal, investigación, operaciones): capa 4 o nada. Las capas intermedias existen pero son trampas si sabés que el problema final es relacional.
