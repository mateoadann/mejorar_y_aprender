Lección 05

Práctica: agente de soporte con memoria

Llegó la hora de ensuciarse las manos. Vas a construir un agente de soporte técnico con memoria persistente usando Cognee. Paso a paso. Con código que funciona. Y con un checklist al final para verificar que entendiste cada decisión, no que copiaste y pegaste.

Qué vas a construir

Un CogneeMemoryAgent que ingiere tickets de soporte, construye un graph+vector de conocimiento, resuelve consultas multi-hop, consolida facts repetidos con memify() y resuelve pronombres dentro de una sesión.

Paso 1 — Setup

Instalá Cognee y configurá la API key de tu provider de LLM favorito (OpenAI, Anthropic, etc.). Cognee usa un modelo para la fase de extracción de entidades durante cognify().

bash · install 
pip install cognee
export LLM_API_KEY="sk-..."

Paso 2 — Ingresar 5 tickets

Vamos a usar 5 tickets de ejemplo que representan casos típicos: reportes de usuario, incidentes de infraestructura y escalaciones. Cada uno se ingresa con cognee.add().

python · ingest tickets 
import asyncio
import cognee

TICKETS = [
    "Ticket #1042: Alice, lead engineer on Project Atlas, "
    "reports the staging API returns 502 intermittently since Monday.",

    "Ticket #1043: Bob from Project Nova escalates a DB connection "
    "timeout. The Postgres cluster in us-east-1 is the suspect.",

    "Ticket #1044: On Tuesday at 15:30 UTC, the production "
    "PostgreSQL cluster in us-east-1 went down for 4 hours. "
    "Services affected: Atlas, Nova, Helix.",

    "Ticket #1045: Carol, PM of Project Helix, asks why dashboards "
    "showed stale data on Tuesday afternoon.",

    "Ticket #1046: Infra team confirms the Tuesday outage root cause "
    "was a runaway vacuum process on the primary replica."
]

async def ingest():
    for t in TICKETS:
        await cognee.add(t)

asyncio.run(ingest())

Paso 3 — Correr cognify()

Acá Cognee hace el trabajo pesado: clasifica, chunkea, extrae entidades (Alice, Atlas, Postgres, us-east-1, Tuesday outage...), arma las relaciones, dedup e indexa en vector + graph. Tardarse unos segundos es normal: está corriendo varios llamados al LLM.

python · build the memory 
async def build():
    await cognee.cognify()
    print("Memoria construida. Entidades y relaciones extraídas.")

asyncio.run(build())
Qué inspeccionar

Si podés, abrí el graph store (Kuzu por default) y mirá los nodos creados. Vas a ver Alice → leads → Atlas, Atlas → depends_on → Postgres, Postgres → affected_by → Tuesday_Outage. Esa cadena es la razón por la que vector solo falla.

Paso 4 — 3 queries con distintos SearchType

Ahora la parte divertida. Vamos a hacer la misma pregunta multi-hop con tres modes distintos y comparar resultados. Esto te enseña por qué el mode importa.

python · compare search modes 
from cognee.api.v1.search import SearchType

QUERY = "Was Alice's project affected by Tuesday's outage?"

async def compare_modes():
    # Mode 1: RAG clásico — solo vector similarity
    rag = await cognee.search(
        query_type=SearchType.RAG_COMPLETION,
        query_text=QUERY,
    )
    print("RAG_COMPLETION:", rag)

    # Mode 2: Graph + LLM synthesis — el default
    graph = await cognee.search(
        query_type=SearchType.GRAPH_COMPLETION,
        query_text=QUERY,
    )
    print("GRAPH_COMPLETION:", graph)

    # Mode 3: Con chain-of-thought
    cot = await cognee.search(
        query_type=SearchType.GRAPH_COMPLETION_COT,
        query_text=QUERY,
    )
    print("GRAPH_COMPLETION_COT:", cot)

asyncio.run(compare_modes())
Qué observar

RAG_COMPLETION probablemente te devuelva algo vago o incompleto (le falta el fact puente). GRAPH_COMPLETION te da la respuesta completa: sí, Atlas fue afectado porque depende de Postgres us-east-1, que cayó el martes. GRAPH_COMPLETION_COT te muestra el razonamiento paso a paso. Anotá las diferencias. Eso es la lección.

Paso 5 — memify() sobre tickets nuevos

Ahora agregá tickets adicionales que repiten un patrón: más outages en el mismo cluster, mismos días de la semana. Después corré memify() y volvé a consultar. Vas a ver cómo la memoria consolida patrones.

python · memify pass 
NEW_TICKETS = [
    "Ticket #1055: Tuesday again. Postgres us-east-1 had a "
    "slowdown between 15:00 and 16:00. Services Atlas and Nova "
    "reported stale reads.",

    "Ticket #1067: Tuesday outage repeated. 14:45 UTC. "
    "Primary replica swap took 12 minutes.",
]

async def consolidate():
    for t in NEW_TICKETS:
        await cognee.add(t)
    await cognee.cognify()
    await cognee.memify()

    result = await cognee.search(
        query_type=SearchType.TEMPORAL,
        query_text="Is there a pattern with Postgres outages?",
    )
    print(result)

asyncio.run(consolidate())
Consolidación en acción

Después de memify(), Cognee debería poder responder con algo parecido a "tres incidentes en martes sobre el mismo cluster sugieren un problema recurrente con la ventana de mantenimiento o con vacuum/replica swap". Es el puente episódica → semántica del que hablamos en la lección 03.

Paso 6 — Pronoun resolution con session_id

Último paso: envolvemos todo en un agente que mantiene contexto de sesión. Cuando un usuario pregunta "y qué hay de su proyecto?", el agente resuelve "su" usando la sesión activa y recuperando el último referente humano mencionado.

python · cognee memory agent 
class CogneeMemoryAgent:
    def __init__(self, client, model="claude-sonnet-4"):
        self.client = client
        self.model = model
        self.sessions = {}  # session_id -> turnos

    async def chat(self, session_id: str, user_msg: str) -> str:
        self.sessions.setdefault(session_id, [])

        # 1. Resolver pronombres usando la sesión
        resolved = await self._resolve_pronouns(
            user_msg, self.sessions[session_id]
        )

        # 2. Consultar memoria persistente (Cognee)
        mem = await cognee.search(
            query_type=SearchType.GRAPH_COMPLETION,
            query_text=resolved,
        )

        # 3. LLM final que sintetiza con contexto
        reply = self.client.messages.create(
            model=self.model,
            max_tokens=1024,
            messages=[
                {
                    "role": "user",
                    "content": (
                        f"Contexto de memoria:\n{mem}\n\n"
                        f"Pregunta: {resolved}"
                    ),
                }
            ],
        ).content[0].text

        # 4. Guardar turno en la sesión
        self.sessions[session_id].append(
            {"user": user_msg, "assistant": reply}
        )
        return reply

    async def _resolve_pronouns(self, msg, history):
        if not history:
            return msg
        last = history[-1]["user"]
        # Heurística mínima: si hay pronombre, incluir contexto
        if any(p in msg.lower() for p in ["su ", "él ", "ella ", "ese "]):
            return f"{last}\n{msg}"
        return msg
