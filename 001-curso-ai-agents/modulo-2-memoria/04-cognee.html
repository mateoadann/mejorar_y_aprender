Lección 04

Cognee: la solución integrada

Cognee es un engine de memoria para agentes que implementa la capa 4 (graph + vector hybrid) de forma open source y pronto-para-producción. Su API es chiquita, sus primitivas son claras, y sus search modes son 14. No 3, no 5. Catorce.

La API real (4 verbos)

Toda la interacción con Cognee se reduce a 4 funciones. Cada una tiene un rol específico y se llaman en orden.

python · cognee core api 
import cognee

await cognee.add("Tu documento acá")   # Ingest
await cognee.cognify()                 # Build graph + embeddings
await cognee.memify()                  # Self-improve memory
await cognee.search("Tu query")        # Retrieve con reasoning

3 stores, 1 engine

Por dentro, Cognee coordina tres stores distintos trabajando juntos. El default arranca liviano con SQLite + LanceDB + Kuzu, pero cada uno se puede swap a una versión de producción sin tocar tu código de aplicación.

Relational SQLite → Postgres
Vector LanceDB → Qdrant/Pinecone
Graph Kuzu → Neo4j/FalkorDB

Relational store

Provenance. De dónde vino el fact, cuándo se ingestó, quién lo accedió, qué versión del pipeline lo procesó. Todo lo "aburrido pero necesario" para auditoría y debugging.

Vector store

Semántica. Qué significa cada chunk y a qué se parece. Es el motor para los modes CHUNKS, CHUNKS_LEXICAL, SUMMARIES y RAG_COMPLETION.

Graph store

Relaciones. Cómo se conectan las entidades entre sí. Quién reporta a quién, qué proyecto usa qué servicio, qué evento afectó a qué sistema. Es el motor de los modes GRAPH_*.

Los 14 search modes REALES

Esta es la tabla completa, sin resumir. Cada mode resuelve un tipo de consulta distinto. Elegir bien cuál usar es la diferencia entre un retrieval mediocre y uno que realmente responde la pregunta.

# Mode Qué hace
01 SUMMARIES Similarity sobre TextSummary
02 CHUNKS Similarity sobre DocumentChunk
03 CHUNKS_LEXICAL BM25 sobre chunks
04 RAG_COMPLETION RAG clásico
05 TRIPLET_COMPLETION Retrieval por triplets
06 GRAPH_COMPLETION Default, LLM + graph traversal
07 GRAPH_SUMMARY_COMPLETION Graph sobre resúmenes
08 GRAPH_COMPLETION_COT Chain-of-thought iterativo
09 GRAPH_COMPLETION_CONTEXT_EXTENSION Graph + follow-ups
10 CYPHER Query Cypher crudo
11 NATURAL_LANGUAGE NL → Cypher automático
12 FEELING_LUCKY Mejor resultado único
13 TEMPORAL Con awareness temporal
14 CODING_RULES Retrieval de reglas de código
Cuál usar por default

GRAPH_COMPLETION es el default por una razón: combina el traversal del grafo con un LLM que sintetiza la respuesta final. Si no sabés qué elegir, arrancá por ahí. Si tus consultas son temporales (eventos, fechas), pasate a TEMPORAL. Si necesitás flexibilidad total, NATURAL_LANGUAGE te traduce la pregunta a Cypher.

El pipeline de cognify()

Cuando llamás a cognify(), esto es lo que pasa por dentro. Entender el pipeline es importante porque cada etapa agrega valor y cada etapa tiene su costo.

Paso 01

Classification

Cognee determina el tipo de documento y elige la estrategia de chunking apropiada.

Paso 02

Chunking

Parte el documento en DocumentChunks de tamaño controlado, respetando semántica cuando puede.

Paso 03

Entity + relationship extraction

Un LLM extrae entidades (Person, Project, Event, etc.) y las relaciones entre ellas. Esto arma los nodos y aristas del grafo.

Paso 04

Deduplication

Si "Alice" aparece 20 veces en distintos documentos, se consolida en un solo nodo. Evita explosión del grafo.

Paso 05

Summaries

Se generan TextSummarys para chunks y secciones. Estos alimentan el mode SUMMARIES.

Paso 06

Dual indexing

Todo se indexa simultáneamente en el vector store y en el graph store. Con metadatos de provenance en el relational.

memify(): optimización estilo RL

memify() es el paso que diferencia a Cognee de los vector stores tradicionales. Toma la memoria existente y la auto-mejora usando señales de uso (parecido a RL): refuerza los caminos útiles, poda los nodos stale, ajusta los pesos de las aristas.

Strengthen paths

Los caminos del grafo que el agente usa con frecuencia reciben mayor peso. Se vuelven retrieval de primera clase.

Prune stale nodes

Los nodos que nunca se consultan y no tienen relaciones vivas se marcan para borrado. El grafo no crece eternamente.

Tune edge weights

Las relaciones ajustan su fuerza según evidencia acumulada. Es la consolidación de Tulving, automatizada.

Comparativa con alternativas

Cognee no está solo en el mercado. Hay cuatro players que se suelen comparar. Acá el mapa:

Tool Modelo Fortaleza Cuándo usar
Cognee Graph + vector hybrid 14 search modes, memify incremental Dominios relacionales, producción seria
Mem0 Layered (user/session/agent) Simple, rápido de integrar Personalización de chatbots, bajo costo
LangMem Hot-path dentro de LangGraph Acoplado al ecosistema LangChain Ya estás casado con LangGraph
Letta Memory-as-OS, stateful Framework propio, agentes persistentes Querés que tu agente sea un "proceso"
Cómo decidir

Si tu problema es relacional (la mayoría de los problemas serios lo son), Cognee te va a dar el retrieval más preciso. Si tu problema es personalización liviana de un chatbot, Mem0 te alcanza. Si ya tenés LangGraph en producción, probá LangMem antes de migrar. Letta es más un experimento arquitectónico que una opción conservadora.
