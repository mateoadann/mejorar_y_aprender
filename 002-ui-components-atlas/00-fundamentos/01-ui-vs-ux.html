UI vs UX vs Design Patterns

Tres terminos que se confunden constantemente y que, cuando los mezclas, le pedis cosas al agente que no tienen sentido. Acá te muestro la diferencia con ejemplos concretos y por que te conviene hablar en terminos de patterns antes que de componentes.

Las definiciones, sin vueltas

Termino Que es Que responde Ejemplo concreto
UI (User Interface) La capa visual. Pixeles, colores, tipografia, espaciados, componentes. "Como se ve?" El boton es de color negro, bordes redondeados, 16px de padding.
UX (User Experience) La experiencia completa al usar el producto. Flujo, friccion, emocion, percepcion. "Como se siente usarlo?" Completar el checkout en 2 pasos sin preguntar datos innecesarios.
Design Pattern Una solucion reusable a un problema de interaccion recurrente. "Como resolvemos este problema?" "Selección única" → Radio Group, Select, Combobox o Segmented Control.

El concepto clave

Un mismo design pattern puede resolverse con distintos componentes segun el contexto. El pattern es el problema, el componente es la herramienta.

Pattern: "Seleccion unica entre N opciones"

Contexto Componente recomendado Por que
2 a 4 opciones Radio Group Todas visibles al mismo tiempo, comparacion instantanea.
5 a 15 opciones Select Ahorra espacio vertical, el usuario busca rapido.
15+ con busqueda Combobox Filtrado por texto reduce la friccion de scroll.
Filtros persistentes visibles Segmented Control / Tabs El usuario necesita ver en que estado esta sin abrir nada.

Patterns comunes que vale la pena conocer

Estos nombres aparecen todo el tiempo en literatura de diseño y en documentacion de frameworks. Mencionarlos explicitamente cuando le hablas al agente te ahorra parrafos de explicacion.

Master-Detail

Lista de items a la izquierda, detalle del seleccionado a la derecha. Ejemplo: Gmail, Finder, Slack.

Wizard / Stepper

Un proceso largo dividido en pasos numerados. Ejemplo: checkout de e-commerce, onboarding de una app.

Optimistic UI

Actualizas la UI antes de confirmar con el servidor, y revertis si falla. Ejemplo: el "me gusta" de Twitter, que se pinta rojo instantaneo.

Empty State

Lo que mostras cuando no hay datos. Ejemplo: "Todavia no creaste ningun proyecto" con un boton "Crear el primero".

Skeleton Loading

Placeholder con la forma del contenido mientras carga, en lugar de un spinner generico. Ejemplo: el feed de Facebook o LinkedIn mientras carga.

Infinite Scroll vs Pagination

Dos formas de mostrar listas largas. Infinite es mejor para consumo pasivo (feeds); pagination es mejor cuando hay que encontrar algo especifico.

Tip para vibe-coding

Cuando le hablas a un coding agent, menciona el PATTERN, no solo el componente. "Quiero un master-detail para mis tickets" le da muchisimo mas contexto que "quiero una lista con un panel". El agente reconoce el pattern, carga todas las convenciones asociadas (foco del teclado, responsive, URL sync) y te genera algo que funciona bien de entrada.
