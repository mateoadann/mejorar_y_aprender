Core Platform — el corazón de Vercel

Estos cuatro servicios son el motor: sin ellos no hay app corriendo. CI/CD automatiza tu deploy, la CDN hace que cargue rápido en cualquier parte del mundo, Fluid Compute ejecuta tu código con un modelo de costos pensado para la era AI, y Observability te cuenta qué está pasando por dentro.

CI/CD — Continuous Integration / Deployment

CI/CD es el servicio que convierte un git push en una app corriendo en producción, sin que toques nada. Vercel presume que ayuda a equipos a "ship 6× faster", y no es marketing vacío: cuando lo comparás con armar un pipeline tradicional (GitHub Actions + scripts + servidor + handoff manual), la diferencia de velocidad es real.

1
git push

Mandás un cambio a tu branch en GitHub, GitLab o Bitbucket.

→
2
Build

Vercel detecta el framework, instala dependencias y compila el proyecto.

→
3
Test / Checks

Corre linters, tests y verifica que el build esté limpio.

→
4
Deploy

Sube los artefactos a la red global, invalida cache y la URL queda viva.

La magia del CI/CD en Vercel

Cada push a main va a producción. Cada push a otra branch genera una preview URL única (lo vemos en profundidad en la página 04). No tenés que configurar nada: si conectás el repo, el pipeline ya está armado.

Content Delivery Network (CDN)

Imaginate que tu servidor está en Virginia (EE.UU.) y un usuario te consulta desde Buenos Aires. Cada request tiene que cruzar 8.500 km ida y vuelta, pasar por varios saltos, y la latencia se dispara. Un CDN resuelve eso: pone copias de tus assets en nodos distribuidos por todo el planeta, y cuando el usuario pide algo, lo sirve desde el nodo más cercano.

SIN CDN
Usuario BA → Servidor VA

Cada request viaja 8.500 km. Latencia alta, la imagen tarda, el sitio se siente pesado.

→
CON CDN
Usuario BA → Nodo Sao Paulo

Copia cacheada a 2.400 km. Respuesta en milisegundos, el sitio se siente instantáneo.

Fluid Compute — "servers in serverless form"

Acá está la clave diferenciadora de Vercel hoy. Fluid Compute es el modelo de ejecución propio de Vercel para funciones serverless, y resuelve un problema enorme: el serverless tradicional cobra por tiempo total de ejecución, incluyendo el rato que tu función se queda esperando a una API externa, una DB o un modelo de IA.

Fluid Compute, en cambio, cobra solo CPU activo. Si tu función llama a OpenAI y espera 8 segundos la respuesta, esos 8 segundos no se facturan — la CPU está inactiva. Esto no es un detalle, es una revolución para apps con IA.

La definición oficial, textual

"Active CPU pricing excludes time spent waiting on external services like databases, APIs, or AI model responses." Traducido: no pagás por esperar. Pagás por calcular.

Serverless tradicional
  • Cobra tiempo total de ejecución
  • Llamás a OpenAI y esperás 8s → pagás 8s
  • El 80% del costo es tiempo muerto
  • Apps AI resultan carísimas
  • Tenés que trabajar contra el modelo de pricing
Fluid Compute
  • Cobra solo CPU activo
  • Llamás a OpenAI y esperás 8s → no se cobra la espera
  • Pagás por cálculo real
  • Apps AI tienen costo alineado a valor
  • El pricing está a favor tuyo

La frase "servers in serverless form" que usa Vercel apunta a esto: mantenés la simpleza del serverless (no administrás servidores, no te preocupás por escalado), pero con un modelo de costos que se parece más a un servidor que a una función clásica — solo pagás por trabajo real.

Observability — la cámara de seguridad de tu app

Una vez que tu app está corriendo, necesitás saber qué pasa por dentro. Qué función falló, cuánto tardó, qué request la disparó, qué error tiró. Observability es el servicio que te da esa visibilidad.

Logs
Qué pasó

Mensajes que escribe tu código (console.log, errores). Es el registro textual, útil para entender el flujo y debuggear.

Métricas
Cuánto y cuándo

Números agregados: requests por minuto, latencia promedio, tasa de errores, uso de CPU. Te dicen la salud general del sistema.

Traces
El recorrido completo

La trayectoria de un request puntual: pasó por la función X, llamó a la DB Y, esperó Z ms en la API externa. Crucial para encontrar cuellos de botella.

La analogía de la cámara de seguridad

Sin observability, cuando algo falla estás ciego — el cliente te dice "no funciona" y vos mirás el techo. Con observability, rebobinás la cinta: ves qué request explotó, qué dato llegó mal, dónde se tardó. Es la diferencia entre adivinar y saber.

Los cuatro juntos, en contexto

Core Platform no es una lista de features sueltas: es un combo que trabaja junto. El CI/CD deja tu código listo, la CDN lo acerca al usuario, Fluid Compute lo ejecuta eficientemente, y Observability te deja ver qué está pasando. Sacá uno y el combo se rompe.

CI/CD — tu código llega a producción sin fricción
CDN — tus assets viajan poco, el usuario espera menos
Fluid Compute — tu código corre con costos alineados al mundo AI
Observability — vos ves todo lo que pasa, siempre
Por qué Fluid Compute cambia el juego para apps AI

Si tu automatización llama a un modelo que tarda 10 segundos en responder, en una plataforma tradicional pagás los 10 segundos completos por cada request. Multiplicá por miles de usuarios y la factura te asfixia. Con Fluid Compute, esos 10 segundos de espera no se cobran — solo el puñado de milisegundos donde hay cálculo real. Para un emprendedor vendiendo automatizaciones AI, esto es la diferencia entre un modelo de negocio rentable y uno que no cierra.
