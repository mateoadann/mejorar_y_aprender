Extras y developer experience

Acá juntamos un puñado de features que por separado parecen pequeños detalles, pero juntos son la razón real por la que Vercel enamora a los developers: Preview Deployments, Analytics, Cron, Environment Variables y Edge Middleware.

La diferencia entre usar Vercel y armar tu propia infra rara vez está en las features grandes — cualquier plataforma sabia hacer CI/CD y servir HTML. La diferencia está en estas cien cosas chicas que te resuelven el día a día y que, cuando ya te acostumbraste, no hay vuelta atrás.

Preview Deployments — la feature que cambia reviews

Cada git push a una branch que no sea main te genera una URL única y pública con esa versión de la app corriendo. Así de simple. Pero cambia todo: los reviews dejan de ser "leete el diff del PR" y pasan a ser "abrí este link y probalo".

1
git push feature/x

Subís cambios a una branch que no es main.

→
2
Vercel detecta

Arranca un deploy aislado de esa branch específica.

→
3
URL única

Tipo proyecto-git-feature-x-org.vercel.app con esa versión viva.

→
4
Review en vivo

Mandás el link al equipo o al cliente. Prueban el cambio antes de mergear.

El momento donde se ve el valor

Estás trabajando con un cliente y le mandás tres opciones visuales de la landing, cada una en su branch. El cliente abre tres URLs distintas, compara en tiempo real, decide. Sin llamadas, sin screenshots, sin "te comparto mi pantalla". Ese flujo es imbatible.

Analytics & Speed Insights

Vercel te da dos productos de métricas que vienen integrados al dashboard, sin que tengas que pegar scripts de terceros ni pedir consentimiento de cookies para la mayoría de los casos. Uno mide cuántos te visitan, el otro mide qué tan rápido lo hacen.

Analytics
Privacy-friendly, sin cookies

Pageviews, visitantes únicos, referrers, paths más visitados. Sin identificar individuos, sin banners de cookies invasivos.

Speed Insights
Core Web Vitals reales

Mide en el navegador de usuarios reales las métricas que Google considera clave: LCP (carga), CLS (estabilidad visual), INP (respuesta a input), TTFB (tiempo al primer byte).

Métrica Qué mide Umbral bueno
LCP (Largest Contentful Paint) Cuánto tarda el elemento más grande en aparecer Menos de 2,5 s
CLS (Cumulative Layout Shift) Cuánto se mueven las cosas mientras carga Menos de 0,1
INP (Interaction to Next Paint) Cuánto tarda la UI en responder a un click Menos de 200 ms
TTFB (Time To First Byte) Cuánto tarda el server en mandar el primer byte Menos de 800 ms

Cron Jobs — tareas programadas sin infra aparte

Cron Jobs son funciones serverless que se ejecutan en un horario. Le decís a Vercel "corré esta función todos los días a las 3 AM" o "cada 15 minutos" y listo. No hace falta armar una EC2, ni usar un proveedor aparte, ni configurar un scheduler externo.

Caso 01
Reportes diarios

Generar el PDF de ventas del día y mandarlo por mail al cliente a las 8 AM.

Caso 02
Sincronización con APIs

Cada hora, bajar datos de una API externa (stock, tipo de cambio, leads de un CRM) y guardarlos en tu DB.

Caso 03
Limpieza y mantenimiento

Borrar tokens expirados, archivar registros viejos, reenviar emails que fallaron. Todo automático, a la noche.

Caso 04
Notificaciones programadas

Recordatorios a usuarios que no terminaron un flujo, resúmenes semanales por mail, alertas.

Environment Variables — secretos por environment

Las environment variables son el lugar donde viven las credenciales de tu app: API keys, connection strings, tokens de terceros. Vercel te las gestiona con un detalle clave: las organizás por environment, que son tres:

Env 01
Development

Variables que se inyectan cuando corrés localmente con vercel dev. Típicamente, apuntan a servicios de prueba.

Env 02
Preview

Las que usa cada Preview Deployment. Ideal para tener una DB de staging, una API key de sandbox, etc.

Env 03
Production

Las reales, solo se inyectan en el deploy de main. DB de verdad, API keys pagas, todo serio.

Se acabó el ".env olvidado en el repo"

Los secrets viven en el dashboard de Vercel (encriptados), se inyectan en build y runtime, y nunca terminan commiteados por error. Además, separás ambientes sin esfuerzo: no mezclás la DB de producción con la de preview, y cada branch puede apuntar a servicios de staging sin tocar una línea de código.

Edge Middleware — código antes del server

Middleware es código que corre ANTES de llegar a tu función o página, directamente en los nodos edge. Cada request pasa por él primero, y ahí podés decidir: redirigir, reescribir la URL, autenticar, hacer A/B testing, bloquear por país, lo que se te ocurra.

1
Request del usuario

Llega al nodo edge más cercano.

→
2
Middleware corre

Tu código en el edge decide qué hacer: redirigir, reescribir, validar.

→
3
Pasa a la app

Si el middleware lo permite, el request sigue a tu función o página.

Uso 01
Autenticación

Verificás el token antes de servir una ruta protegida. Si no hay sesión, redirigís a login sin invocar la función.

Uso 02
Redirects

Movés URLs viejas a nuevas, redirigís por país o idioma, gestionás rutas legacy — todo sin tocar tu app.

Uso 03
A/B testing

Dividís usuarios en grupos en el edge y cada uno ve una variante distinta. Sin flash de contenido, sin scripts del cliente.

Todo junto: la capa invisible de DX

Ninguno de estos cinco productos es "la feature estrella" de Vercel. Pero juntos forman la capa que hace que los devs no quieran irse. Son las cosas que, cuando no las tenés, extrañás todos los días sin darte cuenta de qué exactamente.

Preview Deployments — review en vivo, URL por branch
Analytics & Speed Insights — métricas sin setup
Cron Jobs — tareas programadas sin infra extra
Environment Variables — secrets por ambiente
Edge Middleware — código antes del server, en el edge
La conclusión del módulo

Todo esto es developer experience — es por lo que devs la eligen y por lo que después cuesta soltarla. Cuando cotizás proyectos en tu negocio, estos detalles son los que te ahorran días de infra por cliente. Días que podés cobrar como valor, no como tiempo perdido en YAML.
