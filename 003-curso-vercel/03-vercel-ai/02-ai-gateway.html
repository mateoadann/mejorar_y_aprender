AI Gateway — una sola API para todos los modelos

El AI SDK te abstrae la librería cliente. El AI Gateway te abstrae la cuenta, el billing y la observability. Son dos abstracciones distintas sobre el mismo problema: quiero construir con IA sin atarme a un solo proveedor.

El problema — la pesadilla del multi-proveedor

Si armás una app con IA de verdad, vas a querer probar varios modelos. GPT-5 para razonamiento complejo, Claude 4.7 para escritura larga, Gemini 2.5 para cosas multimodales, Groq para velocidad extrema en tareas simples. Son distintos y cada uno gana en lo suyo.

El problema arranca ahí. Cada proveedor te obliga a:

CUENTAS
N dashboards

Una cuenta separada en cada proveedor, con su proceso de signup, verificación y KYC.

API KEYS
N secretos

Una API key por proveedor, que hay que rotar, proteger y gestionar por separado.

BILLING
N facturas

Cargar tarjeta en cada uno, conciliar costos desde lugares distintos, vigilar límites.

OBSERVABILITY
N dashboards más

Logs de tokens, requests y errores repartidos entre varias UIs. Imposible tener vista global.

La solución — un proxy unificado

El AI Gateway es exactamente eso: un proxy entre tu app y los proveedores. Tu código le habla a una sola dirección, con una sola API key. El Gateway se encarga de rutear al modelo que le pediste, trackear lo que gastás, y devolverte la respuesta.

Analogía — el concentrador de cables. En vez de tener cinco cables distintos entrando a tu compu, metés uno solo a un hub y el hub se encarga del resto. El AI Gateway es el hub, pero para APIs de LLM.

Qué te da — las cinco features clave

FEATURE 01
Unified API key

Una sola credencial para todos los proveedores. Cero rotación cruzada, cero lío para dar acceso a tu equipo.

FEATURE 02
Routing entre modelos

Cambiás de proveedor cambiando una config — "usá Claude en vez de GPT" — sin tocar código de la app.

FEATURE 03
Observability central

Todo el tráfico AI de todos los modelos en un mismo dashboard. Ves tokens, latencia y errores por request.

FEATURE 04
Fallbacks automáticos

Si OpenAI se cae, el Gateway puede routear el request a Anthropic sin que tu app se entere. Alta disponibilidad.

FEATURE 05
Caching

Responses repetidas se cachean y te ahorran plata y latencia. Especialmente útil en prompts que se repiten mucho.

Compatibilidad

El Gateway no te obliga a usar AI SDK — es compatible con los SDKs oficiales también. Le podés hablar con:

AI SDK (Vercel) — La combinación más natural
OpenAI SDK — Si ya tenés código escrito así, reapuntás la URL y listo
Anthropic SDK — Misma idea, compatible con la interfaz oficial

En la práctica esto significa que podés adoptar el Gateway sin reescribir tu app: cambiás el endpoint y la API key, y seguís adelante.

Cuánto te cuesta

No cobra markup — pagás lo del modelo

Esto es importante. El Gateway no es un intermediario que se queda con una parte. Pagás exactamente lo que pagarías yendo directo al proveedor — Vercel recupera el costo porque vos ya estás en su plataforma. Es una ventaja real, sobre todo porque te ahorra el desorden de billing sin sumar costo adicional.

Cuándo lo necesitás y cuándo no

Te conviene si…

  • Tenés o vas a tener 2+ modelos en juego
  • Querés flexibilidad futura sin reescribir código
  • Estás en producción y te importan los SLAs (fallbacks)
  • Tu equipo necesita visibilidad central de costos AI
  • Algunos prompts se repiten mucho (caching)

Probablemente no lo necesitás si…

  • Estás empezando con un solo modelo, solo prototipo
  • Tu app hace 10 requests al día, no hay volumen
  • Ya tenés infra de observability AI propia
  • No te molesta gestionar una sola API key de un solo proveedor
Por qué esto baja la barrera para construir

Antes, construir una app multi-modelo era un proyecto de infra aparte. Con el Gateway, se vuelve una decisión de configuración. Eso cambia lo que un dev solo o un equipo chico puede proponerle a un cliente — vender una solución "usamos el mejor modelo para cada tarea" deja de ser humo y se vuelve realista.
