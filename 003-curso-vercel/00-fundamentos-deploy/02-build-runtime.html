Build y runtime

De código fuente a app ejecutándose. Tu código no corre tal cual lo escribís: pasa por un proceso de "cocción" que lo deja listo para producción. Acá desarmamos qué es el build, qué es el runtime, y por qué son dos momentos distintos.

Qué es el "build"

Pensalo como una cocina profesional. Vos no le servís al cliente los ingredientes crudos — los preparás, los cocinás, los emplatás. El build es exactamente eso: agarrás tu código fuente (crudo) y lo transformás en algo listo para servir al navegador.

🥩
Ingredientes crudos
Tu código fuente: archivos .ts, .jsx, imports, comentarios, nombres largos, todo sin optimizar.
→
🔥
Cocción (build)
El compilador/bundler toma todo, lo transpila, lo junta, lo achica, lo optimiza.
→
🍽️
Plato listo para servir
El bundle de producción: HTML, JS y CSS optimizados, listos para el navegador.

¿Por qué no corrés el código fuente directo?

Porque el navegador no entiende la mitad de lo que escribís. Mirá por qué hace falta el build:

01
TypeScript → JavaScript

El navegador no ejecuta .ts. Hay que compilar a JS puro. Lo mismo con JSX de React.

02
Imports a bundle

Tu app tiene 200 archivos que se importan entre sí. El bundler los junta en pocos archivos finales.

03
Minificación

Saca espacios, renombra variables a una letra, achica el archivo para que cargue rápido.

04
Tree-shaking

Detecta código que nunca usás y lo tira a la basura. Si importás una función de una librería, no se lleva la librería entera.

Qué sale del horno

Cuando corre el build, se genera una carpeta (normalmente dist/, build/ o .next/) con los artefactos listos para producción. Algo así: 

# Lo que escribís vos
src/
├── pages/
│   ├── index.tsx
│   └── about.tsx
├── components/
│   └── Button.tsx
└── styles/
    └── globals.css

         build
           ↓

# Lo que sale al servidor
dist/
├── index.html              # HTML pre-renderizado
├── about.html              # HTML pre-renderizado
├── assets/
│   ├── app.a7f3c2.js       # JS minificado + hash
│   ├── vendor.b1e8d4.js    # Librerías separadas
│   └── styles.c9a2f1.css   # CSS optimizado
└── images/
    └── hero.webp           # Imágenes comprimidas

Fijate el detalle del hash en los nombres de archivo (app.a7f3c2.js). Eso permite cachear agresivo: si cambia el código, cambia el hash, y el navegador descarga el nuevo. Si no cambia, usa el que tiene guardado. Magia gratis gracias al build.

Qué es el runtime

El runtime es el entorno donde corre tu código una vez que ya está buildeado. Es el "motor" que ejecuta las instrucciones. Distintos tipos de código viven en distintos runtimes.

01
Navegador

Chrome, Firefox, Safari. Corre el JS/HTML/CSS del lado del cliente. Runtime de facto del frontend.

02
Node.js

El runtime de JavaScript del lado del servidor. El más usado. La mayoría de apps Vercel corren acá.

03
Deno / Bun

Runtimes alternativos a Node.js, más modernos. Menos comunes pero ganando terreno.

04
Edge runtime

Un runtime ultraliviano que corre en servidores distribuidos por el mundo. Vercel lo usa mucho.

Build time vs runtime — dos momentos distintos

Esta es la distinción más importante de la página. Build time y runtime son dos momentos distintos en el tiempo. Pasan cosas diferentes en cada uno.

🔨
BUILD TIME
Pasa una vez. Antes de desplegar. Se compila TS, se empaqueta todo, se generan los artefactos estáticos.
→
📦
DEPLOY
Los artefactos del build se suben al servidor. Quedan ahí esperando visitantes.
→
⚡
RUNTIME
Pasa cada vez que un usuario visita tu app. Se ejecuta el JS, se renderiza, se responden requests.
Build time
  • Pasa una sola vez por deploy
  • Corre en la máquina de Vercel (o tu CI)
  • Tenés acceso al filesystem, a variables de entorno de build
  • Podés pegarle a APIs para traer datos y congelarlos en HTML
  • Si falla, no se despliega nada
Runtime
  • Pasa cada vez que llega una request
  • Corre en el navegador del user o en un servidor en vivo
  • Tenés acceso a datos dinámicos, sesión, cookies
  • Podés hacer cosas interactivas, cambiar UI, llamar APIs
  • Si falla, el user ve un error

SSG, SSR, CSR, ISR — los cuatro sabores

Según cuándo se renderiza cada página (build time o runtime), las apps web caen en uno de estos cuatro modos. Los ves nombrados por todos lados. Por ahora solo mordelos para identificarlos — profundizamos en el Módulo 02.

Modo Nombre completo ¿Cuándo se renderiza?
SSG Static Site Generation Build time. El HTML queda congelado y se sirve igual a todos.
SSR Server-Side Rendering Runtime, en el servidor. Se genera el HTML en vivo por cada request.
CSR Client-Side Rendering Runtime, en el navegador. El servidor manda un HTML vacío y el JS arma todo.
ISR Incremental Static Regeneration Mezcla: build time + regeneración periódica en runtime. Lo mejor de los dos mundos.
Lo vemos en profundidad después

Cada uno tiene sus casos de uso, sus ventajas y sus cagadas. En el Módulo 02 los desarmamos uno por uno con ejemplos reales. Por ahora alcanza con saber que existen y que se distinguen por cuándo se renderiza la página.

En Vercel el build es automático

Cada vez que hacés git push a tu repo, Vercel detecta el cambio, dispara el build en sus servidores, y si sale bien, publica los artefactos en producción. No tenés que correr nada a mano. Deploy por push: así de fácil.
