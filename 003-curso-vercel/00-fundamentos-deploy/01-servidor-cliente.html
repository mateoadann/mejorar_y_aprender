Servidor y cliente — Quién le habla a quién

Toda la web funciona con dos actores que se pasan mensajes: el cliente pide, el servidor responde. Si entendés bien esta conversación, entendés el 80% de lo que hace Vercel por detrás. Vamos despacio.

La analogía del restaurante

Imaginate que entrás a un restaurante. Vos sos el cliente. El mozo te trae el menú, vos pedís, y desde la cocina te llega el plato. En la web pasa exactamente lo mismo: tu navegador es el cliente, la computadora que tiene la app es el servidor (sí, se llama "servidor" porque sirve, igual que el mozo).

🙋
Cliente
Tu navegador (Chrome, Safari, Firefox). Pide cosas: páginas, imágenes, datos.
↔
🍽️
Servidor
Una computadora prendida 24/7 con tu app. Responde a los pedidos que le llegan.

El cliente nunca "entra" al servidor, igual que vos no entrás a la cocina. Le pedís algo al mozo, y el mozo vuelve con la respuesta. En internet, ese ida y vuelta tiene nombre y apellido: request y response.

Request y response — el pedido y la respuesta

Cada vez que escribís una URL en el navegador, o apretás un botón que carga datos, se dispara una request. El servidor la procesa y devuelve una response. Nada más, nada menos.

El flujo de una request
🌐
1. Cliente
Escribís tunegocio.com en el navegador.
→
📨
2. Request
El navegador manda un mensaje: "dame la home".
→
🖥️
3. Servidor
Recibe el pedido, busca qué devolver.
→
📦
4. Response
Arma un paquete con HTML, CSS, JS, y lo manda de vuelta.
→
🖼️
5. Render
El navegador dibuja la página en pantalla.

Lo importante: toda interacción con internet es este ciclo. Cargar una página, mandar un formulario, ver una foto, pedirle algo a una API. Siempre es request y response, una y otra vez.

HTTP — la lengua que hablan cliente y servidor

Para que el mozo te entienda, los dos tienen que hablar el mismo idioma. En la web ese idioma se llama HTTP (HyperText Transfer Protocol). Es un protocolo, o sea, un conjunto de reglas acordadas sobre cómo se arma un pedido y cómo se arma una respuesta.

¿Y HTTPS?

Es lo mismo que HTTP pero cifrado. La "S" es de Secure. Nadie en el medio puede espiar qué estás pidiendo ni qué te responden. Hoy todo el mundo usa HTTPS — Vercel te lo da gratis y lo vemos en la página 03.

Una request HTTP tiene, como mínimo, tres cosas: un método (qué querés hacer), una URL (a qué recurso), y headers (metadatos del pedido). La response tiene un status code (el famoso 200, 404, 500) y un cuerpo con los datos.

IP y puerto — la dirección exacta del servidor

Para que el pedido llegue al servidor correcto, necesitás una dirección. Esa dirección tiene dos partes: la IP y el puerto.

🏢
IP = el edificio
Una serie de números (ej: 76.76.21.21) que identifica una máquina en internet. Como la dirección de calle.
+
🚪
Puerto = el departamento
Un número (ej: 443 para HTTPS, 80 para HTTP) que indica a qué servicio de esa máquina le hablás.

Una misma máquina puede tener muchos servicios corriendo al mismo tiempo: un servidor web, una base de datos, un servicio de mail. Cada uno escucha en un puerto distinto. La IP te lleva al edificio, el puerto te lleva a la puerta correcta.

Puertos que seguro ya viste

80 es HTTP, 443 es HTTPS, 3000 y 5173 son típicos de desarrollo local (Node y Vite). Cuando corrés npm run dev y te dice "open localhost:3000", esa es la combinación IP local + puerto.

Los 4 métodos HTTP que tenés que conocer

El método de la request es la intención: qué querés que pase con el recurso. Hay muchos métodos, pero el 95% del tiempo vas a trabajar con estos cuatro. Seguimos con el restaurante:

Método Intención Analogía del restaurante Ejemplo web
GET Leer / obtener datos Mirar el menú. No cambia nada, solo mirás. Cargar la home, ver un producto.
POST Crear algo nuevo Hacer un pedido. Se genera una comanda nueva. Registrarte, mandar un formulario, subir una foto.
PUT Modificar algo existente Pedir que te cambien las papas por ensalada. Editar tu perfil, actualizar un producto.
DELETE Borrar algo Cancelar el pedido antes de que salga. Eliminar una cuenta, borrar un post.

Hay más métodos (PATCH, OPTIONS, HEAD), pero con estos cuatro entendés el CRUD completo: Create (POST), Read (GET), Update (PUT), Delete (DELETE). Si mañana diseñás una API, esta es la base.

¿Cómo se ve una request de verdad?

No hace falta que te la aprendas, pero es bueno que la hayas visto una vez. Esto es lo que manda tu navegador cuando pedís una página: 

GET /productos/cafe HTTP/1.1
Host: tunegocio.com
User-Agent: Mozilla/5.0 (Macintosh)
Accept: text/html
Cookie: session=abc123

Y la response que te vuelve:

HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 4582

<!DOCTYPE html>
<html>...</html>

Arriba van los headers (metadatos), abajo va el cuerpo (el contenido). El 200 es el status code: "todo OK". Si fuera 404 sería "no encontré ese recurso", y 500 sería "algo explotó en el servidor".

Resumen visual

CLIENTE
El que pide

Tu navegador, una app mobile, otro servidor. Cualquier cosa que inicie una request.

SERVIDOR
El que responde

Una máquina conectada a internet que escucha requests en una IP y un puerto.

HTTP
El idioma

El protocolo que define cómo se arman los mensajes entre cliente y servidor.

MÉTODOS
La intención

GET lee, POST crea, PUT modifica, DELETE borra. El CRUD de toda la web.

Vercel te abstrae todo esto — pero tenés que saber qué pasa abajo

Cuando hagas deploy en Vercel no vas a tocar IPs, puertos, ni configurar HTTPS a mano. Vercel te lo resuelve. Pero cuando algo falle — y va a fallar — el error te va a hablar de status 500, de CORS, de port already in use, de 404 Not Found.

Si no entendés el modelo cliente-servidor, esos mensajes son chino. Si lo entendés, son pistas clarísimas. Por eso arrancamos acá.
