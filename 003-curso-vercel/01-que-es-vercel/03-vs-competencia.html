Vercel vs la competencia

Todas las plataformas son buenas para algo. Todas son malas para otra cosa. No existe silver bullet, existe el caso de uso correcto.

Antes de comparar: el mindset correcto

Cuando elegís una plataforma no estás eligiendo "la mejor del mercado", estás eligiendo "la mejor para lo que vos estás construyendo, con el equipo que tenés, con el presupuesto que manejás". Un emprendedor solo que quiere lanzar una landing con IA no necesita lo mismo que una corporación con diez microservicios stateful.

Por eso en esta página vamos a ser honestos: Vercel no gana en todo. Gana en lo que gana, y en el resto hay opciones mejores.

Dato que no todos conocen. Vercel en realidad corre encima de AWS para la mayoría de sus workloads. Lo que pagás no es "infraestructura mágica", es la capa de DX, previews, edge network y herramientas construidas sobre la nube de Amazon.

Matriz de posicionamiento: los tradeoffs reales

Cada plataforma se para en un punto distinto de estos ejes. Elegir bien es entender dónde querés estar parado vos.

Simplicidad vs Control

Más simple: Vercel, Railway, Render. Apretás deploy y listo.
Más control: AWS, Cloudflare (al bajar a Workers crudos). Configurás cada pieza.

El tradeoff es claro: simpleza te acelera, control te da techo. Si sos un solo dev o un equipo chico, simpleza gana casi siempre.

Precio predecible vs Escala automática

Predecible: Railway, Render. Pagás por la instancia, sabés cuánto te va a venir la factura.
Escala automática: Vercel, AWS Lambda, Cloudflare. Pagás por uso, podés ser barato con poco tráfico o caro con mucho.

Si tu tráfico es estable elegí predecible. Si tenés picos grandes o querés escalar a cero cuando no hay tráfico, elegí usage-based.

Edge vs Región

Edge-first: Cloudflare, Vercel Edge Functions. Tu código corre cerca del usuario en cientos de puntos.
Región: Railway, Render, AWS tradicional. Tu app vive en una (o pocas) regiones.

Edge es ideal para latencia baja global. Región es más simple para backends stateful con DB cerca.

Frontend vs Backend vs Full-stack

Frontend-first: Vercel, Netlify, Cloudflare Pages.
Backend-friendly: Railway, Render, Fly.io, AWS.
Full-stack balanceado: Vercel (si usás Next.js), Railway.

El error común es forzar un backend pesado y stateful en una plataforma frontend. No pelees contra la herramienta, elegí la que encaja.

¿Por qué Vercel gana en ciertos contextos?

Ahora sí, seamos justos con Vercel. Hay escenarios donde es claramente la mejor opción, y vale la pena entender por qué.

Next.js con experiencia premium

Vercel hizo Next.js. Todas las features del framework están optimizadas para correr en Vercel primero. Si elegiste Next.js, cualquier otra plataforma va a ser "usable pero no ideal".

Preview deployments por PR

Cada pull request genera una URL única con tu cambio corriendo en producción simulada. Para equipos que iteran rápido y necesitan revisar antes de mergear, esto es imbatible. Otros lo tienen, Vercel lo hace mejor.

AI SDK + AI Gateway

El bundle de herramientas para construir productos con IA (AI SDK, AI Gateway con múltiples modelos, streaming nativo, v0) es hoy el más completo entre las PaaS. Para features de IA, Vercel te ahorra semanas de plumbing.

Red global + Edge runtime

Tu app se sirve desde muchos puntos del planeta sin que vos hagas nada. Para productos con usuarios distribuidos globalmente, la latencia baja viene de fábrica.

¿Cuándo NO elegir Vercel?

Ser honesto también significa decir cuándo otra opción es mejor. Acá van los casos donde Vercel no es la respuesta.

Cuándo SÍ tiene sentido Vercel

  • Estás usando Next.js o querés usarlo.
  • Querés previews por PR automáticos sin configurar nada.
  • Vas a construir features de IA y querés el bundle más completo.
  • Tu app es mayormente frontend con algo de API y funciones serverless.
  • Tu equipo valora DX por sobre ahorro de infraestructura.

Cuándo elegir otra cosa

  • Backend stateful pesado: workers largos, procesos en background, websockets persistentes. Mejor Railway, Render o Fly.io.
  • Presupuesto ajustado con tráfico impredecible: Cloudflare Pages puede salir mucho más barato en edge puro.
  • Equipos profundamente en AWS: si ya tenés DevOps y todo tu stack vive en AWS, sumar Vercel es fricción y costo duplicado.
  • Apps con DB relacional central: Render o Railway son más naturales para el patrón "app + Postgres tradicional".

El resumen honesto

Elegí por CASO DE USO, no por marca. Vercel no es "el más barato", es "el más simple y completo para ciertas cosas" (Next.js, IA, previews, frontend global). Si tu caso encaja, es una elección fantástica. Si tu caso es otro, hay opciones que te van a servir mejor y te van a costar menos.
Red flag. Si alguien te vende una plataforma sin preguntarte qué estás construyendo, con qué equipo y con qué presupuesto, no te está ayudando. La respuesta correcta siempre es "depende", y después hay que desarmar el depende con preguntas concretas.
