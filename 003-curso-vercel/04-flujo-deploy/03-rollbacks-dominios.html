Rollbacks, dominios y variables

La vida después del deploy. Qué hacer cuando algo se rompe, cómo conectar tu dominio custom con HTTPS automático, cómo usar subdominios para separar productos y cómo manejar variables de entorno sin filtrar secretos.

Instant Rollback — el paracaídas

Algo explotó en producción. Respirá. Tenés un feature que te salva: Instant Rollback. Con un click, volvés al deploy anterior — sin rebuild, sin esperar pipeline, sin llamar a nadie. Segundos, no minutos.

Cómo funciona por dentro. Vercel conserva todos los deploys anteriores. Cada uno vive en el CDN con su propia URL estable. "Rollback" no es reconstruir nada — es solo reasignar el dominio a un deploy que ya existía.
Sin rebuild. No corre install, no corre build, no sube nada nuevo. Por eso es instantáneo — literalmente segundos.
Pantalla de emergencia. Cuando viste el error en prod y el adrenalinazo te subió, esto es tu amigo. Un click, estabilizaste, después averiguás qué pasó con calma.
Reversible. Si el rollback fue un error, volvés a adelante igual de rápido. No es destructivo.

Mirá el flujo en una escena típica de producción rota:

Martes 11 AM
Deploy A (ok)
→
Martes 11:15
Deploy B (roto)
→
Click "Rollback"
Deploy A (live)
→
Con calma
Arreglás B
Por qué esto cambia el juego

En hosting tradicional, un rollback significaba re-deployar la versión anterior — otra vez install, build, upload. Minutos de downtime con usuarios viendo errores. Acá es instantáneo. Por eso los equipos pueden deployar a producción 50 veces por día sin miedo.

Custom domains

Tu proyecto arranca con una URL tuproyecto.vercel.app, que es gratis y funciona perfecto. Pero en algún momento vas a querer tunegocio.com. Vercel maneja toda la parte de DNS y certificados — vos solo decís qué dominio querés.

OPCIÓN 1
vercel.app gratis

La URL por default. HTTPS incluido, CDN global. Perfecta para MVPs, demos o proyectos internos que no necesitan branding.

OPCIÓN 2
Tu propio dominio

Comprás tunegocio.com en cualquier registrar. En el dashboard de Vercel apuntás el proyecto a ese dominio.

DNS
Dos caminos

Vercel te da instrucciones concretas: o cambiás los nameservers a los de Vercel, o agregás records A/CNAME al DNS que ya tenés.

HTTPS
Automático

Vercel emite un certificado con Let's Encrypt y lo renueva solo. Cero configuración. Cero "oh no, se venció el SSL".

Podés también tener múltiples dominios apuntando al mismo proyecto. Ejemplo: tunegocio.com y tunegocio.ar los dos sirven la misma app, con HTTPS en los dos. Útil para marcas con varios TLDs o para redirigir dominios viejos al actual.

Wildcards y subdominios

Acá empieza lo potente para tu negocio, Mateo. Cuando armás soluciones para empresas con varios productos, o apps multi-tenant, los subdominios son tu amigo. Cada uno puede apuntar a un proyecto distinto, todos bajo la misma marca.

APP PRINCIPAL
app.tunegocio.com

La aplicación principal que usan los clientes. Proyecto separado con su propio pipeline.

API
api.tunegocio.com

Backend/API separado del front. Otro proyecto, otro deploy, otra escala si hace falta.

DOCS
docs.tunegocio.com

Documentación pública, blog, changelog. Proyecto aparte, stack distinto si querés (Nextra, Docusaurus, lo que sea).

MULTI-TENANT
*.tunegocio.com

Wildcard: cliente1.tunegocio.com, cliente2.tunegocio.com, cada uno con su data. Un solo proyecto que detecta el subdominio.

Analogía — el edificio de oficinas. El dominio tunegocio.com es el edificio. Cada subdominio es una oficina distinta — una empresa de marketing en el piso 3, una consultora en el piso 5. Comparten la dirección pero son negocios independientes. Vercel te maneja la portería, las llaves y el correo de cada uno.

Environment Variables

Las env vars son donde guardás todo lo que no puede vivir en el código: API keys, connection strings a DB, secretos de JWT. Vercel las maneja separadas por ambiente, encriptadas, con una UI clara.

Se configuran en el dashboard. Settings → Environment Variables. Una UI donde ponés nombre, valor y en qué environments vale (uno, dos o los tres).
Separadas por environment. La misma variable STRIPE_KEY puede tener un valor en Production (key real) y otro en Preview (key de test). Vercel inyecta el que corresponde.
Encriptadas. Una vez que guardás el valor, Vercel no te lo muestra más — ni siquiera a vos. Solo lo inyecta como process.env.* en runtime.
Ejemplos típicos. OPENAI_API_KEY, STRIPE_SECRET_KEY, DATABASE_URL, CLERK_SECRET_KEY, NEXTAUTH_SECRET. Lo que no podés poner en GitHub.

Así se ve en la tabla de configuración:

Nombre Valor Environments
OPENAI_API_KEY sk-proj-*** (encriptado) Production, Preview
STRIPE_SECRET_KEY sk_live_*** (encriptado) Production
STRIPE_SECRET_KEY sk_test_*** (encriptado) Preview, Development
DATABASE_URL postgres://*** (encriptado) Production
DATABASE_URL postgres://*** (encriptado) Preview, Development
NEXT_PUBLIC_APP_URL https://tunegocio.com Production

Secrets vs plain env vars

Todas las env vars se encriptan en reposo. Pero dentro de eso, hay una distinción importante sobre cómo se comportan en logs y en el UI:

Sensitive (secrets)

  • No se muestran en logs del build
  • No aparecen en la UI después de guardarlas
  • Protegidas contra exposición accidental
  • Ideales para API keys, passwords, tokens

Plain

  • Pueden aparecer en logs si tu código las loguea
  • Visibles en la UI para el equipo
  • Menos protección contra leaks
  • Para cosas no secretas: URLs públicas, flags, feature toggles
Regla simple

Si la variable empieza con NEXT_PUBLIC_, VITE_ o cualquier prefijo "public", va a terminar en el bundle del cliente — NUNCA pongas secretos ahí. El resto, marcalas como sensitive y listo.

El cierre del módulo

Toda la gestión de dominios + SSL + env vars que antes costaba horas de devops configurar, acá son clicks en un dashboard. Esta es la tesis de Vercel en una oración: sacarte de encima las tareas repetitivas del deploy para que vos te dediques a lo que le importa a tu cliente. Pasás de "administrador de servidores" a "resolvedor de problemas del negocio".
