Del preview a producción

Ya viste cómo un push genera un preview. Ahora toca el paso grande — cuando ese cambio se vuelve producción, el dominio real, los clientes reales usándolo. Acá entra el concepto de ambientes y el famoso "merge a main".

Los 3 ambientes (environments)

Vercel separa el trabajo en tres mundos distintos. Entender qué vive en cada uno te evita el 80% de los problemas de deploy. Este es el corazón del modelo mental.

AMBIENTE 1
Development

Tu máquina local. Corre con npm run dev o con vercel dev. Lee variables de .env.local. Acá experimentás, rompés, arreglás — sin que nadie te vea.

AMBIENTE 2
Preview

Una URL por cada branch que pusheás. Es la "antesala" de producción. Lo ven vos, tu equipo, quien tenga el link. Usa variables del ambiente Preview.

AMBIENTE 3
Production

La branch principal (main o master). Tu dominio real apunta acá. Es lo que usan los clientes. Usa variables del ambiente Production.

El código fluye en una dirección — de dev a preview a production. Acá va el mapa de cómo se mueve un cambio tuyo desde tu máquina hasta el cliente final:

El merge a main

Acá está el momento de verdad. Cuando mergeás un Pull Request a main, Vercel lo detecta — es otro push, técnicamente — y automáticamente deploya a producción. Sin botón rojo, sin confirmación, sin "¿estás seguro?".

Lo que pasa

  • PR aprobado y mergeado a main
  • Webhook dispara el pipeline completo
  • Build optimizado al máximo
  • Upload al CDN global
  • Tu dominio apunta al nuevo deploy
  • Todo tu tráfico lo ve — instantáneamente

Lo que NO pasa

  • No hay botón "deploy to prod" manual por default
  • No te pregunta "¿seguro?"
  • No hay staging intermedio obligatorio
  • No hay aprobación de un segundo humano (salvo que la configures)

Esto intimida al principio. "¿Merge es igual a prod? ¿Y si me equivoqué?". Pero es exactamente el workflow moderno: el PR review ES la aprobación, el preview ES el staging, y si algo falla en prod tenés rollback instantáneo (lo ves en la próxima página).

La regla de oro

Merge a main = producción. Pensá dos veces antes de mergear viernes a las 6 PM. Muchos equipos tienen "no deploys los viernes" como convención — porque si explota, te pasás el finde apagando incendios en vez de con tu familia.

Qué es una production deployment

Una deploy de producción no es lo mismo que un preview. Técnicamente el pipeline es el mismo, pero hay diferencias clave en el resultado final:

Build optimizado al máximo. Minificación agresiva, tree shaking, code splitting óptimo, assets comprimidos. El objetivo es el bundle más chico y rápido posible.
Tu dominio custom apunta acá. Si tenés tunegocio.com, ese dominio resuelve al deploy de producción. Los previews usan URLs vercel.app.
Servido via CDN global. Los archivos se replican en edge locations de todo el mundo. Un usuario en Buenos Aires recibe los assets desde la edge de Buenos Aires, no de Oregon.
Variables de entorno de Production. Apunta a tu DB real, tu API key de Stripe real (no la test), tu OpenAI de producción. Zero cruces con dev o preview.

Protected deployments (opcional)

Un feature que ignoras hasta que lo necesitás. Podés configurar que los previews requieran password o SSO para verlos. Útil en varios escenarios reales:

USO 1
Apps internas

Un dashboard de gestión interna no debería ser Googleable. Protección con SSO del Google Workspace de la empresa y listo.

USO 2
Trabajo con cliente

Le mandás el preview a un cliente con password. Nadie más que él lo puede ver. Ideal para mockups que todavía no se pueden mostrar.

USO 3
Features en beta

Querés que un grupo reducido pruebe una feature nueva antes de sacarla. Protección por password, mandás el link a los beta testers.

Rolling Releases (planes superiores)

Para cuando tu app tiene tráfico serio y no querés arriesgar todo de golpe. En vez de pasar 100% del tráfico al nuevo deploy al instante, Vercel hace un shift gradual:

Arranque
10% nuevo
→
Primera ventana OK
50% nuevo
→
Todo verde
100% nuevo
Canary metrics. Durante el shift, Vercel compara métricas (errores, latencia, status codes) entre el deploy nuevo y el anterior. Si el nuevo está peor, suena la alarma.
Rollback automático. Si los números del canary están muy por debajo del viejo, Vercel puede revertir solo — sin intervención humana. Esto salva producciones reales.
Menos riesgo por deploy. Un bug afecta al 10% del tráfico durante unos minutos, no al 100% de golpe. Para apps con usuarios pagos, esto es oro.
Cuándo vale la pena

Rolling Releases tiene sentido cuando tu app recibe cientos o miles de requests por minuto. Si tenés 10 usuarios por día, no necesitás esto — con previews alcanza de sobra. Es una herramienta para escala.
