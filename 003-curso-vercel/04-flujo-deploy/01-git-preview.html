Git push → Preview deployment

Arranca acá el corazón del flujo de Vercel. Cada branch que pusheás genera automáticamente una URL propia y navegable — sin configuración, sin tocar servidores. Esto es lo que diferencia a Vercel del hosting tradicional.

El setup inicial — una sola vez

Antes de que toda la magia arranque, hay un paso de onboarding que hacés una vez por proyecto. Después te olvidás. Son básicamente cuatro clicks:

PASO 1
Conectar cuenta

Vinculás tu Vercel con GitHub, GitLab o Bitbucket. Vercel te pide permisos para leer repos y recibir eventos de push.

PASO 2
Importar repo

Elegís el repo que querés deployar de una lista. No hay que subir archivos a ningún lado — Vercel clona directo desde Git.

PASO 3
Framework detectado

Vercel detecta solo si es Next.js, Svelte, Astro, Vite, etc., y sugiere la configuración. Build command, output dir, install command — todo precargado.

PASO 4
Click "Deploy"

El primer deploy arranca. En menos de un minuto tenés una URL viva. Listo, te olvidás del setup y a trabajar.

Qué pasa cada vez que hacés push a una branch

Una vez que el proyecto está conectado, cada push es un deploy automático. Mirá los seis pasos que ocurren en paralelo mientras vos seguís trabajando:

1 · Push. Hacés git push a tu branch. No tiene que ser main — cualquier branch dispara el proceso.
2 · Webhook. GitHub/GitLab/Bitbucket le avisa a Vercel "hey, hubo un push en tal repo, tal commit". Es una notificación instantánea.
3 · Clone. Vercel agarra una máquina efímera y clona tu repo en ese commit exacto. Ambiente limpio, cero contaminación de deploys anteriores.
4 · Install deps. Corre npm install (o pnpm, yarn, bun — lo que uses). Usa cache inteligente para que no baje todo de cero cada vez.
5 · Build. Ejecuta tu build script — el que compila tu código a los archivos que realmente se sirven. Si falla acá, el deploy no sigue y te avisa.
6 · Deploy preview. Sube los artefactos al CDN y te genera una URL única. En un rato la compartís.

Preview URLs — la joya de la corona

Cada branch tiene su URL única, con forma <hash>-<project>.vercel.app. Esto parece un detalle, pero es lo que cambia cómo trabaja un equipo entero.

ÚNICA
Una URL por branch

Dos branches activas, dos URLs distintas, las dos navegables al mismo tiempo. Nada se pisa con nada.

SHAREABLE
Link pegable

Podés mandársela por WhatsApp a un cliente, a un diseñador, a un stakeholder. Abre el link y ve el cambio — no tiene que instalar nada.

PERSISTENTE
Sobrevive

La URL vive mientras la branch exista. Borrás la branch, se apaga la URL. Así de simple.

REAL
Deploy de verdad

No es una simulación. Es tu app corriendo igual que en producción, con CDN, HTTPS, edge functions. Lo que ves es lo que va a ser.

El valor real: podés ver cambios ANTES de mergear. Antes tenías que clonar la branch, instalar deps, correr build, levantar server local, compartir screenshots. Ahora mandás un link. Chau ciclo de 20 minutos, hola ciclo de 20 segundos.

Comment bot en tus PRs

Cada vez que abrís un Pull Request, Vercel deja automáticamente un comentario con el link del preview de esa branch. No tenés que ir a buscarlo al dashboard — está ahí, abajo del diff, a un click.

Qué ves en el comentario del bot

El estado del build (en curso, listo, falló), el link al deployment, links a logs si algo explotó, y si tenés analytics también te agrega un resumen. Es como tener un semáforo en cada PR.

Analogía — emplatar antes de servir. Los preview deployments son como maquetar el plato antes de servirlo — mirás, probás, ajustás, después mandás a producción. En hosting tradicional era como cocinar y servir directo al cliente sin probar. Acá probás cada decisión antes de que llegue a la mesa.

Esto es lo que engancha a los equipos

Los preview deployments son EL feature que enamora a los equipos y los hace no irse más. Diseñadores revisan sin levantar nada, PMs validan flujos, clientes aprueban cambios por link. Una vez que un equipo probó este workflow, no hay vuelta atrás.
